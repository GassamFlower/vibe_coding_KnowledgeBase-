# Vibe Coding 最实用的 AI 对话技巧

> 从 16 份笔记中归纳的高频、可立即上手的对话技巧。每条都给"什么时候用 + 怎么说 + 为什么有效"。可直接复制到对话里使用。

---

## 一、开场类：把角色和阶段讲清楚

### 1. 先给阶段，再给任务

**什么时候用**：每次开始一个新阶段时。

**怎么说**：

> 当前阶段：项目立项，只做产品讨论和立项设计，暂时不要写代码。

**为什么有效**：AI 默认倾向"动手做"。明确阶段，就关掉了它的自动执行模式，让它进入讨论模式。

---

### 2. 给角色，再给任务

**什么时候用**：需要 AI 用某个视角思考时。

**怎么说**：

> 你现在的角色是我的产品合伙人 / 架构顾问 / 代码审查员。请基于这个角色帮我……

**为什么有效**：角色决定视角。让 AI 当"产品合伙人"，它会追问需求；当"代码审查员"，它会找风险。不给角色，它默认当"听话的码农"。

---

### 3. 给上下文，再问问题

**什么时候用**：希望 AI 基于你的项目情况给判断，而不是泛泛而谈时。

**怎么说**：

> 这是我的立项文档：[贴文档]。请基于这份文档，给出当前项目唯一推荐技术栈。不要列多个方案。说明：为什么选它、为什么不选其他方案、这套技术栈适合当前项目的哪些需求、后期可能遇到哪些风险。

**为什么有效**：AI 没有你的项目记忆。给上下文 + 给约束（"不要列多个方案"），它才会给可用的判断而不是 ABCD 选择题。

---

## 二、约束类：给边界，不让 AI 自由发挥

### 4. 唯一推荐，不要选择题

**什么时候用**：让 AI 做技术判断时。

**怎么说**：

> 请给出唯一推荐方案。不要列多个方案让我选。说明为什么选它、为什么不选其他方案。

**为什么有效**：AI 喜欢"列三种方案让你选"——这是它讨好你的方式，但也是甩锅。逼它给唯一答案，它才会真正权衡。

---

### 5. 框架优先，禁止绕道

**什么时候用**：让 AI 写代码时。

**怎么说**：

> 这个项目必须严格遵守当前框架的官方规范，不允许为了图省事绕开框架设计。能用框架自带能力就用，不够再封装。

**为什么有效**：AI 喜欢"自己写一套"——它觉得这样更可控。但自造轮子往往是 bug 重灾区。把这条写进 Agent 宪法效果更好。

---

### 6. 不要先写代码，先讨论 / 先做计划

**什么时候用**：进入新阶段、新模块时。

**怎么说**：

> 先不要写代码。请基于需求输出实施计划，包含：阶段、任务、验收标准、不做什么、规则引用。我确认后再开始写。

**为什么有效**：AI 默认"边写边想"。先要计划，等于让它先想清楚再动手，避免改一半发现方向错了。

---

### 7. 范围锁死：本轮只做什么

**什么时候用**：每轮任务开始时。

**怎么说**：

> 本轮目标：项目能启动、首页能打开、路由结构清楚、目录规范正确、UI 组件库接入完成、设计 token 准备好。不要一次性做完所有页面和功能。

**为什么有效**：AI 喜欢"一口气做完"——它觉得这样显得能干。但一口气做完 = 一口气错完。锁死范围，每轮交付一个可验收的小块。

---

## 三、追问类：不让 AI 用模糊话糊弄过去

### 8. 要证据，不要结论

**什么时候用**：AI 说"已完成""做好了""已优化"时。

**怎么说**：

> 请不要只说已完成。每一项必须说明：规则来源、对应文件、验证方式、实际结果、是否通过。没有证据的项目标记为未验收。

**为什么有效**："已完成"是 AI 最爱的逃避。逼它给证据（文件、命令、输出、日志），它要么真的去验证，要么承认没做。

---

### 9. 模糊词禁止令

**什么时候用**：AI 用"基本完成""应该没问题""理论上可以"时。

**怎么说**：

> 禁止使用"基本完成""应该没问题""理论上可以""大概""可能"这类模糊词。要么"通过"，要么"未通过"加原因。

**为什么有效**：模糊词是 AI 给自己留退路。砍掉退路，它必须给明确结论。

---

### 10. 用大白话讲一遍

**什么时候用**：AI 解释一堆你看不懂的技术名词时。

**怎么说**：

> 用最直观的语言告诉我，不要说一堆技术名词。如果我看不懂，说明你还没讲清楚。

**为什么有效**：你看不懂的解释，往往 AI 自己也没真懂。逼它用大白话重讲，是检验它是否真理解的最快方式。这条对安全审查尤其重要——你看不懂的安全说明，本身就不算合格。

---

### 11. 让 AI 主动指出盲区

**什么时候用**：做需求 / 设计 / 计划时。

**怎么说**：

> 请主动指出我没想到的问题：权限、数据结构、状态字段、后期扩展、合规风险、性能瓶颈、跨端兼容、安全边界。不要只回答我问的，要补我没问的。

**为什么有效**：AI 默认"问什么答什么"。让它补盲区，它才会动用知识储备帮你查漏。

---

## 四、收尾类：让 AI 在交付前自检

### 12. 完成前的验收清单

**什么时候用**：AI 准备宣布完成时。

**怎么说**：

> 在说"已完成"之前，请按以下清单自检：功能能不能跑？规则有没有违反 Agent 宪法 / 真源文档？接口输入是否后端重新校验？数据库结构变更是否先改 schema/migration？返回示例是否覆盖成功 / 失败 / 边界场景？关键操作有没有日志？回归检查做了吗？每项给证据，未验证项标记为"未验证"。

**为什么有效**：让 AI 在交付前自检，等于让它当自己的审查员。比事后让你来发现问题成本低得多。

---

### 13. 提交前的总结

**什么时候用**：每次准备 Git 提交前。

**怎么说**：

> 提交前请回答：本次改了哪些文件？验证是否通过？有没有副作用？列出文件清单和验证证据，确认后再提交。

**为什么有效**：让 AI 在提交前复述改动，等于让它再过一遍自己的工作。很多低级错误在这一步会被发现。

---

### 14. 看不懂的安全说明，本身就不合格

**什么时候用**：AI 做安全审查时。

**怎么说**：

> 安全审查不要只说"已做安全处理"。请按身份认证 / 权限控制 / 输入校验 / 数据归属 / 注入防护 5 道关卡分别说明：风险是什么、当前在哪里处理、处理规则是什么、如何验证、哪些还没验证。看不懂的解释重讲一遍，不要用技术名词糊弄。

**为什么有效**：安全是 AI 最爱用大词糊弄的领域。逼它按 5 道关卡逐项交证据，它要么真做，要么承认没做。

---

## 五、控制类：防止项目失控

### 15. 改坏了别让它继续补，先回退

**什么时候用**：AI 改乱了，越补越乱时。

**怎么说**：

> 停。当前改动已经偏离目标。请先列出最近 3 次 Git 提交，告诉我每个版本做了什么。我决定回退到哪个版本后，再重新开始这一轮。

**为什么有效**：AI 越补越乱是 Vibe Coding 最常见的灾难。让它停下、回退、重来，比让它继续补成本低 10 倍。**没 Git 就没法回退**——这也是为什么 Git 是 Vibe Coding 的后悔药。

---

### 16. 一阶段一提交

**什么时候用**：每完成一个阶段时。

**怎么说**：

> 本阶段已通过验收。请总结本次改动，然后提交 Git：commit message 用 feat: / fix: / chore: 前缀，简述本次内容。

**为什么有效**：阶段提交 = 存档点。改坏了能回退，出问题能定位。不让 AI 一口气改一堆才提交。

---

### 17. 真源文档先行

**什么时候用**：任何规则变更（目录、接口格式、错误码、权限规则）发生时。

**怎么说**：

> 在改代码之前，先更新真源文档对应章节，说明变更原因。然后再改代码。改完后再回到真源文档确认描述和实现一致。

**为什么有效**：AI 没有持久记忆。真源文档是它的"项目记忆"。规则改了文档没改，下次它就按旧规则写，项目必然漂移。

## 六、组合套路：可直接复制的高质量 Prompt 模板

### 套路 A：项目启动模板

```
当前阶段：项目立项，只做产品讨论和立项设计，暂时不要写代码。
你的角色：我的产品合伙人。

我想做的是：[一句话描述]。
面向用户：[谁]。
解决什么问题：[什么]。
参考产品：[链接 / 截图，可选]。

请追问关键问题，主动指出我没想到的盲区，最后把讨论结果写入立项文档。
立项文档必须包含：项目立项说明、核心功能和边界、产品流程和用户路径、数据和业务对象、技术路线初步判断、项目爆点和差异点、后期规划、验收规则。
```

### 套路 B：技术选型模板

```
这是我的立项文档：[贴文档]。
产品形态：[Web / 小程序 / App / 后台]。
部署要求：[私有 / 公有 / 本地]。
维护能力：[1人 / 小团队 / 中型团队]。

请给出唯一推荐技术栈，不要列多个方案。说明：
- 前端框架 + UI 组件库
- 后端语言 + 框架
- 数据库
- 主要框架和 SDK
- 为什么选它们
- 为什么不选其他方案
- 哪些情况才需要重新评估技术路线

把结果写入技术选型文档。
```

### 套路 C：写代码前的实施计划模板

```
先不要写代码。请基于立项文档、技术选型文档、Agent 宪法和相关真源文档，输出本阶段实施计划。

计划必须包含：
- 阶段拆分
- 每个阶段的任务（做什么、涉及哪些文件、依赖、验收标准）
- 优先级（P0/P1/P2）
- 本阶段不做什么
- 规则引用（要遵守哪些真源文档 / 宪法条款）

我确认后再开始写。
``### 套路 D：写代码前的范围锁死模板

```
本轮目标（只做这些）：
- [任务1]
- [任务2]
- [任务3]

不要一次性做完所有功能。本轮只交付可验收的最小块。
完成后按验收清单自检，每项给证据，未验证项标记为"未验证"。
```

### 套路 E：完成前验收模板

```
在说"已完成"之前，请按以下清单自检并交证据：

功能：
- [核心路径走通了吗？给执行命令和输出]

规则：
- [有没有违反 Agent 宪法 / 真源文档？给具体条款和代码位置]

安全（涉及后端时）：
- [接口输入是否后端重新校验？]
- [权限校验在哪里？]
- [敏感数据是否哈希？]

数据：
- [数据库结构变更是否先改 schema/migration？]

接口：
- [返回示例是否覆盖成功 / 失败 / 边界场景？]

日志：
- [关键操作有没有日志？]

回归：
- [相邻功能有没有被破坏？]

每项给文件 / 命令 / 输出 / 日志作为证据。
未验证项标记为"未验证"，不允许写"基本完成"。
最后给明确结论：能否进入下一阶段。
```

### 套路 F：安全审查模板

```
请对当前后端做安全审查。不要只说"已做安全处理"。

按 5 道关卡分别说明：
1. 身份认证：哪些接口需要登录，在哪里校验
2. 权限控制：哪些接口限定角色，无权限返回什么
3. 输入校验：关键参数（金额、角色、用户ID、订单归属）在哪里后端重新校验
4. 数据归属：合法用户是否会触到不属于自己的数据（水平越权）
5. 注入防护：SQL / 命令 / 模板注入，用的是框架方案还是自写过滤

每道关卡给：风险 / 处理位置 / 处理规则 / 如何验证 / 是否通过。
看不懂的解释用大白话重讲一遍。
最后输出整改清单，按严重 / 风险 / 建议分级。
```

### 套路 G：调试模板

```
不要凭感觉改代码。按系统化调试流程走：

1. 复现：[触发步骤 / 参数 / 环境]
2. 收集证据：[完整错误信息 / 日志 / 最近正常版本]
3. 列出至少 2-3 个根因假设，每个标依据
4. 用最小代价验证假设，逐个排除
5. 定位根因，明确到文件 + 行号
6. 给修复方案：思路 / 影响范围 / 是否触及真源文档
7. 实施修复（改动尽量小，不顺手优化周边）
8. 回归验证：复现步骤重跑 + 相邻功能跑一遍
9. 总结根因和修复方案，必要时写进真源文档历史问题章节
```

### 套路 H：代码审查模板

```
请对本次改动做代码审查。范围：[文件 / 提交 / 模块]。

按以下维度逐项检查，每项给"通过 / 风险 / 严重"判断 + 证据：
- 结构：目录放置、分层、重复实现
- 规则：宪法、真源文档、框架规范
- 安全（涉及后端）：输入校验、越权、密码、注入、过度防御
- 数据：schema/migration、范式、敏感数据、软删除
- 可维护性：命名、硬编码、提前封装、日志/错误/测试

风险分级：
🔴 严重：阻断合并
🟡 风险：建议修
🟢 建议：可选优化

输出整改清单。不擅自重构审查范围外代码。
```

## 七、心法（最关键的 5 条）

1. **AI 没有记忆，文档是它的记忆**。立项文档、技术选型、Agent 宪法、真源文档、Skill，每一份都是把讨论结果固化下来，防止下次它按旧规则写。
2. **AI 默认动手，你要逼它先想**。先讨论 → 再计划 → 再写代码，每一步都要明确告诉它"现在不要写代码"。
3. **AI 默认讨好，你要逼它给判断**。"列三种方案让你选"是讨好，"给唯一推荐"才是判断。
4. **AI 默认结论，你要逼它给证据**。"已完成"不算完成，文件 + 命令 + 输出 + 日志才算完成。
5. **AI 默认前进，你要逼它会停**。改乱了别让它继续补，停下来回退到稳定版本，比继续补成本低 10 倍。Git 是后悔药。
