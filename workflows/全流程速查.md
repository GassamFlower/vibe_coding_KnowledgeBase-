# Vibe Coding 全流程速查

> 9 个流程合并为 1 个速查手册。每个流程保留：触发条件、核心步骤、完成判定、反模式。

---

## 01 项目立项流程

**触发条件**：准备开始新项目 / 旧项目重来  
**输入**：脑子里的产品方向 + 参考产品（可选）  
**输出**：立项文档 + 功能清单 + 阶段计划

### 核心步骤
1. 创建空项目文件夹（英文 + 中划线/下划线）
2. 进入"产品合伙人"模式（明确告诉 AI：当前阶段只做产品讨论，不要写代码）
3. 与 AI 讨论：项目类型、目标用户、核心问题、MVP 边界、长期规划
4. 让 AI 查漏补缺：权限、数据结构、状态字段、后期扩展、合规风险
5. 生成立项文档（8 项内容：立项说明、功能边界、用户路径、业务对象、技术路线、爆点差异、里程碑、验收规则）
6. **功能清单拆解**（三维度：按角色 / 按业务流程 / 按模块）
   - 每个功能写清：给谁用、输入、输出、权限、正常流程、异常边界
   - 复杂功能（订单流转 / 支付退款 / 优惠叠加 / 权限系统）单独拆文档 + 留索引
7. **阶段计划制定**
   - 第一阶段必须是 MVP（能跑通核心流程的最小可用版本）
   - 只给大阶段划分，不细化到技术实现（等选型定了再排细节）
8. 提交 Git：`git commit -m "chore: 完成项目立项"`

### 完成判定
- [ ] 项目文件夹存在且命名规范
- [ ] 立项文档已生成，覆盖 8 项内容
- [ ] 功能清单已拆解（三维度完整，复杂功能已单独拆文档）
- [ ] 阶段计划已制定（第一阶段是 MVP，只给大方向）
- [ ] AI 已确认"暂时不要写代码"
- [ ] 已提交 Git

### 反模式
- ❌ 直接说"帮我做个项目"
- ❌ 跳过讨论直接让 AI 创建文件、安装依赖
- ❌ 立项文档只在对话里聊过没落地成文件
- ❌ **跳过功能清单直接写代码**（AI 自由脑补边界）
- ❌ **功能清单只按模块拆，漏了角色和业务流程维度**
- ❌ **复杂功能硬塞进功能清单**（应单独拆文档 + 留索引）
- ❌ **没做阶段计划就开始写代码**（不知道先做什么，MVP 跑不通）
- ❌ **阶段计划直接细化到技术实现**（选型没定排细节 = 白排）

## 02 技术栈选择流程

**触发条件**：立项文档已生成，准备进入开发  
**输入**：立项文档 + 部署要求 + 维护能力  
**输出**：技术选型文档

### 核心步骤
1. 确定产品形态（网站 / 小程序 / App / 后台 / 纯后端）
2. 让 AI 给"唯一推荐技术栈"（不要列多个方案）
3. 优先选成熟框架和 SDK（查 GitHub star、issue 活跃度、最近提交时间、许可证）
4. 给 AI 画边界（必须遵守当前框架官方规范）
5. 写进技术选型文档（产品形态、前端、后端、数据库、框架 SDK 列表、选的理由、不选的理由、重评估条件）
6. 提交 Git：`git commit -m "chore: 确定项目技术栈"`

### 完成判定
- [ ] 产品形态已确定
- [ ] AI 给出唯一推荐方案并说明理由
- [ ] 框架/SDK 成熟度已核查
- [ ] 技术选型文档已生成
- [ ] 已提交 Git

### 反模式
- ❌ 让 AI 列三四套方案让你选
- ❌ 为了看起来高级选资料少、AI 不熟的方案
- ❌ 技术栈只在对话里聊过没写进文档

## 03 Agent 宪法创建流程

**触发条件**：技术栈确定，即将让 AI 写第一行代码  
**输入**：立项文档 + 技术选型文档  
**输出**：Agent 宪法文件

### 核心步骤
1. 确定宪法文件位置（Codex: `AGENTS.md` / Cursor: `.cursor/rules/` / Claude Code: `CLAUDE.md`）
2. 遵守三条写作原则：短硬清楚、写长期原则、写禁止事项和验收标准
3. 写可执行、可检查、可判断的规则（不要写"认真一点""写好一点"）
4. 套用模板（安全诚信、诚实边界、执行铁律、代码生成器承诺）
5. 项目专属条款（后端/前端/数据库项目各自追加）
6. 提交 Git：`git commit -m "chore: 创建项目 Agent 宪法"`

### 完成判定
- [ ] 宪法文件放在 IDE 会读取的位置
- [ ] 主体不超过几百行
- [ ] 包含禁止事项和验收标准
- [ ] 项目专属条款已追加
- [ ] 已提交 Git

### 反模式
- ❌ 把项目介绍、技术细节、部署流程全塞进宪法
- ❌ 写"注意质量""认真一点"这种模糊口号
- ❌ 把"具体怎么做"全写进宪法（应放进 Skill 或真源文档）

## 04 前端骨架搭建流程

**触发条件**：立项、技术选型、Agent 宪法都已完成  
**输入**：立项文档 + 技术选型文档 + Agent 宪法  
**输出**：前端实施计划 + 最小可运行骨架

### 核心步骤
1. 确定整体设计风格（玻璃拟态 / Material / 企业级 / SaaS 极简 / AI 科技感）
2. 确定前端技术方案和 UI 组件库（不要混：技术方案管结构，UI 库管界面）
3. 给 AI 画边界（必须遵守前端框架官方规范，优先复用 UI 组件库）
4. 定目录、模块、组件规则（目录边界、模块边界、组件复用规则）
5. 准备样式系统（设计 token、多语言规范、主题规则）
6. 写进实施计划（设计风格、技术方案、UI 库、目录结构、模块边界、组件封装、token、多语言、主题、第一阶段内容、验收标准）
7. 搭最小可运行骨架（能启动、首页能打开、路由清楚、目录规范、UI 库接入、token 准备、主题接入）
8. 验收 + 提交 Git：`git commit -m "feat: 搭建前端最小可运行骨架"`

### 完成判定
- [ ] 设计风格已明确
- [ ] 前端框架 + UI 组件库已接入
- [ ] 目录/模块/组件规则已写进实施计划
- [ ] 设计 token、多语言、主题规则已准备
- [ ] 项目能启动、首页能打开
- [ ] 已提交 Git

### 反模式
- ❌ 直接让 AI 写页面，跳过骨架
- ❌ 一次性做完所有页面和功能
- ❌ 没有 UI 组件库，让 AI 到处手写按钮/弹窗/表格
- ❌ 设计 token 没提前定，到处硬编码样式

## 05 数据库设计流程

**触发条件**：前端页面/流程基本确定  
**输入**：前端页面列表 + 立项文档里的业务对象清单  
**输出**：数据库设计文件（schema.sql 或 migration）+ 数据库实施计划

### 核心步骤
1. 从前端反推数据（页面列表 → 数据字段、详情页 → 完整结构、表单 → 提交数据、按钮 → 操作行为、状态 → 数据流转）
2. 从业务流程里找对象（电商：用户/商品/订单；内容：用户/文章/评论；后台：用户/角色/权限）
3. 确认对象之间的关系（一对一、一对多、多对多、关联字段在哪）
4. 确定数据库服务（主数据库用 MySQL 或 PostgreSQL，要不要 Redis）
5. 设计表结构和字段规范（三大范式：字段原子化、围绕主对象、不互相依赖）
6. 规范常见字段（id、created_at、updated_at、deleted_at、password_hash、金额用精确类型、状态字段枚举值、字段名英文下划线）
7. 先生成设计文件，再建表（禁止直接连数据库建表）
8. 写进实施计划（业务对象、对象关系、主数据库、Redis、核心表、字段规范、表间关联、软删除、范式校验、验收标准）
9. 跑通基础结构 + 提交 Git：`git commit -m "feat: 完成数据库设计文件并跑通基础结构"`

### 完成判定
- [ ] 业务对象清单已整理
- [ ] 对象关系已确认
- [ ] 数据库服务已选定
- [ ] 表结构符合三大范式（或反范式有说明）
- [ ] 常见字段规范已确认
- [ ] schema.sql / migration 文件已生成并提交 Git
- [ ] 数据库能连接、能建表、能插入和查询测试数据

### 反模式
- ❌ 直接让 AI 连数据库建表，跳过设计文件
- ❌ 所有数据塞进一个大字段
- ❌ 表关系没设计清楚就开建
- ❌ 密码明文存储
- ❌ 金额用浮点数
- ❌ 没有 migration 记录，直接手动改字段

## 06 后端骨架搭建流程

**触发条件**：立项、技术选型、数据库设计都已完成  
**输入**：立项文档 + 技术选型文档 + 数据库设计文件 + Agent 宪法  
**输出**：项目架构设计文档 + 最小可运行后端骨架

### 核心步骤
1. 复核后端语言和框架（不要重新发散选型，只判断是否过度选型）
2. 让 AI 说明规则来源（语言工程规范、框架最佳实践、项目架构设计文档）
3. 写项目架构设计文档（语言框架、选择原因、语言规范、框架实践、框架最大化利用、目录约束、接口响应规则、错误处理规则、日志规则、数据库连接、权限校验入口、新增模块规则）
4. 写进 Agent 宪法的约束（必须遵守语言工程规范和框架最佳实践）
5. 让 AI 搭最小可运行骨架（启动入口、环境变量、路由分组、健康检查、数据库连接、统一响应、统一错误、日志、参数校验、权限中间件占位、README）
6. 跑通四条线（启动线、接口线、业务线、运维线）
7. 提交 Git：`git commit -m "feat: 搭建后端最小可运行骨架"`

### 完成判定
- [ ] 语言/框架已复核、路线已钉住
- [ ] 项目架构设计文档已生成
- [ ] Agent 宪法已加入后端约束条款
- [ ] 最小可运行骨架已搭好
- [ ] 启动线 / 接口线 / 业务线 / 运维线四条线都跑通
- [ ] 已提交 Git

### 反模式
- ❌ 跳过项目架构设计文档直接写代码
- ❌ 一上来就写完整登录 / 订单 / 支付
- ❌ 让 AI 自由发挥目录结构
- ❌ 能用框架自带能力却让 AI 手写一套

## 07 后端架构验收流程

**触发条件**：AI 报告"后端骨架已搭好，可以开始写业务代码"  
**输入**：项目架构设计文档 + 后端骨架代码  
**输出**：验收报告 + 后端架构实施真源文档

### 核心步骤
1. 让 AI 交证据（不是结论）（每项必须说明：规则来源、对应文件、验证方式、实际结果、是否通过）
2. 验收目录责任（目录责任表：目录、职责、应该放什么、禁止放什么、框架机制、示例文件）
3. 最小模块演练（不写完整业务，只输出文件清单和调用路径）
4. 看接口返回示例（11 种场景：列表成功、详情成功、创建成功、更新成功、删除成功、空列表、参数错误、未登录、无权限、资源不存在、系统异常）
5. 验收框架复用和封装边界（哪些复用框架、哪些自定义封装、去掉有什么影响）
6. 固定启动证据（依赖安装、启动命令、监听端口、健康检查、配置读取、.env.example、数据库连接、请求日志、错误日志）
7. 收口验收报告（验收项、是否通过、证据位置、未通过原因、下一步、是否影响进入业务开发）
8. 生成《后端架构实施真源文档》（目录责任、接口规则、错误处理、启动证据、框架复用边界、新增模块规则）
9. 提交 Git：`git commit -m "chore: 后端架构验收通过，固化实施真源文档"`

### 完成判定
- [ ] 每一项验收都有证据（文件、命令、返回示例、日志）
- [ ] 目录责任表清晰
- [ ] 最小模块演练通过
- [ ] 接口返回示例覆盖 11 种场景
- [ ] 框架复用边界清楚
- [ ] 启动证据包完整
- [ ] 《后端架构实施真源文档》已生成
- [ ] Agent 宪法已加引用条款
- [ ] 已提交 Git

### 反模式
- ❌ 听 AI 说"已经完成"就开始写业务
- ❌ 只看目录数量，被"很多文件夹"骗到
- ❌ 接口"统一"只听结论不看示例
- ❌ 把所有架构细则塞进 Agent 宪法（应放进真源文档）

## 08 后端安全审查流程

**触发条件**：后端骨架验收通过，开始写业务功能之后，定期或在重要功能上线前执行  
**输入**：项目架构设计文档 + 后端架构实施真源文档 + 当前业务功能清单  
**输出**：后端安全边界表 + 系统权限设计表 + 注入风险审查报告 + 安全整改清单

### 核心步骤
1. 拆安全边界（5 道关卡：身份认证、权限控制、输入校验、数据归属、注入防护）
2. 审查接口输入（默认不可信，关键输入和普通输入分开处理）
3. 审查密码和管理员账号规则（密码长度、字符组合、弱密码拦截、哈希存储、失败次数限制）
4. 输出系统权限设计表（每个接口：是否需要登录、允许哪些角色、是否只能操作自己的数据、权限判断在哪里、无权限返回什么）
5. 检查注入风险（SQL 注入、命令注入、模板注入，用框架 ORM 还是字符串拼接）
6. 审查过度防御（区分必须校验的业务规则、框架已处理的基础校验、前端体验层处理、死代码）
7. 整改 + 回归（把所有"未验证"或"风险"项整理成整改清单，逐项修复后重新执行）
8. 提交 Git：`git commit -m "chore: 后端安全审查通过"`

### 完成判定
- [ ] 安全边界表覆盖 5 道关卡，每项有证据
- [ ] 关键输入和普通输入分开处理
- [ ] 密码规则、哈希存储、失败次数限制齐全
- [ ] 系统权限设计表覆盖所有接口
- [ ] 无字符串拼接用户输入的注入风险
- [ ] 无死代码或过度防御
- [ ] 已提交 Git

### 反模式
- ❌ 听 AI 说"已经做了安全处理"就放过
- ❌ 只做"要不要登录"忽略水平越权
- ❌ 让 AI 自己写过滤规则代替参数化查询
- ❌ 为了安全感堆一堆永远走不到的判断分支

## 09 Git 阶段提交流程

**触发条件**：每完成一个明确阶段并且能正常运行 / 每次让 AI 做较大修改之前 / 推远程之前  
**输入**：当前项目状态 + 本次阶段改动总结  
**输出**：一次清晰的 Git 提交 + 远程仓库（如需推送）

### 核心步骤
1. 安装并初始化 Git（`git init && git add . && git commit -m "chore: 项目初始化"`）
2. 养成阶段提交习惯（每完成一个明确阶段并且能正常运行就提交一次）
3. 让 AI 先总结再提交（本次改了哪些文件、验证是否通过、有没有副作用）
4. 规范 commit message（chore/feat/fix/refactor/docs/test）
5. 大改之前先确认存档（当前有没有稳定存档、工作区是否干净）
6. 改坏了就回退（`git status && git diff && git checkout . && git reset --hard <commit-id>`）
7. 推远程之前先确认文件清单（仓库私有还是公开、有没有真实凭证、有没有商业计划）

### 完成判定
- [ ] Git 已安装并初始化
- [ ] 当前阶段已能正常运行
- [ ] AI 已说明本次改动和验证结果
- [ ] commit message 规范
- [ ] 已提交
- [ ] 推远程前已确认文件清单（如适用）

### 反模式
- ❌ 项目已经乱了才想起来用 Git
- ❌ 改一堆功能才提交一次，分不清每次改了什么
- ❌ 不看 `git status` 直接 `git add .`（容易把 .env / 密钥提交进去）
- ❌ 把商业计划 / Agent 宪法 / 真实 case 跟着代码一起推到开源仓库
