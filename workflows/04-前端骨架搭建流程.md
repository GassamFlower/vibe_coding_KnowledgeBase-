# 04 前端骨架搭建流程

> 适用阶段：宪法建好之后，第一次写前端代码。目标：在写第一个页面前先把统一规则定下来，避免 AI 到处手写。

## 触发条件

- 立项、技术选型、Agent 宪法都已完成；
- 准备开始写前端。

## 输入

- 立项文档；
- 技术选型文档（前端框架 + UI 组件库）；
- Agent 宪法。

## 输出

- 前端实施计划；
- 最小可运行前端骨架（能启动、能打开首页）；
- 第一次 Git 提交。

## 执行步骤

### 1. 确定整体设计风格

不要直接说"做得好看一点"。先确定整个 UI 呈现出来的统一视觉语言。常见风格：

- 玻璃拟态 / 液态玻璃 / 半透明模糊背景
- Material Design
- 企业级管理台风格（信息密度高、表格多、操作效率优先）
- SaaS 现代极简风格
- AI 产品科技感风格（深色、渐变、发光、动效）
- 内容社区信息流风格

不知道自己想要什么风格时，找几个喜欢的网站链接 / App 截图发给 AI：

> 请分析这些界面的视觉风格、布局方式、组件特点，以及这种风格适不适合我的项目。参考是为了确定方向，不是直接复制。

### 2. 确定前端技术方案和 UI 组件库

两件事不要混：

- **前端技术方案**（管项目结构）：React / Vue / Angular / Next.js / Nuxt；小程序可选微信原生、Taro、uni-app。
- **UI 组件库**（管界面统一）：Ant Design / Element Plus / Naive UI / MUI / Mantine / Vant / NutUI。

### 3. 给 AI 画边界

明确告诉 AI：

> 这个项目必须遵守当前前端框架的官方规范，优先复用选定的 UI 组件库，不允许到处手写重复组件，也不允许绕开框架最佳实践。

### 4. 定目录、模块、组件规则

- **目录边界**：页面 / 组件 / 接口 / 工具函数 / 样式文件，分别放哪里。
- **模块边界**：登录、用户、订单、内容、设置等功能区域分开，不要混在一起。
- **组件复用规则**：能用 UI 组件库就优先用；不够再基于组件库封装项目业务组件。判断标准：UI 相似度高且出现超过 2 次就封装。

### 5. 准备样式系统

- **设计 token**：颜色、字号、行高、间距、圆角、阴影、边框——必须从统一设计 token 里取，不允许硬编码。
- **多语言规范**：按钮文字、页面标题、提示、错误信息统一管理，不要硬编码文案。
- **主题规则**：先定一个默认主色调，再预留主题扩展方式，主题色不能散落在页面里手写。

### 6. 写进实施计划

让 AI 输出前端实施计划，至少包含：

- 整体设计风格
- 前端技术方案
- UI 组件库
- 目录结构
- 模块边界
- 哪些组件先封装
- 设计 token 怎么定义
- 多语言和主题规则
- 第一阶段只做哪些内容
- 验收标准

### 7. 搭最小可运行骨架

明确告诉 AI：

> 请根据前端实施计划搭建最小可运行前端骨架。不要一次性做完所有页面和功能。本轮目标：项目能启动、首页能打开、路由结构清楚、目录规范正确、UI 组件库接入完成、设计 token 准备好、主题规则接入完成、常用组件有基础版本。

### 8. 验收 + 提交 Git

骨架跑通后立即提交：

```
git add .
git commit -m "feat: 搭建前端最小可运行骨架"
```

之后再一个模块一个模块往里加功能。

## 完成判定

- [ ] 设计风格已明确；
- [ ] 前端框架 + UI 组件库已接入；
- [ ] 目录/模块/组件规则已写进实施计划；
- [ ] 设计 token、多语言、主题规则已准备；
- [ ] 项目能启动、首页能打开；
- [ ] 已提交 Git。

## 反模式

- ❌ 直接让 AI 写页面，跳过骨架；
- ❌ 一次性做完所有页面和功能；
- ❌ 没有 UI 组件库，让 AI 到处手写按钮/弹窗/表格；
- ❌ 设计 token 没提前定，到处硬编码样式。
