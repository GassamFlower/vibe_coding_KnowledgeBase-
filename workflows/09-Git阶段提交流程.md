# 09 Git 阶段提交流程

> 适用阶段：项目从立项到上线全周期。目标：让 AI 的每一次大改都变成可回退的存档点，避免改坏了不可逆。

## 触发条件

- 每完成一个明确阶段并且能正常运行；
- 每次让 AI 做较大修改之前；
- 推远程之前。

## 输入

- 当前项目状态；
- 本次阶段改动总结。

## 输出

- 一次清晰的 Git 提交；
- 远程仓库（如需推送）。

## 执行步骤

### 1. 安装并初始化 Git

第一次进入项目时：

```bash
git init
git add .
git commit -m "chore: 项目初始化"
```

如果没装 Git，先装 Git（不强制使用 GitHub/码云，本地 Git 就能做版本管理和回退）。

### 2. 养成阶段提交习惯

规则：**每完成一个明确阶段并且能正常运行，就提交一次。** 不是每改一行就提交，也不是改完一大堆才提交。

典型阶段提交点：

- 项目创建完成
- 立项文档生成
- 技术选型文档生成
- Agent 宪法创建
- 目录结构搭好
- 前端骨架能启动 / 首页能打开
- 数据库设计文件生成
- 数据库能连接 / 能建表
- 后端骨架能启动 / 健康检查通过
- 后端架构验收通过
- 每一个核心功能做完
- 上线前最终验收

### 3. 让 AI 先总结再提交

写进 Agent 宪法的规则：

> 每完成一个明确阶段，必须先说明本次改了哪些内容，验证是否通过，然后再提交一次 Git 记录。

让 AI 在提交前回答：

- 本次改了哪些文件？
- 验证是否通过？（能启动 / 能跑 / 测试通过）
- 有没有副作用？

确认后再提交。

### 4. 规范 commit message

简单约定：

- `chore:` 杂项 / 立项 / 配置 / 文档
- `feat:` 新功能
- `fix:` 修 bug
- `refactor:` 重构（不影响行为）
- `docs:` 文档变更
- `test:` 测试相关

示例：

```
git add .
git commit -m "feat: 完成登录接口（含密码强度校验和失败次数限制）"
```

### 5. 大改之前先确认存档

每次让 AI 做较大修改之前，先确认：

- 当前有没有一个稳定存档（最近一次提交）？
- 当前工作区是不是干净的（`git status`）？

如果不是，先提交或先 stash，再让 AI 动手。打 boss 之前先存档。

### 6. 改坏了就回退

如果 AI 改乱了：

```bash
git status              # 看改了什么
git diff                # 看具体差异
git checkout .          # 放弃所有未提交的修改（谨慎）
git reset --hard <commit-id>  # 回到某个稳定版本（谨慎）
```

> 没用 Git，AI 改坏了你只能继续让它补，补着补着问题越来越多。有了 Git，至少可以知道每一步发生了什么，也能在出问题时退回稳定版本。

### 7. 推远程之前先确认文件清单

让 AI 列出文件清单：

> 这次准备提交哪些文件？哪些文件会推到远程？有没有不该公开的内容？

确认：

- 仓库是私有还是公开；
- 有没有真实凭证（API Key、Token、数据库连接信息）；
- 有没有商业计划、内部规划、真实 case、Agent 宪法等不该跟着代码开源的内容。

确认没问题再推送：

```bash
git remote add origin <仓库地址>
git push -u origin main
```

如果项目准备开源，公开的应该是代码、README、安装方式、使用说明、贡献指南，而不是商业计划/内部规划/真实 case/Agent 宪法。

## 完成判定

- [ ] Git 已安装并初始化；
- [ ] 当前阶段已能正常运行；
- [ ] AI 已说明本次改动和验证结果；
- [ ] commit message 规范；
- [ ] 已提交；
- [ ] 推远程前已确认文件清单（如适用）。

## 反模式

- ❌ 项目已经乱了才想起来用 Git；
- ❌ 改一堆功能才提交一次，分不清每次改了什么；
- ❌ 不看 `git status` 直接 `git add .`（容易把 .env / 密钥提交进去）；
- ❌ 把商业计划 / Agent 宪法 / 真实 case 跟着代码一起推到开源仓库。
