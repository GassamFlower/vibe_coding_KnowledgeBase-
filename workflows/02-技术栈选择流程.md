# 02 技术栈选择流程

> 适用阶段：立项文档生成之后、写代码之前。目标：把产品形态和技术路线钉死，避免技术路线摇摆。

## 触发条件

- 立项文档已生成；
- 准备进入开发但技术栈未确定。

## 输入

- 立项文档（项目类型、用户场景、核心功能、后期规划）；
- 部署要求、维护能力约束。

## 输出

- 技术选型文档（写入项目根目录）。

## 执行步骤

### 1. 确定产品形态

问自己：产品最终以什么形式呈现给用户？

- 网站 / Web App
- 小程序
- App（原生或跨平台）
- 后台管理系统
- 纯后端接口

答不上来就先别写代码。可以问 AI：

> 请基于立项文档，结合产品目标、用户场景、核心功能和后期规划，判断这个项目最适合做成哪一种形态，并说明理由。给一个主线判断，不要列多个选择题。

### 2. 让 AI 给"唯一推荐技术栈"

把立项文档、产品形态、功能范围、目标用户、部署要求、后期规划交给 AI：

> 请基于以上内容，给出当前项目的唯一推荐技术栈。不要列多个方案。说明：为什么选它、为什么不选其他方案、这套技术栈适合当前项目的哪些需求、后期可能遇到哪些风险。

### 3. 优先选成熟框架和 SDK

判断标准：

- 是否有足够多的人在用；
- 项目是否还在维护；
- 文档是否完整；
- 社区是否活跃；
- 最近更新时间是否合理；
- 授权协议是否允许商业使用（如准备上线运营）。

让 AI 帮你查关键点：GitHub star 数、issue 活跃度、最近提交时间、发布记录、许可证、使用案例。

### 4. 给 AI 画边界

明确告诉 AI：

> 这个项目必须严格遵守当前框架的官方规范，不允许为了图省事绕开框架设计。

### 5. 写进技术选型文档

文档至少包含：

- 产品形态
- 前端用什么（框架 + UI 组件库）
- 后端用什么（语言 + 框架）
- 数据库用什么
- 主要框架和 SDK 列表
- 为什么这么选
- 哪些技术不用、为什么不选
- 什么情况下才需要重新评估技术路线

### 6. 提交 Git

```
git add .
git commit -m "chore: 确定项目技术栈"
```

## 完成判定

- [ ] 产品形态已确定；
- [ ] AI 给出唯一推荐方案并说明理由；
- [ ] 框架/SDK 的成熟度已核查；
- [ ] 技术选型文档已生成；
- [ ] 已提交 Git。

## 反模式

- ❌ 让 AI 列三四套方案让你选；
- ❌ 为了看起来高级选资料少、AI 不熟的方案；
- ❌ 听说某语言"更安全"就当成唯一选择（安全不是只靠语言决定）；
- ❌ 技术栈只在对话里聊过没写进文档。
