# 05 数据库设计流程

> 适用阶段：前端骨架跑通后。建议顺序：先 UI → 再数据库 → 最后后端。UI 会倒逼你想清楚数据形式。

## 触发条件

- 前端页面/流程基本确定；
- 准备让 AI 设计数据库。

## 输入

- 前端页面列表、详情页、表单、按钮、状态字段；
- 立项文档里的业务对象清单。

## 输出

- 数据库设计文件（`schema.sql` 或 migration）；
- 数据库实施计划；
- 第一次 Git 提交（含设计文件）。

## 执行步骤

### 1. 从前端反推数据

让 AI 根据页面回答这些问题：

> 这个列表需要什么表？这个详情页需要哪些字段？这个表单提交后要保存到哪里？这个状态从哪里来？这个按钮操作会改哪条数据？

- 页面列表 → 映射数据字段
- 详情页面 → 推导完整结构
- 功能表单 → 定义提交数据
- 交互按钮 → 确定操作行为
- 页面状态 → 指明数据流转

### 2. 从业务流程里找对象

让 AI 根据页面、用户流程、核心功能整理项目里有哪些业务对象。

- 电商：用户、商品、订单、支付记录、收货地址
- 内容平台：用户、文章、评论、点赞、收藏
- 后台系统：用户、角色、权限、操作日志
- 预约系统：用户、服务项目、预约记录、时间段、门店

先别急着建表，先让 AI 说清楚：项目里有哪些核心对象，每个对象对应什么业务场景。

### 3. 确认对象之间的关系

让 AI 解释清楚：

- 每张表代表什么对象
- 哪些表之间有关联
- 是一对一、一对多，还是多对多
- 关联字段放在哪张表里
- 为什么这样设计

### 4. 确定数据库服务

让 AI 说清楚：

- 主数据库用什么（普通业务项目优先 MySQL 或 PostgreSQL）
- 为什么用它
- 要不要 Redis（除非有明确缓存/队列/高并发需求，否则不要默认必选）
- 本地开发和线上部署怎么处理

让 AI 给唯一推荐方案，不要列多个选择题。

### 5. 设计表结构和字段规范

按三大范式做基本校对：

- **第一范式**：字段要原子化，一个字段只存一个明确含义。
- **第二范式**：字段要围绕当前这张表的主对象。
- **第三范式**：字段之间不要互相依赖。

明确告诉 AI：

> 数据库设计尽量符合前三范式，避免字段混乱、重复存储和不清晰的数据依赖。如果为了查询性能需要反范式设计，必须先说明原因。

### 6. 规范常见字段

- 主键 `id`、`created_at`、`updated_at` 一般由程序或数据库自动维护；
- 不能物理删除的数据（订单、支付、财务）加 `deleted_at` 做软删除；
- 密码用 `password_hash`，禁止明文；
- 金额用分为单位的整数或数据库精确数字类型，说清货币和精度；
- 状态字段定义清楚枚举值（如订单状态：待支付、已支付、已取消、已退款）；
- 字段名用英文下划线命名；
- 关键业务字段（手机号是否支持国际号码、金额是否多币种、状态后续扩展）确认清楚。

### 7. 先生成设计文件，再建表

**禁止直接连数据库建表**。先让 AI 生成 `schema.sql` 或 migration 文件。

更稳的流程：

1. 生成设计文件；
2. 核对业务对象、表关系、字段和三大范式；
3. 确认没问题再执行建表；
4. 后续改字段也先改 schema/migration 再执行变更。

把这条写进 Agent 宪法：

> 任何数据库结构变更，必须先修改 schema 或 migration 文件，说明变更原因，再执行数据库操作。

### 8. 写进实施计划

实施计划至少包含：

- 业务对象清单
- 对象之间的关系
- 主数据库
- 是否需要 Redis
- 核心表清单
- 每张表负责什么业务
- 字段名、字段类型、是否必填
- 哪些字段唯一、哪些可空
- 表间关联
- 哪些数据不能物理删除
- 是否符合前三范式
- 验收标准

### 9. 跑通基础结构 + 提交 Git

目标只是先把数据库基础结构跑通：能连接、核心表能创建、migration 能执行、能插入和查询测试数据。

```
git add .
git commit -m "feat: 完成数据库设计文件并跑通基础结构"
```

## 完成判定

- [ ] 业务对象清单已整理；
- [ ] 对象关系已确认；
- [ ] 数据库服务已选定；
- [ ] 表结构符合三大范式（或反范式有说明）；
- [ ] 常见字段规范已确认；
- [ ] `schema.sql` / migration 文件已生成并提交 Git；
- [ ] 数据库能连接、能建表、能插入和查询测试数据。

## 反模式

- ❌ 直接让 AI 连数据库建表，跳过设计文件；
- ❌ 所有数据塞进一个大字段；
- ❌ 表关系没设计清楚就开建；
- ❌ 密码明文存储；
- ❌ 金额用浮点数；
- ❌ 没有 migration 记录，直接手动改字段。
