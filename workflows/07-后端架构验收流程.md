# 07 后端架构验收流程

> 适用阶段：后端骨架搭好之后、写业务功能之前。目标：用证据证明规则立住了，而不是听 AI 说"已经完成"。

## 触发条件

- AI 报告"后端骨架已搭好，可以开始写业务代码"。

## 输入

- 项目架构设计文档；
- 后端骨架代码。

## 输出

- 验收报告（验收项 / 是否通过 / 证据位置 / 未通过原因 / 下一步 / 是否影响进入业务开发）；
- 后端架构实施真源文档；
- 第一次 Git 提交（骨架作为稳定版本）。

## 执行步骤

### 1. 让 AI 交证据（不是结论）

明确告诉 AI：

> 请根据项目实际业务文档以及项目架构设计文档，对后端架构层代码做详细的审计和验收。每一项必须说明：规则来源、对应文件、验证方式、实际结果、是否通过。没有证据的项目标记为未验收，不允许写通过。

如果只说"已完成、已优化、已符合最佳实践"但没有文件、命令、返回示例、日志结果，不算验收。看不懂解释时直接说"用最直观的语言告诉我，不要说一堆技术名词"。

### 2. 验收目录责任

让 AI 输出目录责任表：

> 请根据当前后端架构，输出目录责任表。每一行包含：目录、主要职责、以后应该放什么、禁止放什么、对应的框架机制、示例文件。请标注哪些目录来自框架推荐或者框架本身，哪些是项目业务自定义的。同时请按当前框架说明：请求入口在哪里，业务规则在哪里，数据库访问在哪里，参数校验在哪里，错误处理在哪里，权限校验入口在哪里。

只看三件事：

1. 每个目录是不是只有一个清楚的主要责任；
2. 有没有说清楚以后应该放什么、不应该放什么；
3. 有没有说明这些目录是框架推荐还是项目自定义。

### 3. 最小模块演练

不要直接写业务，做一次最小模块演练：

> 请做一次最小模块演练，不写完整业务代码，只输出文件清单和调用路径。请说明：需要新增或修改哪些文件；每个文件放在哪个目录；每个文件负责什么；请求从哪里进入；参数在哪里校验；业务规则在哪里处理；数据访问在哪里处理；统一响应和统一错误在哪里处理；这些安排是否符合项目架构设计文档。

只看两点：

1. 文件放置有没有延续目录责任表；
2. 请求入口、参数校验、业务规则、数据访问、响应和错误处理有没有被分开。

### 4. 看接口返回示例

让 AI 输出接口响应样例：

> 请输出当前后端架构的接口响应样例，至少包含：列表成功、详情对象成功、创建成功、更新成功、删除成功、空列表、参数错误、未登录、无权限、资源不存在、系统异常。每个场景都要给出 HTTP 状态码、响应 JSON 示例、由哪个文件或框架机制统一处理。

不要漏：

1. 列表和对象分开看（列表含数组、分页、总数；对象是详情/创建/更新数据）；
2. 错误不只看业务错误码，HTTP 状态码复用框架规范。

### 5. 验收框架复用和封装边界

> 请审查当前后端架构的框架复用和封装边界。哪些能力直接复用了框架？哪些是项目自定义封装？每个自定义封装解决什么问题？如果去掉会有什么影响？是否存在为了显得专业而提前封装的内容？

看：能用框架的地方有没有用框架；自定义封装有没有明确理由；这些规则有没有写进项目架构设计文档。说不清解决什么问题的封装先不要放进架构。

### 6. 固定启动证据

让 AI 交运行证据包：

> 请输出后端架构运行证据包。必须包含：依赖安装命令、启动命令、服务监听端口、健康检查命令和结果、配置读取位置、.env.example 内容检查、数据库连接验证方式和结果、请求日志示例、错误日志示例。每一项都要写明实际执行结果。没有执行的项目标记为未验证。

只看：有没有把"怎么启动、用什么配置、数据库是否连上、日志在哪里看"说固定。"理论上可以启动"不算通过。

### 7. 收口验收报告

> 请汇总本次后端架构验收结果，不要重新发散，也不要重复解释前面的内容。请按以下结构输出：验收项、是否通过、证据位置、未通过原因、下一步处理、是否影响进入业务开发。最后给出明确结论：当前后端架构是否可以开始写业务功能。没有证据的项目必须标记为未验证，不能写基本完成。

### 8. 生成《后端架构实施真源文档》

关键项都通过后，让 AI 整理一份详细的实施真源文档（目录责任、接口规则、错误处理、启动证据、框架复用边界、新增模块怎么组织文件）。然后在 Agent 宪法里加引用：

> 后端开发必须先读取并遵守《后端架构实施真源文档》。新增或修改后端功能前，必须确认目录责任、接口规则、错误处理规则、启动证据和框架复用边界。任何目录调整、规则变更、框架更换或新增关键依赖，都必须先说明原因，并同步更新后端架构实施真源文档。

### 9. 提交 Git

```
git add .
git commit -m "chore: 后端架构验收通过，固化实施真源文档"
```

## 完成判定

- [ ] 每一项验收都有证据（文件、命令、返回示例、日志）；
- [ ] 目录责任表清晰；
- [ ] 最小模块演练通过；
- [ ] 接口返回示例覆盖 11 种场景；
- [ ] 框架复用边界清楚；
- [ ] 启动证据包完整；
- [ ] 《后端架构实施真源文档》已生成；
- [ ] Agent 宪法已加引用条款；
- [ ] 已提交 Git。

## 反模式

- ❌ 听 AI 说"已经完成"就开始写业务；
- ❌ 只看目录数量，被"很多文件夹"骗到；
- ❌ 接口"统一"只听结论不看示例；
- ❌ 把所有架构细则塞进 Agent 宪法（应放进真源文档）。
