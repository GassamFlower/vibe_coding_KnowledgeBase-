# 08 后端安全审查流程

> 适用阶段：后端骨架验收通过、开始写业务功能之后，定期或在重要功能上线前执行。目标：不要听 AI 说"已经做了安全处理"，要看它把风险、规则、处理位置和验证证据一项项交出来。

## 触发条件

- 后端骨架验收通过，准备写业务接口；
- 涉及登录、注册、修改密码、支付、订单、删除、权限相关功能；
- 上线前最后一次安全检查。

## 输入

- 项目架构设计文档；
- 后端架构实施真源文档；
- 当前业务功能清单。

## 输出

- 后端安全边界表；
- 系统权限设计表；
- 注入风险审查报告；
- 安全整改清单。

## 执行步骤

### 1. 拆安全边界（5 道关卡）

让 AI 输出后端安全边界表：

> 请根据项目架构设计文档和当前业务功能，输出后端安全边界表。不要只说已做安全处理。请按接口输入、登录状态、系统权限设计、密码规则、数据归属、注入风险分别说明：风险是什么，当前在哪里处理，处理规则是什么，如何验证，哪些还没有验证。

5 道关卡：

1. **身份认证**：这个请求是谁发的、有没有登录。
2. **权限控制**：这个人能不能做这件事。
3. **输入校验**：前端传进来的数据可不可信。
4. **数据归属**：这条数据是不是属于他自己。
5. **注入防护**：用户填的内容会不会被后端当成命令去执行。

看不懂解释时直接说"用大白话重讲一遍，别用技术名词"。**你看不懂的安全说明，本身就不算合格。**

### 2. 审查接口输入（默认不可信）

> 请审查当前后端接口的输入安全。不要只说已做参数校验。请按接口说明：哪些参数来自前端，哪些参数会影响身份、系统权限、金额、库存、数据归属或数据库写入；每个关键参数在哪里校验；校验失败返回什么；是否有示例请求证明。

只看：有没有把关键输入（价格、数量、角色、用户 ID、订单归属、数据状态）和普通输入分开。前端校验只防用户体验，真正影响业务结果的必须在后端重新判断。

### 3. 审查密码和管理员账号规则

> 请审查当前后端的账号和密码安全规则。请说明注册、登录、修改密码、重置密码、创建管理员账号分别在哪里校验密码强度。请检查是否包含密码长度、字符组合、常见弱密码拦截、密码是否哈希存储（禁止明文保存）、登录失败次数限制、管理员账号额外要求。没有实现的项目标记为未验证，不要默认安全。

至少确认三件事：

1. 密码规则存在（长度 + 字符组合 + 弱密码拦截 + 失败次数限制）；
2. 管理员账号有更严格要求；
3. 后端真的会拦截明显弱密码；
4. 密码用哈希存储，禁止明文。

### 4. 输出系统权限设计表

> 请根据项目功能设计文档，输出系统权限设计表。每个接口都要说明：是否需要登录、允许哪些角色访问、是否只能操作自己的数据、管理员是否可以访问、权限判断在哪个文件或中间件处理、无权限时返回什么、有没有测试证明。

重点防两种越权：

- **垂直越权**：普通用户调用了管理员才能用的接口。
- **水平越权**：合法用户动了不属于自己的数据（比如改链接里的订单号看别人订单）。

很多项目只做"要不要登录"却漏掉"登录后能不能动这条数据"，水平越权大多就是这么来的。

### 5. 检查注入风险

> 请检查当前接口是否存在注入风险。重点检查 SQL 注入、数据库查询注入、命令注入、HTML 或模板内容注入。请说明哪些位置使用了框架 ORM、参数化查询、白名单校验、内容转义或内容清洗；如果存在字符串拼接用户输入的情况，请标记为风险并给出修复方案。

只让 AI 讲清楚一件事：用的是框架/数据库自带的成熟方案，还是它自己临时写的一段过滤代码。**自己造的防护，往往就是漏得最多的。**

### 6. 审查过度防御（死代码）

> 请审查当前接口安全逻辑是否存在过度防御。请区分：必须校验的业务规则、框架已经处理的基础校验、可以由前端体验层处理的提示校验，以及永远不会触发的防御分支（死代码）。不要为了安全感新增重复判断或不可能成立的条件。任何新增安全封装都必须说明解决什么真实风险。

该严的地方：是否登录、是否有权限访问这条数据、是否能修改这条记录、金额和库存是否符合业务规则。不该复杂的地方：输入框提示、格式提醒、前端展示优化。

### 7. 整改 + 回归

把所有"未验证"或"风险"项整理成整改清单，逐项修复后重新执行步骤 1-6，直到全部通过。

### 8. 提交 Git

```
git add .
git commit -m "chore: 后端安全审查通过"
```

## 完成判定

- [ ] 安全边界表覆盖 5 道关卡，每项有证据；
- [ ] 关键输入和普通输入分开处理；
- [ ] 密码规则、哈希存储、失败次数限制齐全；
- [ ] 系统权限设计表覆盖所有接口；
- [ ] 无字符串拼接用户输入的注入风险；
- [ ] 无死代码或过度防御；
- [ ] 已提交 Git。

## 反模式

- ❌ 听 AI 说"已经做了安全处理"就放过；
- ❌ 只做"要不要登录"忽略水平越权；
- ❌ 让 AI 自己写过滤规则代替参数化查询；
- ❌ 为了安全感堆一堆永远走不到的判断分支。
