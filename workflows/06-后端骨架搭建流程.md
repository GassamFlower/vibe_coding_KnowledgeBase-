# 06 后端骨架搭建流程

> 适用阶段：数据库设计已落地。目标：先把规则写进项目架构设计文档，再让 AI 搭最小可运行骨架，避免业务写到一半规则漂移。

## 触发条件

- 立项、技术选型、数据库设计都已完成；
- 准备让 AI 写后端代码。

## 输入

- 立项文档；
- 技术选型文档（后端语言 + 框架）；
- 数据库设计文件；
- Agent 宪法。

## 输出

- 项目架构设计文档；
- 最小可运行后端骨架；
- 第一次 Git 提交。

## 执行步骤

### 1. 复核后端语言和框架

不要重新发散选型，只判断是否过度选型。问 AI：

> 请根据项目架构设计文档，确认当前后端语言和框架是否适合本项目。不要重新发散选型，只判断是否存在为了高级感、性能感或者安全感而过度选型的问题。如果当前语言或框架较重，请说明项目为什么真的需要它，以及后续必须遵守哪些成熟框架规范和最佳实践。

目标：**把路线钉住**。Vibe Coding 最怕技术路线摇摆。

### 2. 让 AI 说明规则来源

让 AI 回答清楚：

- 当前语言的工程规范是什么（依赖管理、配置读取、日志、异常、测试通常怎么处理）
- 当前框架的最佳实践是什么（不同框架的习惯不能混着写）
- 当前项目自己的架构设计文档是什么

### 3. 写项目架构设计文档

让 AI 整理（不要写业务代码）：

> 请根据立项文档和已确定的后端语言、框架，整理项目架构设计文档。不要写业务代码。文档必须写清楚：后端语言和框架、选择原因、不选其他方案的原因、必须遵守的语言工程规范、必须遵守的框架最佳实践、框架最大化利用原则、目录约束、接口响应规则、错误处理规则、日志规则、数据库连接方式、权限校验入口，以及以后新增模块应该怎么放文件。

### 4. 写进 Agent 宪法的约束

把这条加进宪法：

> 后端开发必须遵守当前语言工程规范和框架最佳实践，优先复用框架能力。新增依赖、目录调整、错误码自定义、框架更换，都要先说明原因，并更新项目架构设计文档。

### 5. 让 AI 搭最小可运行骨架

明确告诉 AI：

> 请严格根据项目架构设计文档，基于当前后端语言和框架，按照该语言工程规范、该框架官方文档或主流最佳实践，搭建最小可运行后端骨架。先说明目录结构、目录作用、哪些是框架推荐，哪些是项目自定义。不要先写完整业务功能。骨架必须包含：项目启动入口、环境变量配置、路由分组、健康检查接口、数据库连接、统一响应格式、统一错误处理、日志、基础参数校验、权限中间件占位、README 启动说明。

5 个约束不能丢：

1. 必须基于当前语言和框架；
2. 必须遵守语言规范和框架最佳实践；
3. 必须根据项目架构设计文档执行；
4. 必须先解释目录结构，再写代码；
5. 只搭最小可运行骨架，不写完整业务功能。

### 6. 跑通四条线

- **启动线**：项目能启动、配置能读取、有 `.env.example`、有 `/health` 健康检查。
- **接口线**：路由分组、统一响应、统一错误，HTTP 状态码复用框架规范。
- **业务线**：接口层 / 业务层 / 数据层分开（按当前框架习惯命名，不强求 controller/service/repository）。
- **运维线**：日志、统一错误处理、数据库连接验证、权限校验占位、README 启动说明。

### 7. 提交 Git

```
git add .
git commit -m "feat: 搭建后端最小可运行骨架"
```

骨架跑通后，先去做"后端架构验收流程"，再开始写业务功能。

## 完成判定

- [ ] 语言/框架已复核、路线已钉住；
- [ ] 项目架构设计文档已生成；
- [ ] Agent 宪法已加入后端约束条款；
- [ ] 最小可运行骨架已搭好；
- [ ] 启动线 / 接口线 / 业务线 / 运维线四条线都跑通；
- [ ] 已提交 Git。

## 反模式

- ❌ 跳过项目架构设计文档直接写代码；
- ❌ 一上来就写完整登录 / 订单 / 支付；
- ❌ 让 AI 自由发挥目录结构；
- ❌ 能用框架自带能力却让 AI 手写一套。
