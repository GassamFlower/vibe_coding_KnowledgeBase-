# 01 项目立项流程

> 适用阶段：项目启动前。目标：把模糊想法变成可执行的项目起点，并把讨论结果沉淀成立项文档。

## 触发条件

- 你准备开始一个新项目；
- 或者旧项目重来，需要重新对齐方向。

## 输入

- 你脑子里模糊的产品方向；
- 参考的同类产品链接 / 截图（可选）。

## 输出

- 立项文档（写入项目根目录）；
- 长期路线文档（可选，复杂项目时拆分）。

## 执行步骤

### 1. 创建空项目文件夹

- 文件夹名用英文 + 中划线/下划线，如 `task-manager`、`resume-assistant`；
- 禁止使用中文、空格、特殊符号，禁止叫"新建文件夹""测试项目""demo1"；
- 用 AI IDE 打开这个空文件夹。

### 2. 进入"产品合伙人"模式

明确告诉 AI：

> 当前阶段只做产品讨论和立项设计，暂时不要写代码。

### 3. 与 AI 讨论（不要让它写代码）

让 AI 围绕以下问题追问你、指出问题、拆分功能、判断第一版做什么不做什么：

- 项目是什么类型？面向谁？解决什么问题？
- 核心功能有哪些？数据怎么流转？商业模式是什么？
- 用户场景是什么？核心操作流程是什么？
- 第一版 MVP 边界是什么？哪些功能后面再做？哪些暂时放弃？
- 长期版本往哪里走？
- 有哪些类似产品、竞品、开源方案？差异点和机会在哪？

### 4. 让 AI 帮你查漏补缺

主动让 AI 补盲区：权限、数据结构、状态字段、后期扩展、合规风险等。

### 5. 生成立项文档

让 AI 把讨论结果写入项目根目录。立项文档至少包含：

1. 项目立项说明（是什么 / 面向谁 / 解决什么问题 / 为什么值得做 / 第一版核心目标）
2. 项目核心功能和边界（第一版做什么、不做什么）
3. 产品流程和用户路径（页面怎么走、功能怎么串、数据怎么流动）
4. 数据和业务对象（用户、订单、任务、文章、商品、文件、评论、权限等）
5. 技术路线初步判断（网页 / 小程序 / App / Web MVP）
6. 项目爆点和差异点
7. 后期规划和里程碑（第一版、第二版、长期版本）
8. 验收规则（第一版做到什么程度才算完成——必须有可检查标准）

### 6. 提交 Git

```
git add .
git commit -m "chore: 完成项目立项"
```

## 完成判定

- [ ] 项目文件夹存在且命名规范；
- [ ] 立项文档已生成，覆盖上述 8 项内容；
- [ ] AI 已确认"暂时不要写代码"；
- [ ] 已提交一次 Git。

## 反模式

- ❌ 直接说"帮我做个项目" → AI 自由发挥、项目漂移；
- ❌ 跳过讨论直接让 AI 创建文件、安装依赖；
- ❌ 立项文档只在对话里聊过没落地成文件。
