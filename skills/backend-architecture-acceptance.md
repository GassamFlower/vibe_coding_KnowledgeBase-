# Skill: backend-architecture-acceptance（后端架构验收）

## 何时使用

- AI 报告"后端骨架已搭好，可以开始写业务代码"；
- 用户问"这套架构能用吗""稳不稳""可以开始写业务了吗"；
- 后端骨架阶段收尾。

## 解决什么问题

避免 AI 用"已经完成"糊弄过去。让验收落到证据上：规则来源、对应文件、验证方式、实际结果、是否通过。没有证据就是未验收。

## 执行流程

1. **让 AI 交证据**：

   > 请根据项目实际业务文档以及项目架构设计文档，对后端架构层代码做详细的审计和验收。每一项必须说明：规则来源、对应文件、验证方式、实际结果、是否通过。没有证据的项目标记为未验收，不允许写通过。

2. **验收目录责任**：让 AI 输出目录责任表（目录 / 主要职责 / 应放什么 / 禁止放什么 / 框架机制 / 示例文件）。看每个目录是否只有一个清楚的主要责任。
3. **最小模块演练**：不写完整业务，只输出文件清单和调用路径。看请求入口、参数校验、业务规则、数据访问、响应、错误处理是否分开。
4. **看接口返回示例**：覆盖列表成功 / 详情对象成功 / 创建 / 更新 / 删除 / 空列表 / 参数错误 / 未登录 / 无权限 / 资源不存在 / 系统异常 11 种场景。每个给 HTTP 状态码 + JSON 示例 + 处理位置。
5. **验收框架复用和封装边界**：能用框架的地方有没有用框架？自定义封装有没有明确理由？说不清解决什么问题的封装先不要放进来。
6. **固定启动证据**：依赖安装命令 / 启动命令 / 端口 / 健康检查 / 配置读取位置 / `.env.example` / 数据库连接验证 / 请求日志 / 错误日志，每项写实际执行结果。"理论上可以启动"不算通过。
7. **收口验收报告**：按"验收项 / 是否通过 / 证据位置 / 未通过原因 / 下一步 / 是否影响进入业务开发"输出。最后给明确结论：能不能开始写业务。
8. **生成《后端架构实施真源文档》**：关键项通过后整理出详细规则文件，Agent 宪法只放引用条款，不承载所有细则。
9. **触发 Git 提交**：把验收通过的后端架构作为第一个稳定版本。

## 输出

- 验收报告；
- 《后端架构实施真源文档》；
- Agent 宪法中的引用条款；
- 已通过的 Git 提交。

## 约束

- ❌ 不听"已经完成""可以开始写业务"就放过；
- ❌ 不被目录数量骗到（不是越多越专业）；
- ❌ 接口"统一"只听结论不看示例；
- ❌ 不把所有架构细则塞进 Agent 宪法（应放进真源文档）；
- ❌ 未验证项不能写"基本完成"；
- ❌ 关键项未通过就开始写业务。
