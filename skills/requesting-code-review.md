# Skill: requesting-code-review（代码审查）

## 何时使用

- 一个功能 / 模块 / 修复完成后准备合并或提交前；
- 上线前最后一次审查；
- 用户说"帮我审一下代码""看看有没有问题""检查一下风险"。

## 解决什么问题

用另一个视角检查风险，避免写代码的视角盲区。让 AI 不只看"功能跑没跑通"，还看长期可维护性和潜在风险。

## 执行流程

1. **明确审查范围**：列出本次审查覆盖的文件、提交、模块，以及不审查的部分。
2. **多维审查**：按以下维度逐项检查，每项给"通过 / 风险 / 严重"判断 + 证据：

   **结构维度**
   - 目录放置是否符合项目架构设计文档 / 框架规范？
   - 接口层 / 业务层 / 数据层是否分层清楚？
   - 有没有重复实现（同类 UI 出现超过 2 次未封装、同类工具函数到处复制）？

   **规则维度**
   - 有没有违反 Agent 宪法？
   - 有没有违反真源文档（接口规则、错误处理、目录责任）？
   - 有没有绕开框架自带能力自己手写一套？

   **安全维度**（涉及后端时必查）
   - 接口输入是否默认不可信？关键参数（金额、角色、用户 ID、订单归属）是否后端重新校验？
   - 是否存在垂直越权 / 水平越权？
   - 密码是否哈希存储？弱密码是否拦截？
   - 是否存在 SQL 注入 / 命令注入 / 模板注入风险（字符串拼接用户输入）？
   - 是否有过度防御 / 死代码？

   **数据维度**
   - 数据库结构变更是否先改 schema/migration？
   - 字段是否符合三大范式（或反范式有说明）？
   - 敏感数据（密码、密钥、隐私字段）处理是否到位？
   - 软删除 / 唯一键 / 索引是否合理？

   **可维护性维度**
   - 命名是否清楚？是否有 magic number / 硬编码？
   - 是否有"以后可能用得上"的提前封装？
   - 日志 / 错误处理 / 测试是否齐全？

3. **风险分级**：
   - 🔴 严重：必须修，不修不能合并（安全漏洞、数据丢失风险、规则违反）
   - 🟡 风险：建议修，影响后期维护（结构混乱、重复代码、硬编码）
   - 🟢 建议：可选优化（命名、注释、小重构）
4. **输出整改清单**：每条说明位置、问题、建议、优先级。
5. **回归确认**：整改后重新走一遍相关维度，确认问题真的关闭。

## 输出

- 代码审查报告：范围 / 各维度结论 / 风险分级清单 / 需整改项 / 通过项。

## 约束

- ❌ 不只看功能跑没跑通；
- ❌ 不用"看起来没问题""基本可以"等模糊词；
- ❌ 不擅自顺手重构不在审查范围内的代码；
- ❌ 不把"以后可能用得上"的封装当作加分项；
- ❌ 严重风险项不能"建议优化"，必须"阻断合并"。
