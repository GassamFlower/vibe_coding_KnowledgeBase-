# Skill: writing-plans（编写开发计划）

## 何时使用

- 立项文档已生成，准备进入开发；
- 用户说"帮我写开发计划""接下来怎么做""拆一下任务"；
- 一个新模块准备开始前。

## 解决什么问题

把需求拆成可执行、可验收的任务，避免 AI 一口气写一堆功能导致项目漂移。

## 执行流程

1. **回看真源**：先读取立项文档、技术选型文档、Agent 宪法、相关真源文档，不要凭记忆规划。
2. **列阶段**：把要做的事情按阶段拆分，每个阶段是一个可独立验收的里程碑。典型阶段：
   - 项目初始化
   - 前端骨架
   - 数据库设计
   - 后端骨架
   - 后端架构验收
   - 业务模块（一个一个加，不要批处理）
   - 上线前验收
3. **拆任务**：每个阶段拆成具体任务，每个任务说明：
   - 做什么
   - 涉及哪些文件 / 模块
   - 依赖哪些前置任务
   - 验收标准（必须可检查）
   - 预计触发的 Git 提交点
4. **标优先级**：用 P0（必须有）/ P1（应该有）/ P2（可以有）标记。
5. **标注边界**：明确这一阶段不做什么，防止范围蔓延。
6. **指明规则引用**：每个任务关联要遵守的真源文档 / Agent 宪法条款 / 框架规范。
7. **生成计划文档**：写入项目根目录，例如 `plans/v1-plan.md`。

## 输出

- 开发计划文档，包含：阶段、任务、优先级、依赖、验收标准、不做什么、规则引用。
- 一份"下一阶段任务清单"（短，3-5 项），方便用户直接说"开始下一项"。

## 约束

- ❌ 不写业务代码，只写计划；
- ❌ 不一次性把所有阶段细到天级别（粗到阶段 + 任务即可）；
- ❌ 验收标准不能模糊（"完成登录功能"不算，"注册/登录/修改密码 3 个接口通过测试 + 提交 Git"才算）；
- ❌ 不要把"以后可能用得上"的东西塞进当前计划。
