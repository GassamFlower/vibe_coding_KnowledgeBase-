# Skill: verification-before-completion（完成前验证）

## 何时使用

- AI 准备说"已完成""做好了""可以了"；
- 用户问"做完了吗""能用了吗"；
- 任何一个阶段、功能、修复准备收尾。

## 解决什么问题

防止 AI 写完代码就直接宣布完成。让"完成"必须有可检查的证据，而不是结论。

## 执行流程

1. **暂停宣布完成**：在任何"已完成"声明之前，先走完本流程。
2. **列验收项**：根据任务范围列出验收清单。至少包含：
   - 功能项：能不能跑？核心路径走通了吗？
   - 规则项：有没有违反 Agent 宪法 / 真源文档 / 框架规范？
   - 安全项：接口输入校验、权限校验、敏感数据处理是否到位（涉及后端时）？
   - 数据项：数据库结构变更是否先改 schema/migration？
   - 接口项：返回示例是否覆盖成功 / 失败 / 边界场景？
   - 日志项：关键操作有没有日志？
3. **逐项交证据**：每项必须给出可检查的证据：
   - 文件路径 + 行号
   - 执行命令 + 实际输出
   - 请求 / 响应示例
   - 测试用例 + 通过结果
   - 日志截图或文本
4. **标记未验证项**：没有证据的项目必须标记为"未验证"，不允许写"基本完成""应该没问题"。
5. **回归检查**：确认本次改动没有破坏已有功能（至少跑一遍相邻功能）。
6. **更新真源文档**：如果本次改动涉及规则变更（目录、接口格式、错误码、权限规则等），先更新真源文档再宣布完成。
7. **触发 Git 提交**：通过验收后，按 Git 阶段提交流程提交一次。
8. **输出验收报告**：按"验收项 / 是否通过 / 证据位置 / 未通过原因 / 下一步"输出。

## 输出

- 验收报告（Markdown 或对话内）；
- 已通过的 Git 提交。

## 约束

- ❌ 不在没证据的情况下说"已完成"；
- ❌ 不用"基本完成""应该没问题""理论上可以"等模糊词；
- ❌ 不跳过回归检查；
- ❌ 不在不更新真源文档的情况下改规则；
- ❌ 不在未通过验收时提交 Git。

> 你不需要完全看懂证据，但每一步都要让 AI 留下可检查的结果。没验证就是没验证。
