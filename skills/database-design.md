# Skill: database-design（数据库设计）

## 何时使用

- 用户说"帮我设计数据库""建几张表""这个功能需要什么表"；
- 前端骨架跑通，准备进入数据库阶段；
- 已有数据库结构混乱，需要重新梳理。

## 解决什么问题

防止 AI 上来就建表。先把业务对象、关系、范式、字段规范定清楚，再生成设计文件，最后才建表。

## 执行流程

1. **从前端反推数据**（前端已有时优先用这步）：
   - 列表 → 映射数据字段
   - 详情页 → 推导完整结构
   - 表单 → 定义提交数据
   - 按钮 → 确定操作行为
   - 状态 → 指明数据流转
2. **从业务流程找对象**：让 AI 列出项目核心业务对象（用户、订单、商品、文章、权限、日志等），先不建表。
3. **确认对象关系**：让 AI 解释每张表代表什么、哪些表关联、1:1 / 1:N / N:N、关联字段放哪张表、为什么这样设计。
4. **选数据库服务**：让 AI 给唯一推荐（普通业务优先 MySQL/PostgreSQL；Redis 仅在确有缓存/队列/高并发需求时引入）。
5. **按三大范式设计表结构**：
   - 第一范式：字段原子化
   - 第二范式：字段围绕当前表主对象
   - 第三范式：字段之间不互相依赖
   - 反范式必须有业务原因并说明
6. **规范常见字段**：
   - `id` / `created_at` / `updated_at` 自动维护
   - 不能物理删除的数据加 `deleted_at` 软删除
   - 密码用 `password_hash`，禁止明文
   - 金额用整数（分）或精确数字类型，说清货币和精度
   - 状态字段定义枚举值
   - 字段名用英文下划线
7. **先生成设计文件再建表**：
   - 生成 `schema.sql` 或 migration 文件
   - 核对业务对象、表关系、字段、三大范式
   - 确认后再执行建表
   - 后续变更也先改 schema/migration 再执行
8. **写进实施计划**：业务对象 / 关系 / 主数据库 / 是否 Redis / 核心表 / 字段规范 / 唯一性 / 关联 / 软删除策略 / 范式校验 / 验收标准。
9. **跑通基础结构**：能连接 / 能建表 / 能 migration / 能插入和查询测试数据。
10. **触发 Git 提交**：把设计文件和基础结构作为一次稳定提交。

## 输出

- 数据库设计文件（`schema.sql` 或 migration）；
- 数据库实施计划；
- 已通过的 Git 提交。

## 约束

- ❌ 不直接连数据库建表，跳过设计文件；
- ❌ 不把所有数据塞进一个大字段；
- ❌ 不在表关系没想清楚时开建；
- ❌ 不明文存储密码；
- ❌ 不用浮点数存金额；
- ❌ 不在没有 migration 记录的情况下手动改字段；
- ❌ 不在不说明原因的情况下做反范式设计。
