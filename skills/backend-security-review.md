# Skill: backend-security-review（后端安全审查）

> 来源：Vibe Coding 脚手架化 + 腾讯 Harness 工程实践 + 补充集合
> 五项核心：密钥安全管理 | 敏感日志脱敏 | 依赖漏洞巡检 | 数据库权限管控 | 自动化证据验收

## 何时使用
- 业务功能开发后，重要功能上线前
- 用户说"检查一下安全""有没有漏洞""安全审查"
- 上线门前必做

## 解决什么问题
防止 AI 用"做了安全处理"糊弄过去。安全是分散在请求处理每一步的关卡，每道关卡只检查一个方面，少一条就是一个口子。

---

## 一、配置文件安全

### 检查内容
后端安全最先看配置文件。代码里不要写：数据库地址、各类密钥、接口 token、初始密码。

### 规则
- **密钥隔离**：真正的密钥绝不能出现在代码里
- **示例占位**：示例配置可以放进 `.env.example`，告诉别人"这里该填什么"，而不是把真东西交出去
- **环境注入**：代码读配置时应从环境变量或框架推荐的方式获取
- **密钥重置**：如果曾写入真实密钥到代码，标记是否需要更换

### Prompt
> 请检查当前后端项目是否存在敏感配置写死在代码里的情况。重点检查：数据库地址、数据库密码、登录令牌密钥（如 JWT、第三方接口密钥、管理员初始密码、支付或通知相关密钥）。
> 请输出：配置名称、当前所在位置、是否来自环境变量、是否出现在代码或仓库中、是否登记在配置示例文件（如 .env.example）、是否需要更换。

## 二、日志安全

### 检查内容
日志是为了排查问题，不是把每条请求都原样存下来。看三件事：准许记录范围（哪些能记）、禁止审计明细（哪些不能记）、敏感项脱敏处理。

### Prompt
> 请审查当前后端日志安全。请说明：
> 1. 请求日志、错误日志、数据库日志分别记录哪些内容
> 2. 是否会记录密码、token、密钥、手机号、身份证号、支付信息等敏感数据
> 3. 如果会记录请说明如何脱敏或移除
> 4. 请给出一条正常请求日志和一条错误日志的示例

---

## 三、依赖安全

### 检查内容
第三方库和 SDK 的版本、漏洞、升级建议。重点标出网络请求、登录认证、加密、文件上传、数据库连接的核心依赖。

### Prompt
> 请列出当前后端项目用到的所有框架、第三方库和 SDK 以及各自的版本。
> 请逐项说明：有没有已知的安全漏洞或安全公告，当前版本是不是官方建议升级的版本，如果需要升级会影响哪些现有功能。
> 请重点标出那些负责网络请求、登录认证、加密、文件上传、数据库连接的依赖。

---

## 四、数据库安全

### 检查内容
重要数据不能随便查、随便改、随便删除。

### 规则
- **数据资产分类**：明确哪些是重要数据，哪些接口会读写删它
- **四重安全验证**：身份校验 → 权限校验 → 数据归属校验 → 业务规则校验
- **参数传参防御**：严禁直接根据前端传入的 ID 修改或删除数据
- **软删除**：重要数据不物理删除，用 `deleted_at` 标记
- **事务**：多张表一起变动时，必须用事务。很多操作不是只动一张表
- **审计日志**：关键操作（修改/删除）记录操作人、时间、变更内容

### Prompt
> 请审查当前后端数据库操作安全，请按数据表或数据模型列出：
> 1. 哪些是重要数据
> 2. 哪些接口会读取/修改/删除它
> 3. 每个操作是否经过身份校验、权限校验、数据归属校验和业务规则校验
> 4. 是否存在直接根据前端传入 ID 修改或删除数据的情况
> 5. 是否有软删除、事务或审计日志要求
> 6. 多表操作是否使用了事务

## 五、请求层五道关卡

### 1. 身份认证
- 请求是谁发的？有没有登录？
- Token / Session 验证机制是否完善？
- 未登录请求是否被正确拦截？

### 2. 权限控制
- 这个人能不能做这件事？
- 普通用户能不能调管理员接口？
- 水平越权：用户 A 能不能操作用户 B 的数据？

### 3. 输入校验
- 前端传的数据可不可信？
- 必填字段有没有校验？
- 数据类型和格式有没有校验？

### 4. 数据归属
- 这条数据是不是属于他自己？
- 修改/删除操作有没有做归属校验？

### 5. 注入防护
- 用户输入会不会被当成命令执行？
- SQL 是否使用参数化查询？
- 有没有字符串拼接用户输入？

### Prompt（上述五道关卡综合）
> 请对当前后端做安全审查，逐项检查以下 5 道关卡：
> 1. 身份认证  2. 权限控制  3. 输入校验  4. 数据归属  5. 注入防护
> 每项说明：检查了什么、在哪里检查的、是否通过、未通过的原因。
> 特别检查水平越权：用户 A 能不能操作用户 B 的数据？

## 六、接口验收测试

### 验收标准
功能可用、错误拦截、权限管控、边界校验、数据一致、日志追溯。

### 测试角色
- 普通用户
- 管理员
- 系统管理员
- 未登录/异常请求者

### Prompt
> 请对本次新增或修改的接口，做安全和业务双重验收。请输出：
> 1. 正常请求示例 + 响应
> 2. 参数错误示例 + 响应
> 3. 未登录示例 + 响应
> 4. 无权限示例 + 响应
> 5. 访问他人数据示例 + 响应
> 6. 数据库写入或修改前后的证据
> 7. 日志记录证据
> 请说明哪些测试是通过接口完成，哪些是通过数据库检查完成，没有实际验证的项目标记为未验证。

---

## 附加检查项
- 密码强度校验（注册/登录/修改/重置/创建管理员）
- 密码哈希存储（禁止明文）
- 登录失败次数限制

---

## 反模式
- ❌ "做了安全处理"（太笼统）
- ❌ 只检查了登录，没检查权限
- ❌ 忘了水平越权测试
- ❌ 有字符串拼接用户输入
- ❌ 数据库操作没加事务
- ❌ 真实密钥出现在代码里
- ❌ 日志记录了密码/token
