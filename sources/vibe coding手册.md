## 一、Agent 宪法
### 1、 什么是 Agent 宪法？
是写给 AI 的长期行为准则：

+ 项目边界是什么？
+ 代码规范是什么？
+ 哪些文件不能乱动？
+ 哪些操作必须谨慎？
+ 完成任务之前需要怎么验证？

**一句话：**建立 AI 核心规范/约束行为与边界/写给 AI 的项目规矩

### 2、宪法文件通常放在那里？
不同 IDE 工具，有不同位置和不同文件名。例：codex 的 AGENT.md，Claude code 的 CLAUDE.md。

不用全部记住或者去研究格式，可以直接问：帮我创建一份当前工具会读取的 Agent 宪法，并放到正确位置。

### 3、Agent 宪法为什么重要？
为了让 AI 更稳定，AI 优化会局部问题，就是只看眼前问题，而实际项目是改动一个地方，可能会影响到后面其他问题。

### 4、Agent 宪法不是越多越好
上下文越多，重点也容易被稀释。宪法需要短、硬、清楚。

+ 写长期有效的原则：<font style="color:rgb(15, 17, 21);">要提炼出那些</font>**<font style="color:rgb(15, 17, 21);">不随场景、时间、技术迭代而改变的基本价值观和底层原则</font>**<font style="color:rgb(15, 17, 21);">。</font>
+ 写禁止事项和验收标准：1） <font style="color:rgb(15, 17, 21);">明确列出 Agent </font>**<font style="color:rgb(15, 17, 21);">绝对不允许做的事</font>**<font style="color:rgb(15, 17, 21);">，一旦触犯就是严重违规。这些是刚性约束，通常用于安全、合规、伦理底线。2）从“输出合格”的角度定义，</font>**<font style="color:rgb(15, 17, 21);">当任务完成时，必须满足什么条件才叫“通过了验收”</font>**<font style="color:rgb(15, 17, 21);">。这是结果导向的质量门槛。</font>
+ 写可执行、可检查、可判断的规则：<font style="color:rgb(15, 17, 21);">这是把抽象原则“翻译”成</font>**<font style="color:rgb(15, 17, 21);">具体行动指令</font>**<font style="color:rgb(15, 17, 21);">的核心步骤。一条规则如果无法执行、无法检查、无法明确判断是否达标，那它就是一句空话。</font>

```plain
# 项目 Agent 宪法 (Vibe Coding 总纲)

你是我项目中的 AI 编码助手。在生成任何代码、修改任何逻辑时，你都必须首先通过这部宪法的审查。宪法高于一切编码规范。

## Ⅰ. 永恒的安全与诚信
- 禁止代码中包含任何真实凭证。所有密钥、令牌均从环境变量或安全仓库读取，变量名必须包含 `SECRET_` 前缀。
- 禁止生成任何形式的歧视、仇恨、成人内容。用户输入必须经过 `safety_check()` 清洗，该函数返回值必须为布尔 `true` 时才继续处理。
- **验收标准：** 安全测试函数 `test_safety()` 必须能检测出 10 种以上的越狱攻击模式。

## Ⅱ. 诚实与边界
- 当知识库无匹配或模型不确定时，必须输出“坦诚声明”而非猜测。实现 `respond_with_uncertainty()` 方法，内容为：“抱歉，我目前无法提供确定的答案，建议您 [建议动作]。”
- **验收标准：** `respond_with_uncertainty()` 方法不能接受任何会填充虚假详情的参数，如 `fake_specifics` 等，如果存在这样的参数必须删除。

## Ⅲ. 执行的铁律
- **澄清优先**：任何缺少必填字段的请求，必须触发 `ask_clarification()` 而不是用默认值。`ask_clarification()` 必须列出缺少的字段名。
- **确认即阻断**：任何具有副作用（网络发送、文件删除、数据库写入）的函数，其内部第一行必须是 `if not is_user_confirmed(): return PENDING`，等待确认信号。
- **熔断即终止**：所有网络调用或工具调用必须包裹在 `with_retry(max_attempts=2)` 装饰器里。超过尝试次数必须调用 `request_human_help(error_log)`，并记录日志。

## Ⅳ. 你作为代码生成器的承诺
- 每次输出代码后，必须附带一个“宪法遵守检查清单”，简要说明这段代码是如何满足上述各条的。
- 优先在代码中插入明确的断言和类型检查，让宪法规则“可自行校验”。
```

## 二、Skill
解决三大核心问题：保持宪法简洁、减少过程漂移、 复刻专家经验

skill: 技能包，可复用的专业流程。

### 1、与宪法的区别
 Agent 宪法（必须这样做）：

    - 核心：管长期原则
    - 标准：短、硬、清楚
    - 性质：长期有效规则

Skill 技能（具体怎么做）：

    - 核心：管专项流程
    - 标准：具体操作指南
    - 性质：任务级指令

## 三、前端
### 1、前端基本词汇了解
<u>UI-用户界面：</u>按钮、表单、列表、弹窗、导航、页面结构

<u>UX-用户体验</u>：操作流畅顺滑、流程清晰逻辑、导航明确直观、即时状态反馈

<u>界面风格：</u>视觉界面感受-科技、商务、可爱等等

<u>布局：</u>页面内容的组织方式，内容放在哪里、主次怎么安排、左侧要不要菜单、右侧要不要详情、列表和表单怎么组合

<u>组件：</u>页面里可以复用的小模块，如：按钮、输入框、卡片、表格、弹窗、导航栏、标签页

<u>交互：</u>用户操作之后发生什么，如：按钮点击后的反馈、提交失败后的提示、加载中的状态、删除之前的确认

<u>响应式设计：</u>页面在不同屏幕尺寸下都能正常显示

### 2、前端场景：
网页前端、App 前端、小程序前端以及硬件 GUI。AI 将场景规范讲清楚，再去写页面会好很多。

### 3、不同前端对应不同技术栈
底层能力：

早期前端底层技术：HTML、CSS、JavaScript 

现代工程化能力：

现代前端技术：React、Vue、Svelte 等前端框架

用 TypeScript 提升代码可靠性、用 Vite、Nextjs、Nuxt 这类工具和框架管理项目结构，构建、路由、渲染和工程化

### 4、前端骨架：
1. 整体设计风格
    - 苹果系统里的玻璃拟态、液态玻璃、半透明模糊背景
    - Google 的 Material Design，层级清晰、组件规范完整
    - 后台管理系统常见的企业级管理台风格，信息密度搞、表格多、操作效率优先
    - SaaS 产品常见的现代极简风格，卡片清楚，按钮克制，留白适中
    - AI 产品常见的科技感风格，深色背景、渐变、发光元素、动效更多 
    - 内容社区常见的信息流风格，阅读舒适、内容层级清楚 
2. 前端技术方案
    - 页面怎么放？路由怎么写？目录怎么分？项目怎么启动？构建和部署怎么处理？
    - web 方案：React/Vue、Angular、Next.js/Nuxt
    - 小程序：微信原生框架、Taro、uni-app
3. UI 组件库

解决：界面元素怎么统一

    - 界面元素：按钮 | 输入框、弹窗 | 表格、菜单 | 卡片
    - 常见组件库：Ant Design、Element Plus、Naive UI
    - 移动/扩展：MUI|Vant、NutUI|Mantine

AI prompt：遵循官方规范、优先复用组件、禁止重复造轮子、对齐最佳实践  ——给 AI 画边界

4. 目录结构
    - 页面存放（在哪里）
    - 组件封装
    - 接口定义
    - 工具函数
    - 样式资源
+ 模块边界要清楚：登录、用户、订单、内容、设置等模块，这些功能区域，不要全部混在一起。（AI 后续该某个功能时，才不会乱动）
5. 组件复用规则

按钮、输入框、卡片、表格、弹窗、提示框、导航栏等，这些会反复出现，优先用组件库，组件库不够用再基于组件库封装项目自己的业务组件（判断标准：只要相似率比较高，且出现过 2 次，就可以考虑封装成组件）。

6. 样式系统	
    - 设计变量：颜色、字号、行高、间距、圆角、阴影、边框

所有“颜色、字号、行高、间距、圆角、阴影、边框”都必须统一设计 token（样式规范）里取

    - 多语言规范  
    - 主题规则
7. 实施计划

前端技术方案、UI 组件	库选型、目录结构规范、模块解耦边界、核心组件封装、设计 token 定义、国际化与主题预留。

## 四、后端
### 1、后端是什么？
后端就是业务支持核心系统，通常做的事情：

+ 接收前端请求
+ 处理业务逻辑
+ 读写数据库
+ 判断用户权限
+ 返回处理结果

### 2、后端核心职责，三件事：
1. 数据：用户信息、订单信息、文章内容、商品列表、任务记录、聊天消息等，这些都需要存储起来，同时也需要在合适的适合被查询处理。
2. 业务逻辑：项目真正的处理流程-如，谁能创建订单？订单什么适合能取消？会员能不能访问某个功能？库存不足时怎么处理？
3. 安全：登录认证、数据访问、内容权限、敏感验证、接口防护、密钥安全。

### 3、前端与后端配合
二者之间通常通过接口通信：前端发起请求，后端处理请求，再把结果返回前端-就是一个对话过程。这里就是接口所做之事。

API 接口规范-核心四要素：请求地址、请求方法、参数列表、响应数据。

vibe coding 提示词：接口协议，二者之间不要用模糊指令“帮我写个接口”，说清楚接口解决什么问题，前端会传什么？后端会校验什么？数据会存到哪里？成功返回什么？失败返回什么（业务目标、请求参数、逻辑校验、存储方案、成功响应、异常处理）

### 4、后端常见概念
基本词汇了解：

+ 接口，是前端和后端通信的通道
+ 数据库，是用来长期保存数据的地方
+ 数据表，是数据库里存放某一类数据的接口
+ 字段，是一条数据里的具体信息
+ 业务逻辑，是项目真正的处理流程
+ 权限，是判断一个用户能不能访问某些功能，查看某些数据、执行某些操作
+ 鉴权，是判断用户是谁以及当前他有没有资格访问接口
+ 校验，是检查用户提交的数据是否合理，如手机号格式，金额填写
+ 日志，是系统运行时留下的记录
+ 部署，是把后端服务放到服务器上

### 5、后端常见技术栈
核心要素：语言+框架+数据库+部署环境

### 6、后端最容易翻车的地方
AI 后端核心自查：

+ 权限系统：权限接入
+ 数据校验：逻辑校验
+ 错误处理：统一拦截
+ 接口标准：规范统一
+ 库表结构：架构优化（数据库结构是否合理）
+ 操作日志：全量记录

### 7、后端需求提炼
数据处理：明确业务核心数据流

角色定义：划分用户访问权限

业务操作：梳理核心业务流程

权限控制：明确操作授权范围

数据存储：确定持久化方案

前端业务：设计 API 交互契约

窍门：自己说不清楚，让 AI 反过来问自己，如，让 AI 根据前端，帮你整理数据模型和接口设计，让其根据页面流程推导需要哪些数据表、哪些接口、哪些权限，给出后端方案

### 8、后端副骨架（从业务角度出发）
1. 后端定义：就是一种程序运行逻辑-项目业务认知（核心要素）和个人思维能力（能力支撑）。凡是用户看不见，但负责业务逻辑、数据处理、权限校验、接口调用、任务执行的部分，都先归到后端。
2. 后端场景：
    1. 小脚本场景：批量整理文件、处理表格、生成报表、压缩图片、API 调用、定时任务（无需完整项目，单脚本搞定）
    2. 项目级后端：用户系统、订单系统、内容发布、权限管理、支付回调、后台审核、数据统计等。
        1. 业务核心规则：身份与交易、流转与合规、高权控制
        2. 数据流转：前端请求、安全校验、逻辑处理、持久化存储
        3. 接口协作
3. 后端语言
4. 框架

框架比语言更影响项目质量。语言决定用什么写？框架决定按什么规矩写？<u>AI 选型指南：</u>基于已选后端语言推荐一个适合当前项目的后端框架，说明它适合什么场景？官方目录规范是什么？哪些能力框架自带？哪些能力需要额外 SDK 或库？优先选择成熟、文档完整、社区活跃、AI 熟悉度高的方案。

框架决定开发规矩：

    - 目录划分
    - 接口设计
    - 数据链接
    - 异常处理
    - 权限管控
    - 日志审计

### 9、讨论结果要写进真源文档
后端语言、框架、API、业务规则确认后，写进真源文档，后面 AI 搭建后端骨架之类操作按照这份文档来

## 五、数据库
> 数据库是项目核心数据底座
>

vibe coding 做项目项目建议顺序（非传统开发模式，适用于小白）：先 UI 前端——>数据库——>再后端

### 1、根据前端（UI 反推确定精准架构）
页面列表：映射数据字段（暗示需要哪些数据字段）

详情页面：推导完整结构（暗示数据对象的完整结构）

功能表单：定义提交数据（暗示用户提交什么数据）

交互按钮：确定操作行为（暗示需要哪些操作）

页面状态：指明数据流转（暗示数据流程过程）

prompt：这个列表需要什么表？这个详情页需要哪些字段？这个表单提交后要保存到哪里？这个状态从哪里来？这个按钮操作会改哪条数据？

### 2、数据库服务
数字库、表、字段是什么关系？

数据库有哪些？不同场景用什么数据库？分类的话可以怎么分？

### 3、数据库常见设计概念
主键、自增、外键、索引、查询、写入、更新、删除、迁移

### 4、数据库设计“雷区”
1. 表结构设计随意
2. 所有数据都塞进一个大字段
3. 表之间关系设计不清楚
4. 没有数据校验，否则脏数据进入数据库
5. 没有迁移记录，项目迭代时数据库结构一定会变化，如果没有迁移记录，后续开发、测试等环境不一致
6. 删除太随意，需要考虑是否有历史记录等
7. 密码不能明文存储
8. 唯一性和索引没设计好

AI prompt：当前用的是什么数据库服务？创建了哪个数据库？数据库里有哪些表？每张表有哪些字段？字段分别代表什么？表和表之间怎么关联？这个业务类型为什么适合这个数据库？为什么不选其他数据库？哪些字段需要唯一键？哪些 ID 适合自增？哪些字段需要索引？哪些操作需要事务？哪些数据不能直接删除？敏感信息怎么保存？

### 5、数据设计规范
#### 1、业务流程里找对象
让 AI 根据页面、用户流程、核心功能找一下有哪些核心业务对象 

#### 2、确认对象关系
表和表之间的关系：表代表的对象，表之间的关联关系，关联基数（1 对 1，1 对多，还是多对多），关联字段所在表位置，核心设计利用（为什么要这么设计）。这一步是却确认 AI 理解业务

#### 3、数据库选型（确定数据库服务）
与 AI 探讨：主数据库用什么？为什么用它？要不要用 Redis？本地开发和线上部署怎么处理？

#### 4、设计表结构和字段规范
三大范式去做（告诉 AI 数据库设计尽量符合前三凡是）：

1. 第一范式：单一字段 存储单一含义
2. 第二范式：字段要围绕当前这张表的主对象
3. 第三范式：字段之间不要互相依赖

#### 5、常见字段要提前规范
ID、名称、联系方式等等；字段类型规范，关键业务字段确认如果不清楚，与 AI 对话，根据业务来确认。

##### 6、先保留设计文件，再建表
数据库设计确认后，不要直接让 AI 连数据库建表。先让 AI 生成数据库设计文件，生成文件后，永久保留并提交 Git 确立唯一的结构来源。

更稳的流程：先生成数据库设计文件，再核对业务对象、表关系、字段和三大范式，确认没问题后再执行建表；后续改字段之类的，也先改设计文件，再执行变更。（这里也可以写进 agent 宪法）

#### 7、实施计划
前面几项确认后，再写进实施计划，至少要写清楚：业务对象、对象关系、主数据库、缓存架构（要不要 Redis）、核心表、字段规范（每张表负责什么业务、字段名、字段内容是否必填）、数据约束（哪些字段是唯一，哪些可以为空）、表间关联、删除策略、范式校验、验收标准
