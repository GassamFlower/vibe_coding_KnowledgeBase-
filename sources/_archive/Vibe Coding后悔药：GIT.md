# Vibe Coding后悔药：GIT

## 一、Git 是什么

你可以把 Git 理解成项目里的版本存档工具。

它不是用来写代码的，而是用来记录代码每一次重要变化的。

比如 AI 帮你搭好了项目骨架，提交一次。AI 帮你做完登录功能，提交一次。AI 帮你接好了数据库，再提交一次。

后面如果 AI 把项目改乱了，你就可以回到之前某一个稳定版本。

这就像打游戏一样。

遇到难打的 boss，或者不确定自己能不能打过的时候，最好先存档。打赢了继续往前走，打输了就从存档点还原，避免前面的资源和进度全都浪费掉。

用 AI 写代码也是一样。

每次让 AI 做比较大的修改之前，都应该先确认当前项目有没有一个稳定存档。

## 二、为什么 AI 项目更需要 Git

因为 AI 改代码很快，而且一次可能改很多文件。

你如果看不懂代码，就更难判断它到底改了什么。

没有 Git，AI 改坏了，你只能继续让它补。补着补着，问题越来越多，项目越来越乱。到最后你甚至会放弃整个项目。

而一旦有了git，你不仅能知道它当前改了什么，一旦当前阶段出了问题或者效果不满意，你可以直接回退。

所以越是刚开始用 AI 做项目，越要早点用 Git。

不要等项目已经乱了，才想起来要不要保存版本。那时候很多问题已经混在一起了，想救会非常麻烦。

## 三、要养成阶段提交的习惯

Git 最重要的不是装上，而是要会提交。

规则很简单：每完成一个明确阶段，就提交一次。

当然，也不是每改一行就提交。而是项目完成一个小阶段，并且能正常运行，你确认结果可以接受，就提交一次。

比如项目创建完成，提交一次。目录结构搭好，提交一次。首页能打开，提交一次。接口能响应，提交一次。数据库能连接，提交一次。一个核心功能做完，再提交一次，等等。

你还可以把这条规则写进 agent 宪法里：

每完成一个明确阶段，必须先说明本次改了哪些内容，验证是否通过，然后再提交一次 Git 记录。

这样 AI 就不会只顾着往前写，而是会在每个阶段停下来做总结和存档。

## 四、Git 和 GitHub、码云不是一回事

这里要分清楚。

Git 是本地版本管理工具，你可以理解成：Git 是你电脑里的存档工具，是一个版本管理软件。不光是代码，甚至你写一篇文章你都可以用git来管理文章版本。

而GitHub 和码云则是把这些存档上传到网上保存的地方，就像你用常用的网盘，区别在于网盘不能做版本管理。

你还需要知道的是：远程代码仓库不止这些，还有很多，只是博主最常用的就是这两个

如果你人在国内，想访问更顺，可以用码云，也就是 Gitee。

如果你做的是开源项目，或者希望更多人看到，可以用 GitHub。或者直接一个项目配置两个远程仓库，一个推 GitHub，一个推码云。

最后，关于这些你还要知道的是：Git 本身就是一个版本管理工具，即便你不想把代码推送到远程，只要你在电脑上安装了git做了git版本管理，你也是可以直接使用项目提交或代码回退功能的。

所以不是说你用git就一定要码云或者github，其实远程仓库在以前最大的用途是：假设你在多个电脑异地工作、多个人一起做一个项目，或者你电脑损坏之后代码不会丢失。

## 五、推远程之前，先确认哪些东西能公开

如果项目只是自己用，或者还没准备公开，远程仓库就应该设成私有。

私有项目，就将仓库权限设为私有即可，这个很简单。

可是，如果你的项目准备开源，也不是非要把整个项目文件夹全部开源出去。

因为项目里不只有代码。

它可能还有立项文档、商业计划、产品规划、agent 宪法、内部策略、真实 case 分析。

尤其是现在这个全民 AI 干代码的时代，代码、页面、功能，甚至一整套项目结构，要复制都很快。

真正重要的，反而是你的想出来的 case。

也就是你为什么做这个项目，你面对的真实场景是什么，你怎么理解用户需求，你怎么设计产品路线，你怎么约束 AI 按你的思路去开发。

这些东西，很多时候比代码本身更值钱。

所以开源项目真正应该公开的，是你愿意让别人看到、使用、学习和参与的部分，比如代码、README、安装方式、使用说明、贡献指南。

但商业计划、内部规划、真实 case、agent 宪法这些内容，除非你明确决定公开，否则不要让它们跟着代码一起进入开源仓库。

推远程之前，最好先让 AI 列出文件清单：

这次准备提交哪些文件？

哪些文件会推到远程？

有没有不该公开的内容？

你确认没问题，再让它推送。

## 六、最后

用 AI 写代码，Git 不是可有可无的工具。

它是你的后悔药，也是项目的存档点。

没有 Git，AI 每一次大改都可能变成不可逆操作。有了 Git，你至少可以知道每一步发生了什么，也能在项目出问题时退回稳定版本。

所以正式让 AI 写项目之前，至少做三件事。

第一，安装 Git。

第二，养成每完成一个阶段就提交一次的习惯。

第三，推远程之前，确认仓库是私有还是公开，确认哪些文件能推，哪些文件不能推。
