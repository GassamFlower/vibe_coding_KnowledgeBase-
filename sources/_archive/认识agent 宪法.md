# 认识agent 宪法

## **一、什么是 agent 宪法？**

agent 宪法，本质上是一份写给 AI 的长期行为准则。

它不是某一次对话里的临时要求，而是这个项目长期有效的工作原则。每次 AI 进入项目之前，都应该先知道这些规矩：项目边界是什么，代码规范是什么，哪些文件不能乱动，哪些操作必须谨慎，完成任务之前需要怎么验证。

很多新手用 AI 写代码，习惯直接上来一句：帮我加个功能、帮我修个 bug、帮我优化一下页面。AI 当然会做，但如果你没有提前告诉它项目规则，它就只能根据当前上下文和模型自己的习惯来猜。

猜对了还好，猜错了就可能开始乱改文件、乱抽象、乱兼容、乱补丁。你以为它是在帮你推进项目，实际上它可能正在一点点把项目写乱。

所以 agent 宪法的意义，就是先给 AI 立规矩。不是等它犯错之后再一句一句纠正，而是在项目开始之前，就先告诉它：这个项目应该怎么做，哪些事情绝对不能做，遇到冲突时优先听谁的。

## **二、宪法文件通常放在哪里？**

不同 AI IDE 或 Agent 工具，有不同的规则文件格式。

在 Codex 里，常见的是 `AGENTS.md`。在 Cursor 里，比较常见的是 `.cursor/rules`\`目录下的规则文件，以前也有很多人用 <span style="color: rgba(143, 143, 143, 1;">`.cursorrules`</span>，但现在更推荐使用新的 rules 方式。在 Claude Code 里，常见的是 \`CLAUDE.md\`，也可以放在 <span style="color: rgba(143, 143, 143, 1;">`.claude/CLAUDE.md` 里。

![](attachment/1.png)

当然，还有一些工具会使用自己的规则文件。你不用死记这些名字，也不用一开始就研究所有格式。如果你不确定当前 IDE 应该用什么文件、放在哪个目录，最简单的方式就是直接问它：帮我创建一份当前工具会读取的 agent 宪法，并放到正确位置。

你真正需要理解的是：这些文件名虽然不同，但目的相同。它们都是为了让 AI 每次进入项目时，先知道这个项目的规矩。

## **三、agent 宪法为什么重要？**

agent 宪法最重要的作用，不是让 AI 更听话，而是让 AI 更稳定。

AI 最大的问题之一，就是很容易根据当前对话做局部最优，通俗点说，就是偷懒。

你让它修一个按钮，它可能只盯着按钮；你让它修一个接口，它可能只盯着接口；你让它加一个功能，它可能只想着怎么把功能加上去。

但真实项目不是这样。一个前端改动可能涉及组件规范，一个接口改动可能涉及权限模型，一个功能改动可能涉及数据库设计、错误处理、测试覆盖和后期维护。每一次改动都不是孤立的。

如果没有 agent 宪法，AI 很容易只解决眼前问题，却破坏长期结构。这也是很多人用 AI 做项目最后变成死代码的原因。不是 AI 不会写代码，而是你没有告诉它什么叫好代码。

它只知道把功能做出来，却不知道这个项目后面还要维护、迭代、上线、接用户、排查问题。所以你必须提前告诉它：架构优先，设计优先，不要为了当前功能破坏项目边界，不要用表层方案掩盖底层问题，不要没有证据就改，不要没有验收就说完成。

这些规则看起来很普通，但长期下来差别非常大。没有宪法的 AI，就像一个能力很强但没有项目经验的实习生；有宪法的 AI，至少知道这个项目的工作边界和质量标准。

## **四、agent 宪法不是越长越好**

这里也要提醒一下：agent 宪法不是写得越多越好。

很多人一开始会犯一个错误，就是把所有东西都塞进规则文件里。项目介绍、技术细节、部署流程、数据库结构、历史问题，全部往里面堆，最后写成几千行。看起来很完整，实际上 AI 反而更难抓重点。

因为这类规则文件通常会进入 AI 的上下文。你写得越长，它消耗的上下文越多，重点也越容易被稀释。

真正好的 agent 宪法，应该短、硬、清楚。它应该写长期有效的原则，而不是流水账；应该写禁止事项和验收标准，而不是一堆空话；应该写可执行、可检查、可判断的规则，而不是模糊的口号。

AI 不是人，它不适合听“认真一点”“写好一点”“注意质量”这种话。你要告诉它，什么叫认真，什么叫质量，什么情况不能交付。

## 最后

如果说市面上的 AI IDE 是 Vibe Coding 的工具入口，那么 agent 宪法就是 Vibe Coding 的第一层工程底座。它解决的是：AI 在这个项目里应该遵守什么规矩。

没有 agent 宪法，你当然也可以让 AI 写代码。但那更像是在靠模型自由发挥。模型强的时候，结果可能还不错；模型状态差一点，或者项目复杂一点，问题就会开始堆积。

你会发现 AI 老是不按要求来，老是忘记规则，老是修一个问题又制造新的问题。很多时候，不是 AI 完全不行，而是你没有给它一套稳定的项目规则。

所以普通人想真正用 AI 做项目，第一步不是学某个按钮怎么点，而是先学会给 AI立规矩。
