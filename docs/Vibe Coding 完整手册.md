# Vibe Coding 完整手册

> 本手册合并自 sources/ 下的全部原始笔记，已去重、补漏、统一说法，并保留原作者惯用的通俗例子。
> 适用对象：没有专业代码基础、希望靠 AI 把项目真正做出来并长期维护的人。

---

## 序：Vibe Coding 是什么

Vibe Coding 不是让每个人都变成程序员，而是让不懂技术的人也能用技术解决自己的实际问题。它本质上只是团队角色发生了改变：以前你找程序员说需求、再让程序员开工；现在你面对的是 AI，本质还是要**把需求和规则说清楚**。

一句话总纲：**沟通清楚、规则写清楚、证据留下来，项目后面才不会靠猜继续写。**

AI 很强，但 AI 也很懒，它更像一种指令型、规则型生物——有些事情你不说，它就不会管。

## 第一部分：项目准备

正式写代码之前，至少要做四件事：立项、选技术栈、写 Agent 宪法、了解 Skill。

### 第 1 章 立项：别上来就让 AI 写代码

#### 1.1 先创建一个空项目文件夹

正式开始前，先用 AI IDE 打开一个空项目文件夹。它暂时可以没有代码，意义是给项目一个独立空间，后面讨论出来的立项文档、开发计划、Agent 宪法、项目说明都沉淀在这里。

文件夹名建议英文 + 中划线/下划线，例如 `task-manager`、`resume-assistant`。不要用中文、空格、特殊符号，也不要叫"新建文件夹""测试项目""demo1"。

#### 1.2 为什么要先讨论

AI 可以写代码，但它不能替你决定你真正想做什么。立项阶段和 AI 讨论，主要做三件事：

1. **让 AI 对你的产品有基本认知**：不要只说"我要做一个后台系统"，要让它逐步理解项目类型、目标用户、核心问题、主要流程。
2. **完善你的想法**：你一开始的想法通常很粗糙，AI 一问用户是谁、核心功能是什么、数据怎么流转、商业模式是什么，你就会发现很多地方还没想明白。
3. **让 AI 帮你查漏补缺**：你可能只想到功能没想到权限，只想到页面没想到数据，只想到第一版没想到后期扩展。

立项阶段暴露问题，比代码写到一半再发现方向错了要好得多。

#### 1.3 怎么和 AI 讨论

一开始不用说得太复杂，可以直接告诉 AI：

> 我准备做一个什么类型的产品，大概功能是什么，你觉得这个产品的核心设计应该是什么？你觉得这个想法怎么样？

让 AI 追问你、指出问题、拆分功能、判断第一版应该做什么、不应该做什么。可以让它分析用户场景、整理核心功能、判断项目价值、拆分 MVP、梳理数据流、推演用户使用流程，再让它帮你找竞品、开源方案、差异点和机会，最后做长期路线规划。

#### 1.4 讨论时不要让 AI 直接开写

很多 AI 一看到你说"我要做一个项目"，马上就会开始创建文件、安装依赖、写页面、写接口。这时你要拦住它，明确告诉它：

> 当前阶段只做产品讨论和立项设计，暂时不要写代码。

你要先让 AI 进入"产品合伙人"和"架构顾问"的角色，而不是一上来就进入"代码生成器"的角色。

#### 1.5 讨论完之后要生成立项文档

讨论得差不多之后，让 AI 在项目文件夹下生成立项文档。它的作用是把已经讨论清楚的内容固定下来，让项目有一个可以回看的依据。没有它，项目很容易在多轮对话、多次修改、多次上下文切换中漂移——今天说给个人用户用，明天 AI 可能写成企业后台；今天说第一版只做核心流程，明天 AI 可能塞进一堆复杂功能。

立项文档至少包含：

- **项目立项说明**：是什么、面向谁、解决什么问题、为什么值得做、第一版核心目标。
- **项目核心功能和边界**：第一版做什么、不做什么；哪些功能必须有，哪些后面再做，哪些暂时放弃。
- **产品流程和用户路径**：从进入到完成核心操作的步骤，页面怎么走，功能怎么串，数据怎么流动。
- **数据和业务对象**：核心对象有哪些（用户、订单、任务、文章、商品、文件、评论、权限等），会影响后续数据库和接口设计。
- **技术路线初步判断**：网页 / 小程序 / App / Web MVP；前端、后端、数据库大概怎么考虑。
- **项目爆点和差异点**：相比同类产品最值得做、最有差异、最可能打动用户的地方。
- **后期规划和里程碑**：第一版、第二版、长期版本往哪里走。
- **验收规则**：第一版做到什么程度才算完成——不是 AI 说完成就完成，要有可检查的标准。

项目小就一份文档够了；项目稍复杂可以拆成"项目立项说明 / 项目详细规划 / 长期路线"。文档不是形式主义，是防止项目跑偏。

#### 1.6 立项不追求完美

小白刚开始，最重要的不是写一份完美文档，而是把当前能确定的东西先写下来。后面随着继续讨论、开发、验证，文档可以不断更新。立项不是一次性决定未来所有事情，而是先给项目一个清晰起点。

### 第 2 章 选择项目技术栈

#### 2.1 先确定产品形态

问自己一个问题：你的产品最终以什么形式呈现给用户？网站？小程序？App？后台管理系统？还是只给其他系统调用的纯后端接口？

如果这个问题答不上来，就先别让 AI 写代码。如果不确定，可以直接让 AI 根据立项文档分析，结合产品目标、用户场景、核心功能和后期规划，判断项目最适合做成哪一种形态，并说明理由。

注意：不是让 AI 随便替你拍板，而是让它基于已经讨论过的项目事实给出明确的主线判断。对没有代码基础的人来说，**最怕的不是没有选择，而是选择太多**。

#### 2.2 技术栈不是越高级越好

新手很容易被各种技术名词带偏：今天听说一个框架很火，明天看到一个工具很新，就觉得自己的项目也应该用。但做项目不是比谁用的技术更新。

技术栈最重要的不是炫，而是适合——适合产品形态、适合项目复杂度、适合 AI 理解和维护、适合后期迭代。选择技术栈时一定要克制，目标是让项目结构清楚、边界清楚、后期能维护，而不是堆一堆听起来很厉害的名词。

#### 2.3 优先选择成熟框架和 SDK

互联网技术发展到现在，很多功能早就有成熟方案。**做 Vibe Coding 时非常关键的一点：不要动不动就让 AI 从零开始造**。你不会代码，更不应该让 AI 在没有框架约束的情况下自由发挥。

成熟框架的价值，不只是提供现成能力，还在于它通常有自己的项目结构、开发规范、最佳实践和社区经验。它不仅告诉你"能怎么写"，也告诉你"应该怎么写"。

新手最欠缺的就是规范。AI 已经是一个很强的编码助手，但如果没有规范，它就容易乱写、绕路、打补丁。如果你选了成熟框架，并明确要求 AI 必须遵守这个框架的官方规范、目录结构和推荐写法，项目质量通常会稳定很多。

可以直接告诉 AI：

> 这个项目必须严格遵守当前框架的官方规范，不允许为了图省事绕开框架设计。

仅仅这一句话，就能在一定程度上减少 AI 漂移。

#### 2.4 怎么判断一个技术靠不靠谱

你不需要自己硬查所有技术细节，但至少要知道怎么让 AI 帮你判断一个框架或 SDK 靠不靠谱。让 AI 查几个关键点：

- 是不是有足够多的人在用；
- 项目是否还在维护；
- 文档是否完整；
- 社区里有没有大量讨论；
- 最近更新时间是否太久远；
- 授权协议是否允许商业使用。

开源项目可以让 AI 帮你查 GitHub 的 star 数、issue 活跃度、最近提交时间、发布记录和使用案例。star 数不是唯一标准，但能说明关注度。授权协议也不能忽略——准备上线运营、收费、接用户，就必须让 AI 确认许可证是否允许商用。

如果对某个 SDK 有疑问，继续追问：这个项目是不是很久没更新了？它有没有明显风险？适不适合商业项目？以后用户量变大，会不会有明显瓶颈？

#### 2.5 让 AI 给你"唯一推荐方案"

应该让 AI 推荐技术栈，但不要只问"这个项目用什么技术栈？"

应该把已经整理好的立项文档、产品形态、功能范围、目标用户、部署要求和后期规划交给 AI，然后让它基于这些内容给出**当前项目的唯一推荐技术栈**。

重点是"唯一推荐"。新手最怕的不是 AI 不给方案，而是 AI 给太多方案——三四套技术栈每一套看起来都有道理，最后反而不知道选哪个。如果存在备选方案，也不要让 AI 把它们变成选择题，备选只用来说明取舍理由和潜在风险。

> 用 AI 做项目，最怕技术路线不稳。今天用这个，明天换那个，后天又觉得另一个更高级，项目很容易变成一堆拼起来的代码。

#### 2.6 技术栈要写进项目文档

技术栈讨论完之后，一定要写进项目文档，不要口头聊完就让 AI 开始写代码。文档里至少说明：产品形态、前端用什么、后端用什么、数据库用什么、主要框架和 SDK 是什么、为什么这么选、哪些技术不用、什么情况下才需要重新评估技术路线。

每次 AI 想引入新框架、新库、新架构时，都应该先回到这份文档看一眼：是不是符合当前技术路线？有没有破坏项目边界？有没有让项目变复杂？

### 第 3 章 Agent 宪法：给 AI 立规矩

#### 3.1 什么是 Agent 宪法

Agent 宪法本质上是写给 AI 的长期行为准则。它不是某一次对话里的临时要求，而是这个项目长期有效的工作原则。每次 AI 进入项目之前，都应该先知道这些规矩：

- 项目边界是什么
- 代码规范是什么
- 哪些文件不能乱动
- 哪些操作必须谨慎
- 完成任务之前需要怎么验证

很多新手用 AI 写代码，习惯直接上来一句"帮我加个功能""帮我修个 bug""帮我优化一下页面"。AI 当然会做，但如果你没有提前告诉它项目规则，它就只能根据当前上下文和模型自己的习惯来猜。猜对了还好，猜错了就可能开始乱改文件、乱抽象、乱兼容、乱补丁。

**一句话：建立 AI 核心规范、约束行为与边界、写给 AI 的项目规矩。**

#### 3.2 宪法文件通常放在哪里

不同 AI IDE 或 Agent 工具，有不同的规则文件格式：

- Codex：`AGENTS.md`
- Cursor：`.cursor/rules/` 目录下的规则文件（旧的 `.cursorrules` 已不推荐）
- Claude Code：`CLAUDE.md`，或 `.claude/CLAUDE.md`

不用死记这些名字，也不用一开始研究所有格式。如果不确定当前 IDE 应该用什么文件、放哪个目录，最简单的方式就是直接问它：

> 帮我创建一份当前工具会读取的 Agent 宪法，并放到正确位置。

#### 3.3 宪法为什么重要

Agent 宪法最重要的作用不是让 AI 更听话，而是让 AI 更稳定。

AI 最大的问题之一，就是很容易根据当前对话做**局部最优**，通俗点说就是偷懒。你让它修一个按钮，它可能只盯着按钮；你让它修一个接口，它可能只盯着接口。但真实项目不是这样——一个前端改动可能涉及组件规范，一个接口改动可能涉及权限模型，一个功能改动可能涉及数据库设计、错误处理、测试覆盖和后期维护。

如果没有 Agent 宪法，AI 很容易只解决眼前问题却破坏长期结构。这也是很多人用 AI 做项目最后变成死代码的原因。**不是 AI 不会写代码，而是你没有告诉它什么叫好代码。**

> 没有宪法的 AI，就像一个能力很强但没有项目经验的实习生；有宪法的 AI，至少知道这个项目的工作边界和质量标准。

#### 3.4 宪法不是越多越好

很多人一开始会把所有东西都塞进规则文件——项目介绍、技术细节、部署流程、数据库结构、历史问题，全部堆成几千行。看起来很完整，AI 反而更难抓重点，因为这些规则文件通常会进入 AI 的上下文，写得越长消耗的上下文越多，重点越容易被稀释。

真正好的 Agent 宪法应该**短、硬、清楚**：

- **写长期有效的原则**：要提炼出那些不随场景、时间、技术迭代而改变的基本价值观和底层原则。
- **写禁止事项和验收标准**：1）明确列出 Agent 绝对不允许做的事，一旦触犯就是严重违规，这些是刚性约束，通常用于安全、合规、伦理底线；2）从"输出合格"的角度定义，当任务完成时必须满足什么条件才叫"通过了验收"，这是结果导向的质量门槛。
- **写可执行、可检查、可判断的规则**：一条规则如果无法执行、无法检查、无法明确判断是否达标，就是一句空话。

AI 不是人，它不适合听"认真一点""写好一点""注意质量"这种话。你要告诉它什么叫认真、什么叫质量、什么情况不能交付。

#### 3.5 一份宪法模板示例

```markdown
# 项目 Agent 宪法 (Vibe Coding 总纲)

你是我项目中的 AI 编码助手。在生成任何代码、修改任何逻辑时，你都必须首先通过这部宪法的审查。宪法高于一切编码规范。

## Ⅰ. 永恒的安全与诚信
- 禁止代码中包含任何真实凭证。所有密钥、令牌均从环境变量或安全仓库读取，变量名必须包含 `SECRET_` 前缀。
- 禁止生成任何形式的歧视、仇恨、成人内容。用户输入必须经过 `safety_check()` 清洗，返回值为布尔 `true` 时才继续处理。
- 验收标准：安全测试函数 `test_safety()` 必须能检测出 10 种以上的越狱攻击模式。

## Ⅱ. 诚实与边界
- 当知识库无匹配或模型不确定时，必须输出"坦诚声明"而非猜测。
- 验收标准：`respond_with_uncertainty()` 不能接受任何会填充虚假详情的参数，如 `fake_specifics`，必须删除。

## Ⅲ. 执行的铁律
- 澄清优先：任何缺少必填字段的请求，必须触发 `ask_clarification()` 而不是用默认值，并列出缺少的字段名。
- 确认即阻断：任何具有副作用（网络发送、文件删除、数据库写入）的函数，其内部第一行必须是 `if not is_user_confirmed(): return PENDING`，等待确认信号。
- 熔断即终止：所有网络调用或工具调用必须包裹在 `with_retry(max_attempts=2)` 装饰器里，超过尝试次数必须调用 `request_human_help(error_log)` 并记录日志。

## Ⅳ. 作为代码生成器的承诺
- 每次输出代码后，必须附带一个"宪法遵守检查清单"，简要说明这段代码如何满足上述各条。
- 优先在代码中插入明确的断言和类型检查，让宪法规则"可自行校验"。
```

### 第 4 章 Skill：让 AI 用上专家经验

#### 4.1 Skill 解决什么问题

Skill 解决的不是"再写一份更长的规范"，而是把一套专业工作流程，变成 AI 可以按需调用的技能。它通常是语义命中后才加载，所以更适合承载那些具体、复杂、但不是每次都需要的流程。

主要解决三个问题：

1. 让 Agent 宪法保持简洁，不要什么都往里面塞。
2. 让 AI 在 Vibe Coding 过程中减少漂移，不要每次都自由发挥。
3. 让小白也能借用高手的工程经验。

#### 4.2 什么是 Skill

很多小白刚开始 Vibe Coding 时，最常见的状态是所有事情都靠一句话硬推：想做功能就说"帮我加一个登录"；遇到报错就说"帮我修一下"；页面不好看就说"帮我优化一下 UI"。

这种方式前期很爽，但 AI 每次都在根据当前这句话自由发挥。你不知道它有没有先拆需求、看项目结构、复现 bug、打开浏览器验收、跑测试。于是它这次一个做法、下次另一个做法，遇到问题就猜，猜错了就继续改，最后代码越来越乱。

**Skill 就是把流程交给 AI**。可以把它理解成 AI 的专项能力包，或者敲代码遇到某一类问题时应采用的处理思路和执行流程。

> 如果说 Agent 宪法是项目员工手册，那么 Skill 就像岗位 SOP。宪法告诉 AI 这个项目的长期原则，Skill 告诉 AI 遇到某一类具体任务时应该按什么流程执行。

#### 4.3 Skill 和宪法有什么区别

| 维度 | Agent 宪法 | Skill |
|------|------------|-------|
| 核心 | 管长期原则 | 管专项流程 |
| 标准 | 短、硬、清楚 | 具体操作指南 |
| 性质 | 长期有效规则 | 任务级指令 |
| 何时加载 | 每次进入项目都加载 | 语义命中后按需加载 |

把所有"具体怎么做"都写进宪法，宪法就会越来越长，最后变成 AI 抓不住重点的大杂烩。而且宪法长期占用上下文，这些不一定每次都用得上的流程会在每一次任务里反复消耗 token。Skill 的价值，就是把这些复杂流程拆出去，让它们在需要的时候再被加载。

当然，Skill 也不是完全不消耗 token，所以好的 Skill 也应该克制，该写流程写流程，该引用资料引用资料。

#### 4.4 Skill 为什么重要

**第一，减少 AI 漂移。** 一开始你让它做 A，聊着聊着它开始做 B；一开始架构是这样，改着改着它按另一套思路写——这就是漂移。Skill 把某一类任务的标准做法固定下来，AI 需要处理这类任务时，IDE 或 Agent 会根据语义自动加载对应 Skill，让它回到一套稳定的工作流程里。

**第二，复制高手经验。** 很多新手真正缺的不只是代码能力，而是工程经验——不知道需求应该怎么拆、bug 应该怎么查、代码审查要看哪些风险、上线前要验证什么。但很多开源 Skill 本身就是有经验的工程师把自己的工作方法沉淀出来的结果。

> 你自己可能还不是编程大师，但你可以让 AI 按编程大师总结出来的流程工作。

#### 4.5 Skill 怎么使用

Skill 不是让你每次手动排顺序。很多时候，AI IDE 会根据你的表达自动判断该用哪个 Skill：

- 你说要排查问题，它可能匹配调试类 Skill；
- 你说要写开发计划，它可能匹配计划类 Skill；
- 你说要检查页面，它可能匹配浏览器或前端验收类 Skill。

也可以主动点名某个 Skill，让 AI 按那个 Skill 执行。

Skill 也不是普通提示词。一个 Skill 通常会有自己的说明文档：叫什么、什么时候应该使用、解决什么问题、执行时要遵守什么流程、最后应该输出什么结果。有些还会带参考资料、模板、脚本，甚至专门的命令。

使用 Skill 时不要只靠猜，可以直接问 IDE：当前有哪些 Skill？某个 Skill 是干什么的？它有没有命令或脚本？如果要做这个任务，应该触发哪个 Skill？

#### 4.6 普通人怎么安装 Skill

很多时候你不需要自己研究目录结构，也不需要手动复制文件，直接把 Skill 的链接丢给 AI IDE，告诉它：

> 帮我把这个 Skill 安装好，并确认它能被当前工具识别。

如果不知道适不适合当前项目，也可以直接问 IDE：这个 Skill 适合我现在这个项目吗？它会解决什么问题？会不会和现有 Agent 宪法冲突？安装之后应该怎么触发？有没有命令或脚本需要注意？

因为 Skill 本身就是给 AI 用的，你完全可以让 AI 帮你读它的设计文档，解释它的使用方式，再判断它适不适合当前项目。如果项目比较特殊（前端框架不同、后端语言不同、部署方式不同），也可以让 IDE 帮你做适配。

#### 4.7 一组常用的工程类 Skill

| Skill | 解决什么问题 |
|-------|--------------|
| `brainstorming` | 偏需求和设计，避免 AI 一上来就写代码 |
| `writing-plans` | 偏开发计划，把需求拆成可执行任务 |
| `systematic-debugging` | 偏系统化排查，核心是不要猜、先找根因 |
| `verification-before-completion` | 偏完成前验收，防止 AI 写完代码就直接说完成 |
| `requesting-code-review` | 偏代码审查，用另一个视角检查风险 |
| `dispatching-parallel-agents` | 偏多 Agent 并行，适合多个独立任务同时推进 |
| `browser` | 偏浏览器验证，适合前端页面、交互、截图和本地服务检查 |
| `sliver-engineering-workflow` | 一套完整工程流，强调架构优先、真源优先、严格验收、不漂移、不乱补丁 |

不需要死记调用顺序，只需要知道它们大概解决什么问题。真正执行时 IDE 往往会自动选择，想强制使用也可以直接点名。

## 第二部分：前端

### 第 5 章 前端技术认知

#### 5.1 什么是前端

很多人一听到前端，第一反应就是网页。这不算错，但不完整。更准确地说，**所有用户能看到、能点击、能操作的部分，都可以理解成前端**。

网站里的页面、按钮、表单、弹窗是前端；App 里的首页、列表、个人中心、支付页面也是前端；小程序里的页面、组件和交互，同样属于前端。甚至一些硬件设备上的屏幕界面——智能手表、车机屏幕、收银机界面、工业设备触控屏——只要它是给用户看的图形界面，也可以理解成前端。

所以，前端不是网页的代名词。前端本质上是**用户和系统之间的交互层**：它负责把系统能力展示给用户，也负责把用户的操作传递给系统。

#### 5.2 为什么 Vibe Coding 要先理解前端

对大多数普通项目来说，前端是你最容易感知到的部分，也是 Vibe Coding 里最适合先落地的部分。原因很简单：前端会决定这个项目长什么样，也会倒逼你想清楚这个项目到底有什么数据、有哪些页面、有哪些操作流程。

一个后台系统，你先把页面做出来，就会马上看到需要哪些列表、表单、详情页、按钮、筛选条件和状态字段。这些东西一旦画出来，项目的数据形式其实就慢慢清楚了——用户是什么结构，订单是什么结构，任务是什么结构，权限是什么结构，很多时候不是你空想出来的，而是在做页面和流程的时候被逼出来的。

这对 AI 也非常重要，因为 AI 很怕需求模糊。你只说"我要做一个管理系统"，AI 很难真正理解你要什么。但你先让它做出页面、流程和交互，它就能更直观地理解这个项目。

> 换句话说，前端不仅是展示层。在 Vibe Coding 里，前端还经常是**项目理解层**。它能帮助你和 AI 一起把一个模糊想法，变成看得见、能讨论、能修改的东西。

但如果你只会说"帮我做得好看一点""帮我优化一下页面""帮我做得高级一点"，AI 就只能自由发挥——堆一堆渐变、卡片、阴影和动画，看起来热闹，却不一定真的好用。所以理解前端最重要的不是马上学会写 HTML、CSS、JavaScript，而是学会怎么**描述前端**。

#### 5.3 前端最重要的是表达清楚

在 Vibe Coding 里，你不一定要会写前端代码，但最好知道一些基本词汇：

- **UI（用户界面）**：用户能看到的按钮、表单、列表、弹窗、导航和页面结构。
- **UX（用户体验）**：用起来顺不顺、流程清不清楚、操作会不会迷路、反馈是不是及时。
- **风格**：整个界面给人的视觉感受，偏简洁、商务、科技感、可爱、工具型、内容型等。
- **布局**：页面内容的组织方式，内容放在哪里、主次怎么安排、左侧要不要菜单、右侧要不要详情、列表和表单怎么组合。
- **组件**：页面里可以复用的小模块，如按钮、输入框、卡片、表格、弹窗、导航栏、标签页。
- **交互**：用户操作之后发生什么，如按钮点击后的反馈、提交失败后的提示、加载中的状态、删除之前的确认。
- **响应式设计**：页面在不同屏幕尺寸下都能正常显示。

> 在 Vibe Coding 里，前端术语就是你和 AI 沟通页面效果的语言。你越能说清楚，AI 越不容易瞎猜。

#### 5.4 常见的前端场景

前端按运行场景大概可以分成几类：

1. **网页前端**：浏览器打开的网站、后台系统、官网、Web App——管理后台、SaaS 平台、数据看板、个人博客、工具网站。
2. **App 前端**：手机 App 里的界面，可以是原生 App，也可以是跨平台 App。
3. **小程序前端**：运行在平台提供的小程序环境里，如微信小程序、支付宝小程序、抖音小程序。不完全等同于网页，也不完全等同于 App。
4. **硬件 GUI**：图形用户界面，常见于智能硬件、工业设备、车机、仪器仪表。

跟 AI 说"做一个前端"时，最好告诉它具体是哪一种场景。不同场景对应不同技术栈、设计规范和交互规范。如果不确定当前场景应该遵守什么规范，可以直接问 AI：这个前端场景常见的设计规范是什么？这个平台有哪些交互习惯？这个页面应该按网页、App、小程序，还是硬件 GUI 的方式设计？

#### 5.5 不同前端对应不同技术栈

前端不是一种技术，而是一类开发场景，按技术栈可以粗略分成两层：

**底层能力：**
- HTML 负责页面结构
- CSS 负责样式和布局
- JavaScript 负责交互和逻辑

**现代工程化能力：**
- React、Vue、Svelte 等前端框架
- TypeScript 提升代码可靠性
- Vite、Next.js、Nuxt 等工具和框架管理项目结构、构建、路由、渲染和工程化

注意：现代网页前端不是脱离了 HTML、CSS、JavaScript。恰恰相反，大多数现代 Web 技术栈最终还是会编译、打包成浏览器能运行的 HTML、CSS 和 JavaScript。所以前者是底层能力，后者是现代工程化能力。

不同前端场景对应不同体系：

- **App 前端**：原生 iOS 常见语言是 Swift，原生 Android 是 Kotlin 或 Java；跨平台常见有 Flutter、React Native。
- **小程序前端**：微信小程序常见 WXML、WXSS 和 JavaScript，也有 Taro、uni-app 做跨端开发。
- **硬件 GUI**：有些用 C、C++、Qt，嵌入式设备会用 LVGL 这类轻量级 GUI 框架。

你不需要一开始全学会，但必须知道：现代前端框架不是魔法；不同前端场景对应不同运行环境和技术栈。否则你让 AI 做 App，它可能给你写网页；你让 AI 做小程序，它可能按普通 Web 项目设计；你让 AI 做硬件界面，它可能还在给你写 React 页面。**方向错了，后面越写越麻烦。**

#### 5.6 前端在项目里负责什么

前端负责用户看得见、摸得着的部分，通常不负责真正的数据存储，也不负责核心业务规则。

- 登录功能：前端负责展示登录页面、收集账号密码、调用接口、显示登录结果；账号密码是否正确、用户权限是什么、登录状态怎么保存，通常属于后端。
- 订单系统：前端负责展示订单列表、筛选条件、详情页面和操作按钮；订单数据从哪里来、订单状态怎么计算、支付结果怎么处理，通常属于后端。

所以前端和后端不是谁更重要的问题，而是分工不同。前端更靠近用户，后端更靠近数据和业务规则。分清楚层级，项目才不会越修越乱。

### 第 6 章 搭建前端项目架构

每次拿到一个前端项目，第一步都不是让 AI 先写页面，而是先把前端骨架定下来。它包括：整体设计风格、前端技术方案、UI 组件库、目录结构、模块边界、组件复用规则、样式系统，以及后面的实施计划。这些东西先有标准，AI 后面才知道该按什么方式写。

#### 6.1 先确定整体设计风格

设计风格不是某一个页面好不好看，而是整个 UI 呈现出来的统一视觉语言。常见风格包括：

- 苹果系统里的玻璃拟态、液态玻璃、半透明模糊背景；
- Google 的 Material Design，层级清楚、组件规范完整；
- 后台管理系统常见的企业级管理台风格，信息密度高、表格多、操作效率优先；
- SaaS 产品常见的现代极简风格，卡片清楚、按钮克制、留白适中；
- AI 产品常见的科技感风格，深色背景、渐变、发光元素、动效更多；
- 内容社区常见的信息流风格，重点是阅读舒服、内容层级清楚。

这些风格最后都会落到具体细节上：颜色、字体、字号层级、行高、间距、圆角、卡片样式、按钮形态、页面布局和信息密度。

如果不知道自己想要什么风格，可以找几个喜欢的网站把链接发给 AI；如果是 App，就找几张参考截图发给 AI。让 AI 分析这些界面的视觉风格、布局方式、组件特点，以及这种风格适不适合你的项目。

> 注意，这不是让你抄 UI。成熟产品的视觉节奏、留白方式、组件组织方式可以学习。参考是为了确定方向，不是直接复制。

#### 6.2 再确定前端技术方案和 UI 组件库

这两个东西不要混在一起：

- **前端技术方案**解决的是项目怎么组织、怎么运行——页面怎么放、路由怎么写、目录怎么分、项目怎么启动、构建和部署怎么处理。常见方案有 React、Vue、Angular、Next.js、Nuxt；小程序则有微信原生框架、Taro、uni-app。
- **UI 组件库**解决的是界面元素怎么统一——按钮、输入框、弹窗、表格、菜单、卡片、提示信息怎么写。常见组件库有 Ant Design、Element Plus、Naive UI、MUI、Mantine、Vant、NutUI；特殊方案还有 Radix UI（无样式组件基础库）、shadcn/ui（组件源码方案）、Tailwind UI（模板型组件方案）。

> 一句话：前端框架管项目结构，UI 组件库管界面统一。框架不一定自带 UI 组件库，UI 组件库也不一定是框架。

如果你不懂技术、主要靠 AI 写前端，强烈建议优先选成熟方案——成熟框架通常有目录规范、开发规范和最佳实践，成熟组件库也已经把按钮、表单、弹窗、提示、表格统一好了。你不是不能从零写，而是没必要让 AI 在没有边界的情况下从零发明一套前端结构。

可以直接告诉 AI：

> 这个项目必须遵守当前前端框架的官方规范，优先复用选定的 UI 组件库，不允许到处手写重复组件，也不允许绕开框架最佳实践。

这句话就是在给 AI 画边界。

#### 6.3 把目录、模块和组件规则定下来

前端骨架真正要解决的不是先做多少页面，而是后面的页面应该怎么写。

1. **目录边界要清楚**：页面放哪里、组件放哪里、请求接口放哪里、工具函数放哪里、样式文件放哪里，都要有规则。
2. **模块边界要清楚**：登录、用户、订单、内容、设置这些功能区域不要全部混在一起。模块清楚，AI 后面改某个功能时才不会到处乱动。
3. **组件复用规则要清楚**：按钮、输入框、卡片、表格、弹窗、提示框、导航栏会反复出现。能用 UI 组件库就优先用，组件库不够再基于组件库封装项目自己的业务组件。判断标准：只要 UI 相似度比较高，并且在项目里出现超过 2 次，就应该考虑封装成组件。

可以把这条写进 Agent 宪法：

> 同类 UI 结构出现超过 2 次时，必须优先抽象为可复用组件，不允许复制粘贴重复实现。

#### 6.4 提前准备样式系统

1. **设计 token**：这里的 token 不是 AI 的 token，而是设计变量——颜色、字号、行高、间距、圆角、阴影、边框。必须在写页面前定好，否则 AI 今天写 `16px`，明天写 `18px`，后天又写 `15px`，最后项目里到处都是重复 CSS。
   > 所有颜色、字号、间距、圆角、阴影，都必须从统一设计 token 里取，不允许到处硬编码样式。
2. **多语言规范**：如果项目未来可能支持中英文，前端文案就不要到处硬编码。按钮文字、页面标题、提示信息、错误信息都应该统一管理。中文两个字能放下，英文可能变成一整句，一开始不考虑，后面切语言时很容易出现按钮撑爆、标题换行、布局错位。
3. **主题规则**：主题不是为了炫技，而是为了让颜色体系可控。先确定一个默认主色调，再预留主题扩展方式。主题色不能散落在每个页面里手写，否则后面想换主色调就要全项目到处改。

#### 6.5 写进实施计划，再让 AI 搭骨架

前面这些确定之后，一定要落实到详细的实施计划里。不要口头聊完就直接让 AI 写代码。实施计划至少写清楚：

- 整体设计风格是什么
- 前端技术方案是什么
- UI 组件库用什么
- 目录结构怎么组织
- 模块边界怎么划分
- 哪些组件先封装
- 设计 token 怎么定义
- 多语言和主题规则怎么预留
- 第一阶段只做哪些内容
- 验收标准是什么

注意，这一步仅仅是做骨架，不要让 AI 一次性做完所有页面和功能。这一轮目标只有一个：**项目能跑起来，基础结构是对的**。比如项目能启动、首页能打开、路由结构清楚、目录规范正确、UI 组件库接入完成、设计 token 准备好、主题规则接入完成、常用组件有基础版本。

骨架跑通之后，先提交一次 Git，然后再一个模块一个模块往里加功能。

> 做前端，最怕的不是页面丑，最怕的是没有统一规则。

### 第 7 章 小程序踩坑案例：单位差异

这是真实遇到过的坑：HTML 画布用 px（CSS 像素），小程序 `app.wxss` 也用 px（逻辑像素），但两者 1:1 不等价。

| 属性 | HTML 画布 | 小程序 app.wxss | 实际渲染差异 |
|------|-----------|------------------|--------------|
| 按钮高度 | 48px | 8px + 14px 字体 + 8px ≈ 34px | 小程序按钮矮 30% |
| 按钮圆角 | 9999px（胶囊型） | 6px（几乎直角） | 胶囊变方块 |
| 按钮内边距 | 0 32px | 8px 16px | 小程序按钮窄 |
| 卡片圆角 | 24px（radius-xl） | 10px（radius-md） | 小程序卡片方 |
| 卡片内边距 | 32px | 14px | 小程序卡片挤 |
| 输入框高度 | 48px | 11px + 14px + 11px ≈ 36px | 小程序输入框矮 |
| 阴影 | 0 4px 8px | 0 2px 12px | 差异不大 |

**核心结论**：HTML 画布中 48px 在手机上约等于 96rpx，而小程序里按钮实际只有 34px（约 68rpx）。整个小程序的组件尺寸都偏小，圆角体系从"胶囊型"变成了"近似直角"。

**修复方案**：把 `app.wxss` 中所有组件的 px 值改为 rpx，对齐 HTML 画布。

启示：跨端设计稿不能简单照搬，必须让 AI 提前确认目标端的单位换算规则。

## 第三部分：数据库

### 第 8 章 数据库技术认知

#### 8.1 什么是数据库

数据库就是专门用来存数据的地方。前端负责展示数据，后端负责处理数据，而数据库负责长期保存数据。

一个用户注册之后，账号、手机号、密码哈希、创建时间要保存到数据库；一个订单创建之后，订单号、用户、商品、金额、状态、支付时间也要保存到数据库；一篇文章发布之后，标题、内容、作者、发布时间、审核状态同样要保存到数据库。

如果没有数据库，这些数据就只能临时存在内存里，程序一重启就没了；或者被写死在代码里，根本无法真正运营。

> 数据库是项目的数据底座。

#### 8.2 为什么 Vibe Coding 必须理解数据库

很多小白用 AI 做项目时最容易忽略数据库。前端页面能看、后端接口能调，就以为项目完成了。但实际上很多时候数据只是假的，或者只是临时存在本地，根本没有真正进入数据库。

这类项目看起来能跑但不能真正用：刷新页面数据没了；换账号数据乱了；部署上线发现本地数据带不上去；想加新功能发现原来的数据结构支撑不了；想统计用户行为发现之前根本没有记录。

在 Vibe Coding 里这个问题尤其常见，因为 AI 很容易为了让功能先跑起来，临时造一个简单的数据结构。前期看起来没问题，后期功能一复杂，就会发现字段混乱、关系不清、重复数据一堆、权限也不好控制。

> 你不一定要会亲手写复杂 SQL，但你必须知道：数据怎么存，决定了项目后面怎么查、怎么改、怎么扩展、怎么维护。

#### 8.3 建议顺序：先 UI，再数据库，再后端

完全靠 Vibe Coding 做项目，更建议的顺序是：**先写 UI，再设计数据库，最后写后端**。

这不是传统开发里唯一正确的顺序，而是更适合小白和 AI 协作的方式。原因很简单：UI 最容易把一个模糊想法变具体。

- 页面里有什么列表，列表就暗示了你需要哪些数据字段；
- 页面里有什么详情页，详情页就暗示了数据对象的完整结构；
- 页面里有什么表单，表单就暗示了用户会提交什么数据；
- 页面里有什么按钮，按钮就暗示了需要哪些操作；
- 页面里有什么状态，状态就暗示了数据流转过程。

也就是说，UI 会倒逼你和 AI 把数据形式想清楚。等 UI 和流程基本确定之后，再让 AI 根据页面反推数据库结构就会靠谱很多——它不是凭空设计数据库，而是根据已经看得见的页面、字段、状态和操作来设计。

> 用 UI 明确产品形态，再用数据库固定数据结构，最后用后端把页面和数据连接起来。

#### 8.4 数据库服务、数据库、表和字段是什么关系

很多人会把它们都叫数据库，但严格一点说不是一回事。

- **数据库服务**：正在运行的数据库软件。比如在服务器上安装并启动了 MySQL 或 PostgreSQL，这个运行起来的东西就是数据库服务。
- **数据库**：某个项目的数据空间。一般来说一个项目对应一个或多个数据库；对小白来说先理解成一个项目对应一个数据库就够了。
- **数据表**：存放某一类数据，比如用户表专门存用户，订单表专门存订单。
- **字段**：每条数据要记录哪些信息，比如用户表里有用户名、手机号、密码、头像、注册时间。
- **记录**：一张表里的一条完整数据，比如用户表里的一条记录就是一个用户。

> 把它想象成一个办公楼：数据库服务像整栋办公楼，数据库像办公楼里的一个公司，数据表像公司里的不同部门，字段像每个部门固定要登记的信息项，记录就是一条条真实登记进去的信息。

在 Vibe Coding 的时候，一定要让 AI 把这个关系说清楚：当前用的是什么数据库服务？创建了哪个数据库？这个数据库里有哪些表？每张表存什么？每张表有哪些字段？字段分别代表什么？表和表之间怎么关联？

#### 8.5 常见数据库类型和怎么让 AI 选型

按是否开源分：

- **开源**：MySQL、PostgreSQL、SQLite、MongoDB、Redis、ClickHouse 等。MySQL/PostgreSQL 是非常常见的关系型数据库；SQLite 更轻量，适合本地应用、小工具、原型项目或嵌入式场景；MongoDB 是文档型数据库，结构灵活；Redis 通常用于缓存、临时状态、排行榜、验证码、会话；ClickHouse 常见于数据分析和日志分析。
- **不开源**：Oracle Database、Microsoft SQL Server、IBM Db2 等，通常出现在企业级系统、传统行业、金融、电信、政企项目里，有成熟的商业支持、权限体系、审计能力，但使用成本、部署成本和学习成本更高。

按数据模型分：

- **关系型数据库**：MySQL、PostgreSQL、SQLite、Oracle、SQL Server，用表组织数据，适合用户、订单、商品、文章、权限这类结构清晰的数据。
- **非关系型数据库**：MongoDB 用文档存数据，结构灵活，适合变化快、结构不那么固定的场景。
- **缓存数据库**：Redis 通常不长期保存核心业务数据，用来加速访问、保存临时状态。
- **向量数据库**：AI 项目里更常见，存文本、图片、音频等内容的向量表示，方便做相似度搜索、知识库检索、RAG。

对普通 Vibe Coding 项目，大多数业务项目常见建议是优先考虑 MySQL 或 PostgreSQL。但这不代表你拍脑袋选，更合理的方式是让 AI 根据业务类型给出推荐，但必须要求它说明理由：

> 这个项目更适合 MySQL 还是 PostgreSQL？为什么？如果选 MySQL，后期可能有什么限制？如果选 PostgreSQL，会不会增加部署和维护成本？这个项目有没有必要用 MongoDB、Redis 或向量数据库？如果没有，为什么不需要？最后给出明确结论：当前项目推荐哪个数据库、原因是什么、不选其他数据库的理由是什么、后期如果项目变复杂是否需要引入新的数据库组件。

#### 8.6 数据库设计常见概念

- **主键**：一条数据的唯一标识，比如每个用户都有用户 ID。
- **唯一键**：保证某个字段不能重复的约束，比如手机号、邮箱、用户名、订单号。
- **自增**：让数据库自动生成连续编号，比如用户 ID 从 1 开始自动往后加。
- **外键**：两类数据之间的关联，比如一笔订单属于某个用户，订单表里就需要记录它对应的是哪个用户。
- **索引**：让数据库查询更快的结构。数据少时感觉不到，数据多了没有索引会变慢。但索引不是越多越好，太多会影响写入速度、让结构更复杂。
- **查询**：从数据库里找数据；**写入**：保存新数据；**更新**：修改已有数据；**删除**：移除数据，通常要非常谨慎，尤其是真实业务数据。
- **迁移**：数据库结构发生变化时的变更记录。新增字段、新增表、修改字段类型都应该通过迁移来管理，而不是随便手动改。

#### 8.7 数据库最容易翻车的地方

数据库最容易翻车的地方，通常不是一开始跑不起来，而是后面越做越乱：

1. **表结构设计太随意**：该分开的数据没分开，所有东西塞进一张表，前期省事，后面查询、统计、权限、扩展都会变麻烦。
2. **所有数据都塞进一个大字段**：后面很难筛选、统计、排序和维护。
3. **表之间关系没设计清楚**：用户、订单、商品、权限怎么关联，如果一开始没想清楚，后面接口和业务逻辑都会变乱。
4. **没有数据校验**：金额、手机号、邮箱、状态、数量都需要校验，否则脏数据会进入数据库。
5. **没有迁移记录**：项目迭代时数据库结构一定会变化，没有迁移记录后面开发环境、测试环境、线上环境很容易不一致。
6. **删除太随意**：很多真实业务里数据不能说删就删，删除之前要考虑是否有历史记录、是否影响统计、是否需要恢复。
7. **敏感信息处理不当**：密码不能明文存储，密钥不能放进前端，用户隐私数据不能随便暴露，数据库连接信息也不能泄露。
8. **唯一性和索引没设计好**：该唯一的字段没有唯一键，后面就可能出现重复手机号、重复邮箱、重复订单号；该加索引的字段没有索引，数据一多查询就会变慢。

### 第 9 章 数据库设计流程

#### 9.1 先从业务流程里找对象

数据库一定是从业务里长出来的。第一步不是问 AI 要几张表，而是让 AI 根据页面、用户流程和核心功能先整理项目里有哪些业务对象。

- 电商项目：用户、商品、订单、支付记录、收货地址。
- 内容平台：用户、文章、评论、点赞、收藏。
- 后台系统：用户、角色、权限、操作日志。
- 预约系统：用户、服务项目、预约记录、时间段、门店。

这些业务对象，后面大概率就是数据库里的核心表。

如果前端已经比较清楚，可以让 AI 根据页面反推数据；如果前端没做完整，也可以让 AI 从立项文档和功能说明里推，但要多和 AI 讨论，因为需求文档不一定写得细，AI 很容易漏掉业务对象。

> 到这一步，先别急着建表。先让 AI 说清楚：项目里有哪些核心对象，每个对象对应什么业务场景。

#### 9.2 确认对象之间的关系

数据库最容易乱的地方，往往不是少一两个字段，而是表和表之间的关系没想清楚。

- 用户和订单：一个用户可以有多个订单，一个订单只属于一个用户，订单表里通常需要 `user_id`。
- 文章和评论：一篇文章可以有多条评论，一条评论属于一篇文章也属于一个用户，评论表里通常需要 `article_id` 和 `user_id`。
- 用户和角色：如果一个用户只有一个角色，可以在用户表里放 `role_id`；如果一个用户可以有多个角色，就需要用户角色关联表。

让 AI 解释清楚：每张表代表什么对象、哪些表之间有关联、是一对一/一对多/多对多、关联字段放在哪张表里、为什么这样设计。

> 这一步的目的是确认 AI 理解的是你的业务，而不是自己猜了一套表结构。你可以信任 AI 的技术判断，但前提是你要把业务讲清楚，并让它把设计理由解释出来。

#### 9.3 确定数据库服务

对象和关系确认之后，再确定用什么数据库。常见业务项目优先考虑 MySQL 或 PostgreSQL。Redis 也经常出现，但通常不是主数据库，而是用来做缓存、登录状态、验证码、限流、队列、临时数据。

> 除非项目明确有缓存、队列或高并发需求，否则不要一开始就把 Redis 当成必选项。

让 AI 说清楚：主数据库用什么、为什么用它、要不要 Redis、本地开发和线上部署怎么处理。对不懂技术的人来说，优先选择成熟、常见、文档多、AI 熟悉的方案。

#### 9.4 设计表结构和字段规范

这里要用数据库设计里的**三大范式**做基本校对。

- **第一范式（字段要原子化）**：一个字段只存一个明确含义。手机号就放手机号，地址就放地址，备注就放备注，不要把手机号、地址、备注拼成一个大字符串。
- **第二范式（字段要围绕当前这张表的主对象）**：用户表主要放用户信息，订单表主要放订单信息。订单表可以通过 `user_id` 关联用户表，就不要把用户头像、昵称这类和订单本身无关的信息反复塞进订单表。但有些订单信息需要保留当时的快照（如收货人姓名、收货手机号、收货地址），因为用户后面改了地址历史订单不能跟着变——这种重复是有业务原因的，必须让 AI 说明清楚。
- **第三范式（字段之间不要互相依赖）**：用户表里已经有 `city_id`，城市名称可以从城市表里查，就不要随手再存一份 `city_name`，否则城市名称一改两边可能不一致。

> 三大范式的核心不是把表拆得越多越好，也不是字段越少越好，而是让数据更清楚、少重复、少冲突。复杂业务为了查询性能确实可能做反范式设计，但这种情况必须让 AI 先说明原因。

#### 9.5 常见字段要提前规范

以用户表为例，常见字段可能有：

```
id  username  phone  password_hash  last_login_ip  created_at  updated_at  last_login_at
```

数据库字段建议用英文下划线命名，后面写代码、写 SQL、接接口都会更方便。

- `id` 是主键，唯一标识一条数据。
- `created_at` 是创建时间，`updated_at` 是更新时间，一般由程序或数据库自动维护。
- 如果有些数据不能真的删除（订单、支付记录、财务数据），可以考虑加 `deleted_at` 做软删除——前台看不到了，但数据还在数据库里，方便后台审计和后续排查。

几个坑要提前避开：

- 密码不要存明文，要存 `password_hash`；
- 金额不要随便用浮点数，可以用分为单位的整数，或者数据库里的精确数字类型，并说清楚货币是什么、最多几位小数；
- 状态字段要定义清楚（如订单状态是待支付、已支付、已取消、已退款），不能让 AI 随便写几个字符串就结束；
- 字段类型可以让 AI 推荐，但关键业务字段一定要确认（手机号是否支持国际号码、金额是否支持多币种、状态后面会不会扩展）。

#### 9.6 先保留设计文件，再建表

数据库设计确认之后，不要直接让 AI 连数据库建表。先让 AI 生成数据库设计文件，比如 `schema.sql`，或者当前框架里的 migration 文件。这份文件一定要保留下来，并且提交到 Git——它是数据库结构的来源。

更稳的流程：先生成数据库设计文件 → 核对业务对象、表关系、字段和三大范式 → 确认没问题再执行建表 → 后面要改字段也先改 schema 或 migration 再执行变更。

这条也可以写进 Agent 宪法：

> 任何数据库结构变更，必须先修改 schema 或 migration 文件，说明变更原因，再执行数据库操作。

#### 9.7 写进实施计划，再让 AI 动手

实施计划里至少写清楚：

- 有哪些业务对象；
- 对象之间是什么关系；
- 使用什么主数据库；
- 要不要 Redis；
- 有哪些核心表；
- 每张表负责什么业务；
- 字段名、字段类型、是否必填；
- 哪些字段唯一、哪些字段可以为空；
- 表和表怎么关联；
- 哪些数据不能物理删除；
- 是否符合前三范式；
- 验收标准是什么。

这一步不要一口气写完整业务，目标只是先把数据库基础结构跑通：数据库能连接、核心表能创建、migration 能执行、能插入测试数据、能查询测试数据。跑通之后先提交一次 Git，再开始写接口和业务功能。

## 第四部分：后端

### 第 10 章 后端技术认知

#### 10.1 什么是后端

简单来说，后端就是用户看不见，但项目真正运行时离不开的部分。

你在页面上点击登录按钮，前端负责展示输入框和按钮，但账号密码是否正确要由后端判断。你在后台里查看订单列表，前端负责展示表格，但订单数据从哪里来要由后端查询。你点击支付、提交表单、上传文件、修改资料、删除数据，这些操作最终都需要后端处理。

后端通常负责几件事：接收前端请求、处理业务逻辑、读写数据库、判断用户权限、返回处理结果。

> 从用户视角看，后端是看不见的；但从项目视角看，后端往往决定了这个项目是不是真的能用。

#### 10.2 两种后端场景

新人接触 Vibe Coding 一般会遇到两种后端场景：

**第一种是小脚本**：批量整理文件、处理表格、生成报表、压缩图片、调用某个 API、定时跑一个任务。这种不一定要做成完整项目，一个脚本就够了。逻辑通常没那么复杂，AI 也比较容易写出来。

> Vibe Coding 很重要的意义在于技术平权——不是让每个人都变成程序员，而是让不懂技术的人也能用技术解决自己的实际问题。技术是工具，不是人生标准。

**第二种是项目级后端**：用户系统、订单系统、内容发布、权限管理、支付回调、后台审核、数据统计。这种后端不是一个小任务，它是在支撑一个产品的业务逻辑。

#### 10.3 项目级后端管什么

项目级后端主要管三件事：

1. **业务规则**：谁能登录、谁能下单、什么状态可以退款、什么内容需要审核、什么操作必须管理员才能做。这些规则不能只写在前端，因为前端是用户能接触到的地方，页面上的很多东西都可以被绕过或篡改。真正关键的判断必须放在后端。
2. **数据流转**：前端提交数据之后不是直接塞进数据库，后端要先校验、再处理、最后再决定怎么写入数据库。比如保存文章要判断标题是否为空、内容是否合法、用户有没有发布权限、是否需要审核；比如创建订单要判断库存、价格、优惠券、收货地址、支付状态。
3. **接口协作**：在大多数业务项目里，前端和后端通过 API 协作。API 是前端和后端之间约定好的沟通入口。前端说我要登录，后端返回是否成功或失败原因；前端说我要订单列表，后端返回当前用户能看到的订单数据。

> 所以在现代的多数业务场景中：后端经常表现为 API 接口。但 API 只是入口，背后的业务判断才是重点。

#### 10.4 为什么 Vibe Coding 也要理解后端

很多小白用 AI 做项目时最容易先做前端，这个方向没问题，但如果项目想真正跑起来就一定会进入后端。前端可以先把页面做出来，但页面里的数据不能一直是假数据；用户提交的内容要保存，登录状态要判断，权限要控制，数据要查询，操作要记录，异常要处理，这些都属于后端。

更重要的是，很多时候如果你只是让 AI 给你写一个网站，却没有开发配套后端，很容易留下安全问题。比如一些 API Key、Token、数据库连接信息、第三方服务密钥，如果直接写在前端代码里，用户是有机会看到的——因为前端代码最终要发到用户浏览器里运行，**只要代码到了用户设备上，它就不再是安全的秘密**。

> 涉及密钥、权限、支付、数据库写入、敏感数据处理的部分，通常都应该放到后端。前端负责发起请求和展示结果，后端负责真正执行敏感操作。

如果你完全不懂后端，就很容易被 AI 带偏：你以为功能做好了，实际上数据只是写死在页面里；你以为登录能用了，实际上没有真正的权限校验；你以为后台能管理数据，实际上只是前端假列表；你以为删除按钮能删除数据，实际上后端根本没有设计安全确认和操作记录。

> 前端能帮你把项目"看起来"做出来，后端则决定这个项目"实际上"能不能运转，以及能不能安全运转。

#### 10.5 前端和后端怎么配合

前端和后端之间通常通过接口通信。前端发起请求，后端处理请求，再把结果返回前端。可以理解成一个对话过程：前端问后端要数据，后端查完之后返回数据；前端告诉后端用户提交了表单，后端检查、保存，再告诉前端处理结果。

**接口（API）**就是前端和后端之间约定好的通信方式。一个接口通常要说明几件事：前端要请求什么地址、用什么方法、要传哪些参数、后端会返回什么数据、失败时会返回什么错误。

在 Vibe Coding 里，接口非常重要。如果接口没设计清楚，前端和后端就会各写各的——前端以为后端会返回一个字段，后端实际返回另一个字段，最后页面就会出错。

所以你让 AI 写后端时，不要只说"帮我写个接口"。要让它说清楚：这个接口解决什么问题、前端会传什么、后端会校验什么、数据会存到哪里、成功返回什么、失败返回什么。这就是后端开发里非常重要的**合同意识**。

#### 10.6 后端常见概念

- **接口**：前端和后端通信的通道。
- **数据库**：长期保存数据的地方。
- **数据表**：数据库里存放某一类数据的结构。
- **字段**：一条数据里的具体信息。
- **业务逻辑**：项目真正的处理流程。
- **权限**：判断用户能不能访问某些功能、查看某些数据、执行某些操作。
- **鉴权**：判断用户是谁，以及他有没有资格访问当前接口。
- **校验**：检查用户提交的数据是否合理。
- **日志**：系统运行时留下的记录，排查问题、追踪错误、分析用户操作经常都要靠日志。
- **部署**：把后端服务放到服务器上，让真实用户可以访问。

#### 10.7 常见后端技术栈

后端常见语言有 JavaScript、TypeScript、Python、Java、Go、PHP、Ruby、Rust 等：

- **JavaScript / TypeScript**：常见于 Node.js 后端，如 Express、NestJS、Fastify。
- **Python**：Django、FastAPI、Flask。适合小脚本、自动化、数据处理、AI 应用、轻量后端。
- **Java**：Spring Boot，成熟、适合企业后台、复杂业务系统、大型项目，但整体结构相对重。
- **Go**：部署简单、性能好，适合现代 API 服务、并发任务、云服务，也适合很多中小型后端项目。
- **PHP**：Laravel 比较常见，是成熟的 Web 开发语言，容易上手。
- **Ruby**：Rails 比较经典。
- **Rust**：也可以写后端，但对新手来说门槛更高。

数据库方面常见的有 MySQL、PostgreSQL、SQLite、MongoDB、Redis 等。

> 你不需要把这些都学会。对 Vibe Coding 来说，最重要的是知道后端有语言、有框架、有数据库、有部署环境。不同技术栈适合不同项目，不能让 AI 随便选，也不能项目写到一半随便换。

#### 10.8 框架比语言更影响项目质量

语言决定用什么写，框架决定按什么规矩写。框架会影响：目录怎么分、接口怎么组织、数据库怎么连接、错误怎么返回、权限怎么接入、日志怎么记录。

框架大概分两类：

- **偏轻**：主要处理路由和 API。灵活，但很多项目规范、模块都需要自己补。
- **更完整**：除了 API 还会提供项目结构、数据库支持、配置管理、权限能力、命令行工具、任务处理等。

对非技术用户来说，如果是后台系统、SaaS、订单系统、权限复杂的项目，更建议优先考虑后者。因为规则越清楚，AI 越不容易乱发挥。

> 无论是前端还是后端，假设你不具备专业知识、不具备驾驭 AI 的能力，就一定要用框架去约束 AI，让 AI 在框架下跳舞。

#### 10.9 让 AI 给"唯一推荐方案"

让 AI 根据项目情况推荐后端语言时，要强调"唯一推荐"。非技术用户最忌讳的就是技术路线摇摆：今天觉得 Python 简单，明天觉得 Go 专业，后天又觉得 Java 稳，项目很快就乱了。

可以这样问 AI：

> 请根据我的项目类型、核心功能、用户规模、数据库方案、部署方式、后期维护难度等，推荐一个最适合当前项目的后端语言。只给一个主推荐方案，不要列多个选择题，并说明为什么选它，为什么不选其他方案。

同样，选框架时：

> 请基于已选后端语言，推荐一个适合当前项目的后端框架。说明它适合什么场景，官方目录规范是什么，哪些能力框架自带，哪些能力需要额外 SDK 或库。优先选择成熟、文档完整、社区活跃、AI 熟悉度高的方案。

> 不要为了看起来高级，选一个资料少、AI 也不熟的方案。你不是在比技术名词，你是在做一个能跑、能维护、能继续迭代的项目。

#### 10.10 和 AI 交流时，先问业务，不要先问代码

新人让 AI 写后端最常见的错误是直接说"帮我写登录接口""帮我写订单接口"。这样 AI 很容易直接开始写代码。但你真正需要的不是先让它写接口，而是先让它把项目的后端技术边界讲清楚。

可以先问：

- 这个项目的后端主要负责哪些业务逻辑？
- 前端需要通过哪些 API 和后端交互？
- 每个 API 背后有哪些业务校验？
- 每个 API 会读写哪些数据库表？
- 哪些接口需要登录后才能访问？
- 哪些接口需要管理员权限？
- 用户身份认证准备怎么做？
- 权限校验放在哪一层？
- 错误返回格式怎么统一？
- 后端适合用什么语言和框架？这个框架自带哪些能力，哪些能力需要额外 SDK 或库？

也可以直接给 AI 这段提示词：

> 先不要写代码。请基于当前项目的前端流程、数据库设计和业务需求，分析这个项目的后端技术方案：后端要负责哪些业务逻辑，前端需要哪些 API，每个 API 涉及哪些数据库表、权限校验、参数校验和异常情况；同时推荐一个适合当前项目的后端语言和框架，并说明框架自带哪些能力，哪些能力需要额外 SDK 或库。

#### 10.11 讨论结果要写进真源文档

后端语言、框架、API 边界、业务规则确认之后，一定要写进项目真源文档。因为后面 AI 搭后端骨架、写接口、接数据库、做权限都要按这份文档来。

文档里至少写清楚：后端语言和框架选什么、为什么选它、哪些方案不选、后端负责哪些业务逻辑、前端需要哪些 API、哪些规则必须在后端校验。

可以直接告诉 AI：

> 请把本次后端语言、框架、API 边界和业务规则写进项目真源文档。后续后端开发必须以这份文档为准；如果要更换语言、框架或核心 SDK，必须先说明原因，并更新文档。

这一步的核心，是**防止 AI 中途换路线**。

#### 10.12 后端最容易翻车的地方

后端比前端更容易藏问题——前端有问题你一眼就能看出来，但后端有问题很多时候表面看不出来：接口没有权限校验，页面可能照样能打开；数据库结构设计不合理，前期可能也能跑；错误处理没做好，测试时可能碰不到；日志缺失，出问题后才发现没法查。

用 AI 写后端时，不能只看功能是否跑通，还要让 AI 检查几个问题：

- 权限有没有做；
- 数据有没有校验；
- 错误有没有处理；
- 接口返回是否统一；
- 数据库结构是否合理；
- 关键操作有没有日志；
- 重要流程有没有测试。

> 这些东西不一定每个小项目都要做到非常复杂，但你必须知道它们存在。否则项目越往后做，后端问题越难救。

### 第 11 章 后端架构搭建

#### 11.1 常见开发场景

很多人没有技术背景，用 Vibe Coding 写后端最直接的操作就是让 AI 上来就写业务：我要个登录功能，我要个订单列表，再加一个支付功能。AI 也能写而且很快，但如果这个后端不是一次性小脚本，而是以后还要继续加模块、继续维护的项目级后端，就不能只按功能往前堆。

如果骨架没定好，AI 很容易给每个功能临时组织一套写法。

> 后端骨架阶段，不是阻止你写业务，而是在写业务之前先定规则。只有等这些规则先定好了，后面写功能才不会乱。后端骨架的核心就是一句话：**这个项目以后能不能按同一套规则继续写下去**。

#### 11.2 先把规则写进项目架构设计文档

这一步不是马上让 AI 搭代码，而是先把两类东西写清楚：

1. **业务边界**：项目有哪些用户场景、主要业务逻辑是什么、主要功能有什么。
2. **工程规则**：后端语言和框架是什么、目录怎么放、接口怎么返回、错误怎么处理、日志怎么记录、后面新增模块应该继续按什么规则写。

把这两类合在一起，就是**项目架构设计文档**。对于没有技术背景的人来说，你不可能靠自己逐行看代码来判断后端结构稳不稳，你真正能做的是先把规则写进文档，然后让 AI 后面所有后端开发都按这份文档执行。

写这份文档时，先做两件事：

**第一，复核后端语言和框架。** 让 AI 回答清楚：为什么用它、为什么不用其他方案、当前项目是否真的需要这个复杂度。要先做这个复核，因为 Vibe Coding 最怕技术路线摇摆。

> 我看很多人一上来就会选择一些比较复杂、重量级的语言，比如 Java、C 语言等。开发语言没有太大的优劣之分，只有适不适合一个项目。如果你做的是常见业务系统、后台管理、内容平台、普通 API 服务，很多时候根本不需要这么重的路线。简单不是唯一标准，但它还必须贴合你的项目。

很多人对自己写的代码不放心，怕出安全问题，听说某个语言"更安全"就把它当成唯一选择——这是要注意的。**安全不是只靠语言决定的，决定安全的往往是业务代码**。

可以这样问：

> 请根据项目架构设计文档，确认当前后端语言和框架是否适合本项目。不要重新发散选型，只判断是否存在为了高级感、性能感或者安全感而过度选型的问题。如果当前语言或框架较重，请说明项目为什么真的需要它，以及后续必须遵守哪些成熟框架规范和最佳实践。

这段提示词的目标只有一个：**把路线钉住**。

**第二，让 AI 说明规则来源。**

- 当前语言的工程规范是什么（依赖管理、配置读取、日志、异常、测试通常怎么处理）；
- 当前框架的最佳实践是什么（不同语言、不同框架都有自己的习惯，不能混着写）；
- 当前项目自己的架构设计文档是什么（语言和框架只能告诉你一般怎么写，但它不知道你的用户、订单、权限、支付这些模块应该怎么分）。

立项文档解决"项目要做什么"，语言和框架解决"代码按什么规范写"，后端骨架要做的就是把这两部分合到项目架构设计文档里。

这里还要加一条约束：**框架最大化利用**。用框架就是为了少踩坑——HTTP 状态码、参数校验、错误处理、中间件、日志、路由分组，很多成熟框架本来就有推荐写法，能用框架自带能力就不要让 AI 在框架外面再手写一套。

这条建议写进 Agent 宪法：

> 后端开发必须遵守当前语言工程规范和框架最佳实践，优先复用框架能力。新增依赖、目录调整、错误码自定义、框架更换，都要先说明原因，并更新项目架构设计文档。

写项目架构设计文档时可以这样要求 AI：

> 请根据立项文档和已确定的后端语言、框架，整理项目架构设计文档。不要写业务代码。文档必须写清楚：后端语言和框架、选择原因、不选其他方案的原因、必须遵守的语言工程规范、必须遵守的框架最佳实践、框架最大化利用原则、目录约束、接口响应规则、错误处理规则、日志规则、数据库连接方式、权限校验入口，以及以后新增模块应该怎么放文件。

#### 11.3 再根据文档搭建最小可运行后端骨架

当项目架构设计文档写清楚之后，再让 AI 根据文档开始搭建后端骨架代码。**一定要强调：是根据文档搭建，不是重新自由发挥**。

可以这样要求 AI：

> 请严格根据项目架构设计文档，基于当前后端语言和框架，按照该语言工程规范、该框架官方文档或主流最佳实践，搭建最小可运行后端骨架。先说明目录结构、目录作用、哪些是框架推荐，哪些是项目自定义。不要先写完整业务功能。骨架必须包含：项目启动入口、环境变量配置、路由分组、健康检查接口、数据库连接、统一响应格式、统一错误处理、日志、基础参数校验、权限中间件占位、README 启动说明。

几个约束不能丢：

1. 必须基于当前语言和框架；
2. 必须遵守语言规范和框架最佳实践；
3. 必须根据项目架构设计文档执行；
4. 必须先解释目录结构，再写代码；
5. 只搭最小可运行骨架，不写完整业务功能。

如果它解释不清楚目录作用、规则来源和自定义原因，先不要继续。这一阶段不要贪功能，主要是先跑通四条线：

1. **启动线**：项目必须能启动、配置必须能读取。端口、数据库地址、密钥不要写死在代码里，要有配置样板（如 `.env.example`），同时要有健康检查接口（如 `/health`）。这条线解决"后端能不能正常跑起来"。
2. **接口线**：路由放哪里要符合当前框架习惯，接口返回要统一。要确认两类成功场景——列表（数组、分页、总数）和对象（详情、创建后的数据、更新后的数据）；失败场景也要统一——参数错误、未登录、无权限、系统异常，HTTP 状态码要尽量复用框架规范和通用标准。如果确实需要业务错误码，先写清楚规则，不能让 AI 想到哪写到哪。这条线解决"以后前端、接口测试、继续用 Vibe Coding 加功能时能不能按同一套接口规则对接"。
3. **业务线**：接口层只处理请求和响应，业务规则放到业务层，数据库读写放到数据层。但不要死记目录名——不同框架叫法不一样（controller/service/repository、handler/usecase/model）。让 AI 按当前框架解释：请求入口在哪里、业务规则在哪里、数据库访问在哪里、参数校验和权限校验在哪里。如果某个框架没有对应概念，就让它说明用什么框架机制替代，而不是自己新造一套。这条线解决"以后新增功能时不会把请求、业务判断和数据库操作全堆在一个函数里"。
4. **运维线**：日志要有、统一错误处理要有、数据库连接要能验证、权限校验要有占位、README 要写清楚怎么启动、怎么配置、怎么测试。这条线解决"项目出问题时能不能启动、能不能查日志、能不能知道数据库和权限这些基础入口在哪里"。

### 第 12 章 后端架构验收

#### 12.1 先验收的不是功能，而是规则

也许你验收时会直接问：这套架构搭建好了吗，能用吗？你大概率会得到这样的回答：好了，可以开始写业务代码了。这肯定不是合格的验收。

对于没有技术背景的你来说，后端架构验收阶段要看的，是**以后继续加功能时这套规则能不能立住**。所以你要让 AI 直接交证据：

> 请根据项目实际业务文档以及项目架构设计文档，对后端架构层代码做详细的审计和验收。每一项必须说明：规则来源、对应文件、验证方式、实际结果、是否通过。没有证据的项目标记为未验收，不允许写通过。

> 如果 AI 只说"已完成、已优化、已符合最佳实践"，但没有文件、命令、返回示例和日志结果，这就不算验收。如果你看不懂它的解释，可以直接说：用最直观的语言告诉我，不要说一堆技术名词。

#### 12.2 验收目录责任

不要被目录数量带偏——目录不是越多越好，关键是每个目录的责任划分是否清晰。让 AI 输出一张目录责任表：

> 请根据当前后端架构，输出目录责任表。每一行包含：目录、主要职责、以后应该放什么、禁止放什么、对应的框架机制、示例文件。请标注哪些目录来自框架推荐或者框架本身，哪些是项目业务自定义的。同时请按当前框架说明：请求入口在哪里，业务规则在哪里，数据库访问在哪里，参数校验在哪里，错误处理在哪里，权限校验入口在哪里。

看这张表时只看三件事：

1. 每个目录是不是只有一个清楚的主要责任；
2. 它有没有说清楚以后应该放什么、不应该放什么；
3. 它有没有说明这些目录是框架推荐还是项目自己定义的。

如果每个目录都写"处理业务逻辑"，那就是没说清。

#### 12.3 用最小模块演练，验收规则能不能落地

目录责任表只能说明 AI 会解释，不能证明它真的会按规则组织代码。所以这一步不要让它直接写业务，而是做一次最小模块演练——不看业务复杂度，只看请求链路有没有分层。

可以这样要求：

> 请做一次最小模块演练，不写完整业务代码，只输出文件清单和调用路径。请说明：需要新增或修改哪些文件；每个文件放在哪个目录；每个文件负责什么；请求从哪里进入；参数在哪里校验；业务规则在哪里处理；数据访问在哪里处理；统一响应和统一错误在哪里处理；这些安排是否符合项目架构设计文档。

看回答时只看两点：

1. 文件放置有没有延续上一节的目录责任表；
2. 请求入口、参数校验、业务规则、数据访问、响应和错误处理有没有被分开。

如果 AI 每次回答的文件都不一样，或者说"都可以写在一个接口里"，这套架构就不适合继续往下堆业务。

#### 12.4 接口返回必须看示例，不能只听"统一"

接口统一返回是后端架构里非常容易被糊弄的一项。AI 很容易说"我已经统一了成功和失败响应"，但你要问具体场景。

接口成功返回是有不同场景的：列表、详情对象、创建、更新、删除、空列表，这些都必须规范统一，要让它给出具体示例。

可以这样要求：

> 请输出当前后端架构的接口响应样例，至少包含：列表成功、详情对象成功、创建成功、更新成功、删除成功、空列表、参数错误、未登录、无权限、资源不存在、系统异常。每个场景都要给出 HTTP 状态码、响应 JSON 示例、由哪个文件或框架机制统一处理。

两个地方不要漏：

1. **列表和对象要分开看**：列表通常涉及数组、分页、总数；对象通常是详情、创建后的数据、更新后的数据。不能一句"data 里放数据"就结束。
2. **错误不能只看业务错误码**：HTTP 状态码要尽量复用框架规范和通用标准。如果项目确实需要业务错误码，也必须写进项目架构设计文档，说明错误码格式、命名规则、谁负责维护。

#### 12.5 验收框架复用和封装边界

用框架的目的就是尽量复用成熟能力。能用框架自带的参数校验、异常处理、HTTP 状态码、中间件、日志机制，就不要让 AI 在框架外面再手写一套。

> 请审查当前后端架构的框架复用和封装边界。哪些能力直接复用了框架？哪些是项目自定义封装？每个自定义封装解决什么问题？如果去掉会有什么影响？是否存在为了显得专业而提前封装的内容？

看它有没有把边界说清楚：能用框架的地方有没有用框架，自定义封装有没有明确理由，这些封装规则有没有写进项目架构设计文档。如果一个封装说不清解决什么问题，也没被当前架构使用，只是"以后可能用"，那就先不要放进架构。

**后端架构要稳，不是要复杂。** 这条原则也要被后端架构实施真源文档收录，并在 Agent 宪法中引用：

> 后端开发必须遵守《后端架构实施真源文档》中的框架最大化利用原则。HTTP 状态码、参数校验、错误处理、路由分组、中间件、日志、依赖注入等能力，优先复用当前框架推荐机制。确需自定义时，必须说明原因，并更新后端架构实施真源文档。

#### 12.6 验收启动、配置、数据库和日志，要固定启动证据

这一步不是让你看懂所有命令，也不是让你学会部署。它的作用是把启动方式、配置来源、数据库连接和日志入口固定下来。否则后面继续开发时，AI 很可能又要重新猜：依赖怎么装、服务怎么启动、端口是多少、数据库配置在哪里、健康检查怎么访问。

让它交一个运行证据包：

> 请输出后端架构运行证据包。必须包含：依赖安装命令、启动命令、服务监听端口、健康检查命令和结果、配置读取位置、.env.example 内容检查、数据库连接验证方式和结果、请求日志示例、错误日志示例。每一项都要写明实际执行结果。没有执行的项目标记为未验证。

你不需要逐行理解命令。只看它有没有把"怎么启动、用什么配置、数据库是否连上、日志在哪里看"说固定。如果 AI 只说"理论上可以启动"，不算通过；如果这次说一个启动命令，下次又换另一套，说明运行规则还没有收住。

#### 12.7 把验收结果收口成实施真源文档

到了最后，不要让 AI 散着回答，也不要让它重新把前面的内容讲一遍。这一步只做收口：哪些已经通过、证据在哪里、哪些还没有验证、当前后端架构能不能进入业务开发。

> 请汇总本次后端架构验收结果，不要重新发散，也不要重复解释前面的内容。请按以下结构输出：验收项、是否通过、证据位置、未通过原因、下一步处理、是否影响进入业务开发。最后给出明确结论：当前后端架构是否可以开始写业务功能。没有证据的项目必须标记为未验证，不能写基本完成。

关键项没有通过（目录责任没说清、接口返回没有示例、错误处理没有统一、启动证据没有固定），就不要急着写业务。关键项都通过了，下一步不是把所有架构约束都塞进 Agent 宪法——Agent 宪法只放最高层约束，不承载所有后端细则。正确做法是：让 AI 根据验收通过的结果整理出一份详细的《后端架构实施真源文档》，再在 Agent 宪法里写一条引用关系：

> 后端开发必须先读取并遵守《后端架构实施真源文档》。新增或修改后端功能前，必须确认目录责任、接口规则、错误处理规则、启动证据和框架复用边界。任何目录调整、规则变更、框架更换或新增关键依赖，都必须先说明原因，并同步更新后端架构实施真源文档。

详细规则放在实施真源文档里，Agent 宪法只负责引用它、约束 AI 不能绕开它。以后业务代码越写越多，也不会把 Agent 宪法越写越臃肿。

> 验收通过后，记得让 AI 先提交一次 Git，把这套后端架构作为第一个稳定版本。后面继续写业务功能时，就基于这个版本往下加；如果还有没验证的项，就不要让 AI 用"基本完成"糊过去，没验证就是没验证。

### 第 13 章 后端安全：接口与系统权限设计

#### 13.1 先把安全拆成边界

后端安全是分散在请求处理的每一步里的，可以把它想成请求进来后要过的一道道关卡，每道关卡只负责检查一个方面：

1. **身份认证**：这个请求是谁发的、有没有登录。
2. **权限控制**：这个人能不能做这件事。
3. **输入校验**：前端传进来的数据可不可信。
4. **数据归属**：这条数据是不是属于他自己。
5. **注入防护**：用户填的内容会不会被后端当成命令去执行。

这几条防线各自挡的风险不一样，少一条就是一个口子，所以不能用一句"我做了安全处理"打包带过。要让 AI 按这些维度把边界一条条说清楚：

> 请根据项目架构设计文档和当前业务功能，输出后端安全边界表。不要只说已做安全处理。请按接口输入、登录状态、系统权限设计、密码规则、数据归属、注入风险分别说明：风险是什么，当前在哪里处理，处理规则是什么，如何验证，哪些还没有验证。

如果它回答得太专业、你看不进去，别硬着头皮装懂，直接跟它说"用大白话重讲一遍，别用技术名词"。**你看不懂的安全说明，本身就不算合格**——这一点贯穿后面每一步，回答看不懂就让它换种说法再讲。

> 如果这些边界都说不清，后面就不要急着问能不能上线。

#### 13.2 接口输入默认不可信

接口安全里最重要的一条原则是：**默认前端输入不可信**。

因为后端接口本质上是公开的。真正想钻空子的人根本不会老老实实从你设计好的页面进来——他们用一些现成工具就能绕开整个前端，直接对着后端接口发请求。你在页面上做的格式校验、按钮置灰、字段隐藏，对他们一点用都没有。

> 前端校验主要是为了用户体验以及防止无意义的 API 请求占用服务器资源。而真正影响业务结果的内容，必须在后端重新判断一遍。比如价格、数量、角色、用户 ID、订单归属、数据状态，这些都不能因为前端传了，后端就当成事实。

可以这样要求：

> 请审查当前后端接口的输入安全。不要只说已做参数校验。请按接口说明：哪些参数来自前端，哪些参数会影响身份、系统权限、金额、库存、数据归属或数据库写入；每个关键参数在哪里校验；校验失败返回什么；是否有示例请求证明。

你不用看懂全部代码，只看它有没有把关键输入和普通输入分开。普通输入错了，最多是页面显示不对；关键输入错了，可能就是越权、改错数据、金额错误、库存错误。

#### 13.3 密码和管理员账号必须强制约束

安全不只是防别人乱传参数，也包括强制用户遵守基本规则，最典型的就是账号和密码。这里有两层要管。

**第一层是密码强度。** 攻击者破解密码靠的不是一个个手动猜，而是拿几百万条常见密码组成字典让程序批量去试，这叫暴力破解。如果再用别处泄露的账号密码来碰运气，就叫撞库。`abc123`、`123456` 这种密码几秒就会被试出来。

所以注册、登录、修改密码、重置密码、创建管理员账号这些接口，不能允许过弱密码一路通过。

**第二层是密码怎么存。** 密码绝不能明文写进数据库，必须先经过哈希（一种不可逆的处理）再存，否则数据库一旦泄露，所有人的密码就等于直接公开了。尤其是管理员账号——管理员密码太弱，影响的不是一个用户，而是整个系统。

> 请审查当前后端的账号和密码安全规则。请说明注册、登录、修改密码、重置密码、创建管理员账号分别在哪里校验密码强度。请检查是否包含密码长度、字符组合、常见弱密码拦截、密码是否哈希存储（禁止明文保存）、登录失败次数限制、管理员账号额外要求。没有实现的项目标记为未验证，不要默认安全。

至少要确认三件事：密码规则存在、管理员账号有更严格要求、后端真的会拦截明显弱密码。

#### 13.4 系统权限设计要提前定

系统权限设计是后端安全里最容易被 AI 写乱的地方。

先分清两件事：一是登录，确认你是谁，这叫认证；二是权限，确认你能做什么，这叫授权。**登录通过，不代表什么都能做。**

权限出问题通常是两种越权：

- **垂直越权**：普通用户调用了本该只有管理员能用的接口，比如改别人角色、删后台数据。
- **水平越权**：人是登录的、身份也合法，但他动了不属于自己的数据——比如把链接里的订单号一改，就看到了别人的订单。

很多 Vibe Coding 项目只做了"要不要登录"，却漏掉"登录之后能不能动这条数据"，水平越权大多就是这么来的。

> 所以哪些接口必须登录、哪些只有管理员能访问、哪些普通用户只能操作自己的数据，这些不能写到哪算哪。如果项目涉及多角色或多层级权限（普通用户、商家、运营、管理员），就应该在立项或功能设计阶段先讲清楚。

可以这样要求：

> 请根据项目功能设计文档，输出系统权限设计表。每个接口都要说明：是否需要登录、允许哪些角色访问、是否只能操作自己的数据、管理员是否可以访问、权限判断在哪个文件或中间件处理、无权限时返回什么、有没有测试证明。

这张表不是形式主义，它是给后面继续写业务用的规则。有了它，AI 后面新增接口时才知道应该沿用哪一套系统权限设计，而不是每次重新猜。

#### 13.5 不要让用户输入直接进入关键语句

还有一类常见风险叫注入。原理一句话能说清：**用户输入的内容，本来只该是"数据"，却被后端当成了"指令"去执行。**

最典型的是 SQL 注入——如果后端把用户填的内容直接拼接成数据库查询语句，别人就能在输入框里填入一段精心构造的内容，让数据库执行计划外的操作，比如绕过登录、拖走整张用户表。除了数据库，系统命令、HTML、模板拼接也有同类问题。

> 用户输入的内容，不能被后端直接拼进数据库语句、系统命令、查询条件、HTML 或模板内容里。

正确的防法不是让 AI 自己发明一套过滤规则，而是优先用框架和数据库工具提供的成熟机制：参数化查询（也叫预编译，把"指令"和"数据"分开，让用户输入永远只当数据处理）、ORM、查询构造器、白名单字段校验、内容转义或内容清洗。

> 请检查当前接口是否存在注入风险。重点检查 SQL 注入、数据库查询注入、命令注入、HTML 或模板内容注入。请说明哪些位置使用了框架 ORM、参数化查询、白名单校验、内容转义或内容清洗；如果存在字符串拼接用户输入的情况，请标记为风险并给出修复方案。

不要让 AI 只说"已经过滤"。只要让它讲清楚一件事：用的到底是框架、数据库本身自带的成熟方案，还是它自己临时写的一段过滤代码。如果是后者，还拍胸脯说"已经安全"，就要特别警惕——**自己造的防护，往往就是漏得最多的**。

#### 13.6 该严的地方严，不该复杂的地方别乱加

安全不是把代码写得越复杂越好。一听你说"安全"，有些 AI 就会拼命加防御——给那些根本不可能发生的情况也写上判断：前面已经强制登录了，后面又反复判断"如果没登录怎么办"；参数类型已经校验过了，再叠一堆永远为真或永远为假的条件分支。

这类分支在实际运行中根本走不到，专业上叫**死代码**。它不会让系统更安全一分，只会白白增加代码量、消耗资源，还会把真正的业务逻辑淹没，让人误以为"判断这么多，肯定很安全"。

真正要严的是会影响业务结果的地方：是否登录、是否有权限访问这条数据、是否能修改这条记录、金额和库存是否符合业务规则。这些必须由后端按真实业务逻辑校验。而像输入框提示、格式提醒、前端展示优化这种，就不必上升成复杂的后端安全逻辑。

> 请审查当前接口安全逻辑是否存在过度防御。请区分：必须校验的业务规则、框架已经处理的基础校验、可以由前端体验层处理的提示校验，以及永远不会触发的防御分支（死代码）。不要为了安全感新增重复判断或不可能成立的条件。任何新增安全封装都必须说明解决什么真实风险。

#### 13.7 安全要点小结

总结起来就几句话：前端传来的关键数据，后端要重新判断；密码和管理员账号，后端要强制约束；多角色项目，系统权限设计要提前写清楚；用户输入不能直接进入关键语句；该严的地方必须严，不该复杂的地方不要乱加。

> 说到底，后端安全也不要听 AI 说"已经做了安全处理"，要看它把风险、规则、处理位置和验证证据一项项交出来。

## 第五部分：工程实践

### 第 14 章 Git：Vibe Coding 的后悔药

#### 14.1 Git 是什么

你可以把 Git 理解成项目里的版本存档工具。它不是用来写代码的，而是用来记录代码每一次重要变化的。

比如 AI 帮你搭好了项目骨架，提交一次；AI 帮你做完登录功能，提交一次；AI 帮你接好了数据库，再提交一次。后面如果 AI 把项目改乱了，你就可以回到之前某一个稳定版本。

> 这就像打游戏一样。遇到难打的 boss，或者不确定自己能不能打过的时候，最好先存档。打赢了继续往前走，打输了就从存档点还原，避免前面的资源和进度全都浪费掉。

用 AI 写代码也是一样——每次让 AI 做比较大的修改之前，都应该先确认当前项目有没有一个稳定存档。

#### 14.2 为什么 AI 项目更需要 Git

因为 AI 改代码很快，而且一次可能改很多文件。你如果看不懂代码，就更难判断它到底改了什么。

没有 Git，AI 改坏了，你只能继续让它补。补着补着，问题越来越多，项目越来越乱。到最后你甚至会放弃整个项目。而一旦有了 Git，你不仅能知道它当前改了什么，一旦当前阶段出了问题或者效果不满意，你可以直接回退。

> 所以越是刚开始用 AI 做项目，越要早点用 Git。不要等项目已经乱了，才想起来要不要保存版本。那时候很多问题已经混在一起了，想救会非常麻烦。

#### 14.3 养成阶段提交的习惯

Git 最重要的不是装上，而是要会提交。规则很简单：**每完成一个明确阶段，就提交一次。**

也不是每改一行就提交，而是项目完成一个小阶段并且能正常运行，你确认结果可以接受，就提交一次。比如项目创建完成提交一次，目录结构搭好提交一次，首页能打开提交一次，接口能响应提交一次，数据库能连接提交一次，一个核心功能做完再提交一次。

这条规则可以写进 Agent 宪法：

> 每完成一个明确阶段，必须先说明本次改了哪些内容，验证是否通过，然后再提交一次 Git 记录。

这样 AI 就不会只顾着往前写，而是会在每个阶段停下来做总结和存档。

#### 14.4 Git 和 GitHub、码云不是一回事

- **Git** 是本地版本管理工具——你电脑里的存档工具，是一个版本管理软件。不光是代码，甚至你写一篇文章都可以用 Git 来管理文章版本。
- **GitHub 和码云（Gitee）** 是把存档上传到网上保存的地方，就像网盘，区别在于网盘不能做版本管理。

如果人在国内想访问更顺，可以用码云；如果做开源项目或希望更多人看到，可以用 GitHub；或者一个项目配置两个远程仓库，一个推 GitHub 一个推码云。

> Git 本身就是一个版本管理工具，即便你不想把代码推送到远程，只要在电脑上安装了 Git 做了版本管理，你也是可以直接使用项目提交或代码回退功能的。

远程仓库在以前最大的用途是：假设你在多个电脑异地工作、多个人一起做一个项目，或者你电脑损坏之后代码不会丢失。

#### 14.5 推远程之前，先确认哪些东西能公开

如果项目只是自己用，或者还没准备公开，远程仓库就应该设成私有。

如果项目准备开源，也不是非要把整个项目文件夹全部开源出去——因为项目里不只有代码，它可能还有立项文档、商业计划、产品规划、Agent 宪法、内部策略、真实 case 分析。

> 尤其是现在这个全民 AI 干代码的时代，代码、页面、功能甚至一整套项目结构，要复制都很快。真正重要的反而是你想出来的 case——你为什么做这个项目、面对的真实场景是什么、怎么理解用户需求、怎么设计产品路线、怎么约束 AI 按你的思路去开发。这些东西很多时候比代码本身更值钱。

所以开源项目真正应该公开的是你愿意让别人看到、使用、学习和参与的部分，比如代码、README、安装方式、使用说明、贡献指南。但商业计划、内部规划、真实 case、Agent 宪法这些内容，除非明确决定公开，否则不要让它们跟着代码一起进入开源仓库。

推远程之前，最好先让 AI 列出文件清单：这次准备提交哪些文件？哪些文件会推到远程？有没有不该公开的内容？确认没问题再让它推送。

#### 14.6 Git 三件事

用 AI 写代码，Git 不是可有可无的工具。它是你的后悔药，也是项目的存档点。正式让 AI 写项目之前，至少做三件事：

1. 安装 Git；
2. 养成每完成一个阶段就提交一次的习惯；
3. 推远程之前，确认仓库是私有还是公开，确认哪些文件能推、哪些不能推。

### 第 15 章 真源文档：让项目不靠猜继续写

整个 Vibe Coding 流程里反复出现一个概念——**真源文档**。它的本质是把"已经讨论清楚的内容"和"必须遵守的规则"固定下来，让项目不靠猜继续写。

常见的真源文档有：

- **立项文档**：项目是什么、面向谁、第一版做什么、不做什么。
- **技术选型文档**：产品形态、前后端技术栈、数据库、主要框架和 SDK、为什么这么选、什么时候重新评估。
- **项目架构设计文档**：业务边界、工程规则、目录约束、接口规则、错误处理、日志规则、框架最大化利用原则。
- **后端架构实施真源文档**：目录责任、接口规则、错误处理、启动证据、框架复用边界。
- **数据库设计文件**（`schema.sql` 或 migration）：表结构、字段、关系、约束。
- **系统权限设计表**：每个接口的认证、授权、数据归属。

每次 AI 想引入新框架、新库、新架构、修改规则时，都必须先回到对应真源文档看一眼：是不是符合当前路线？有没有破坏项目边界？有没有让项目变复杂？确需变更时，先说明原因，再更新文档。

> 沟通清楚、规则写清楚、证据留下来，项目后面才不会靠猜继续写。

### 第 16 章 全流程速查

把一个完整 Vibe Coding 项目从 0 到 1 的关键步骤收口如下：

1. **立项**：创建空项目文件夹（英文命名）→ 和 AI 讨论产品 → 明确告诉 AI"暂时不要写代码"→ 生成立项文档。
2. **技术选型**：确定产品形态 → 让 AI 给唯一推荐方案 → 写进技术选型文档。
3. **Agent 宪法**：让 AI 在当前工具识别的位置创建宪法 → 短、硬、清楚 → 写长期原则、禁止事项、验收标准。
4. **前端骨架**：定设计风格 → 选前端框架 + UI 组件库 → 定目录/模块/组件复用规则 → 定设计 token/多语言/主题 → 写进实施计划 → 搭最小可运行骨架 → 提交 Git。
5. **数据库设计**：从业务找对象 → 确认对象关系 → 选数据库服务 → 按三大范式设计表结构 → 规范常见字段 → 先生成 schema/migration 再建表 → 提交 Git。
6. **后端骨架**：复核语言框架 → 写项目架构设计文档 → 搭最小可运行骨架（启动线/接口线/业务线/运维线）→ 验收 → 整理实施真源文档 → 提交 Git。
7. **后端安全**：拆边界 → 接口输入默认不可信 → 强制密码和管理员账号规则 → 提前定系统权限设计表 → 防注入 → 不要过度防御。
8. **持续迭代**：每完成一个阶段提交一次 Git → 每次变更先更新真源文档 → 推远程前确认文件清单。
