# 全流程 AI 对话 Prompt 模板库

> 这是 [AI协作全流程操作手册.md](./AI协作全流程操作手册.md) 的配套工具箱。
> 主手册告诉你"每个阶段做什么、怎么决策"；本库提供"每个阶段可直接复制的 prompt"。
>
> **用法**：进入某个阶段 → 翻对应部分 → 复制 prompt → 把 `[方括号]` 内容替换成你的实际情况 → 发送给 AI。

---

## 序：Prompt 三种类型

每个阶段的 prompt 分三类，对应四段式工作循环：

| 类型 | 用在 | 目标 | 特征 |
|---|---|---|---|
| 🗣️ **挖掘型** | 对话挖掘期 | 把模糊需求挖清楚 | 让 AI 反问、补盲区、给判断 |
| ⚡ **执行型** | 执行交付期 | 让 AI 按规则落地 | 范围锁死、验收前置 |
| ✅ **验收型** | 验收固化期 | 让 AI 给证据 | 禁止模糊词、要文件/命令/输出 |

**核心原则**：挖掘型让你思考，执行型让 AI 动手，验收型让 AI 证明。

## 第 0 部分：通用模板（每阶段都用）

### 0.1 全局开场模板（每次对话必用）

```
当前阶段：[立项 / 选型 / 宪法 / 前端 / 数据库 / 后端 / 业务 / 验收 / 调试 / 反哺]
你的角色：[产品合伙人 / 架构顾问 / 数据库架构师 / 后端架构师 / 前端架构顾问 / 安全审查员 / 代码审查员 / 调试员 / 复盘员]

上下文：
- 立项文档：[贴或引用路径]
- 技术选型文档：[贴或引用路径]
- 真源文档：[贴或引用路径]
- Agent 宪法：[贴或引用路径]

约束：给唯一推荐，不要列多方案让我选
本轮范围：只做 [具体任务]，不一次性做完
验收标准：[具体可检查标准]

完成后用验收清单自检，每项给证据。
```

### 0.2 全局收尾模板（每次宣布完成前必用）

```
在说"已完成"之前，请按以下清单自检：
1. 走验收清单，每项给证据（文件 / 命令 / 输出 / 日志）
2. 未验证项标记"未验证"，不允许写"基本完成"
3. 回归检查（相邻功能有没有被破坏）
4. 同步真源文档（如有规则变更）
5. Git 提交（commit message 用 feat/fix/chore 前缀）
6. 反哺 Skill（如发现盲点）

最后给明确结论：能否进入下一阶段 / 下一轮。
```

### 0.3 通用追问模板（5 种场景）

**场景 A：AI 回答太泛**

```
你刚才的回答太泛。请基于我的项目情况（[贴关键上下文]）重新回答：
1. 给具体判断，不要"通常来说""一般情况下"
2. 给唯一推荐，不要列多方案
3. 说明为什么不选其他方案
```

**场景 B：AI 用模糊词**

```
禁止使用"基本完成""应该没问题""理论上可以""大概""可能"。
要么"通过"，要么"未通过"加原因。
刚才说的"基本完成"——具体哪些通过、哪些未通过？给清单。
```

**场景 C：AI 列多方案甩锅**

```
你列了 A/B/C 三套让我选——这是甩锅。
请基于我的项目情况给唯一推荐，并说明为什么不选另外两套。
如果三套真的差不多，说明选型维度不够，请补充维度后重新判断。
```

**场景 D：AI 解释看不懂**

```
用最直观的语言告诉我，不要说一堆技术名词。如果我看不懂，说明你还没讲清楚。
特别是 [挑一处看不懂的]，用大白话重讲一遍，举个生活中的例子。
```

**场景 E：要证据不要结论**

```
请不要只说"已完成"。每一项必须说明：
- 规则来源（哪份文档哪条）
- 对应文件（路径 + 行号）
- 验证方式（命令 / 测试用例）
- 实际结果（输出 / 截图 / 日志）
- 是否通过
没有证据的项目标记为"未验证"。
```

### 0.4 通用反甩锅模板

```
请检查你的回答：
1. 有没有列多方案让我选？有就改成唯一推荐
2. 有没有用"建议""尽量""通常"等软词？有就改成硬规则
3. 有没有只给结论不给证据？有就补文件 + 命令 + 输出
4. 有没有回避我的问题？有就正面回答
改完重新发我。
```

## 第 1 部分：立项阶段

### 1.1 🗣️ 项目启动 prompt

```
当前阶段：项目立项，只做产品讨论和立项设计，暂时不要写代码。
你的角色：我的产品合伙人。

我想做的是：[一句话描述]
面向用户：[谁]
解决什么问题：[什么]
参考产品：[链接 / 截图，可选]

请按以下方式追问我：
1. 围绕项目类型、目标用户、核心问题逐项深挖
2. 主动指出我没想到的盲区（权限、数据结构、状态字段、后期扩展、合规风险、性能瓶颈、跨端兼容、安全边界）
3. 帮我判断第一版 MVP 该做什么、不该做什么
4. 给唯一推荐，不要列多方案让我选
5. 讨论清楚后告诉我"现在可以写立项文档了"，等我确认再写

记住：本轮不写代码、不建项目、不装依赖。
```

### 1.2 🗣️ 盲区追问 prompt

```
请主动指出我没想到的问题。特别检查：
1. 权限模型：哪些角色？每个角色能做什么不能做什么？
2. 数据归属：用户 A 的数据用户 B 能不能看到？
3. 状态机：核心业务对象（订单/任务/文章）有哪些状态？状态怎么流转？
4. 后期扩展：第一版做的决定会不会限制后期扩展？
5. 合规风险：涉及用户数据 / 支付 / 内容审核吗？
6. 性能瓶颈：哪些场景可能高并发？
7. 跨端兼容：要在哪些端跑？数据怎么同步？
8. 安全边界：哪些操作必须登录？哪些必须二次验证？

不要只回答我问的，要补我没问的。给判断，不要列方案。
```

### 1.3 🗣️ MVP 边界 prompt

```
基于刚才讨论，请帮我判断第一版 MVP 边界：

必须做（P0）：[核心闭环必需的功能]
建议做（P1）：[提升体验但非必需]
可选做（P2）：[后期再做也行]
明确不做：[会拖慢第一版或偏离核心的]

每项给理由：为什么这个是 P0？为什么这个不做？
如果我不做 P0 里的某项，会怎样？
如果我想加 P2 里的某项到第一版，会牺牲什么？
```

### 1.4 ⚡ 立项文档生成 prompt

```
现在请把刚才的讨论结果写入项目根目录的 docs/立项文档.md。
必须包含 8 项内容：
1. 项目立项说明（是什么 / 面向谁 / 解决什么问题 / 为什么值得做 / 第一版核心目标）
2. 项目核心功能和边界（第一版做什么、不做什么）
3. 产品流程和用户路径（页面怎么走、功能怎么串、数据怎么流动）
4. 数据和业务对象（用户、订单、任务、文章、商品、文件、评论、权限等）
5. 技术路线初步判断（网页 / 小程序 / App / Web MVP）
6. 项目爆点和差异点
7. 后期规划和里程碑（第一版、第二版、长期版本）
8. 验收规则（第一版做到什么程度才算完成——必须有可检查标准）

生成后让我 review，我确认后再提交 Git。
不要在文档里留"建议方案"或"可选方案"——只写最终决定。
```

## 第 2 部分：技术选型阶段

### 2.1 🗣️ 选型挖掘 prompt

```
这是我的立项文档：[贴文档]
产品形态：[Web / 小程序 / App / 后台]
部署要求：[私有 / 公有 / 本地]
维护能力：[1人 / 小团队 / 中型团队]
已有技术栈偏好：[如果有，写出来]

请给出唯一推荐技术栈，不要列多个方案。说明：
- 前端框架 + UI 组件库
- 后端语言 + 框架
- 数据库
- 主要框架和 SDK
- 为什么选它们（结合我的项目情况，不要泛泛而谈）
- 为什么不选其他常见方案（至少列 3 个被淘汰方案及理由）
- 哪些情况才需要重新评估技术路线
```

### 2.2 🗣️ 选型深挖 prompt

```
请基于以下维度重新评估你的选型：
1. 团队学习成本：我要花多久能上手？有没有更熟的方案？
2. 生态成熟度：关键依赖还在维护吗？最新 release 时间？
3. 长期维护：3 年后这个技术栈还在吗？有没有替代风险？
4. 招聘成本：如果要招人，这个技术栈好招吗？
5. 性能边界：这个技术栈能撑住 [预期用户量] 吗？
6. 跨端能力：如果要扩展到 [其他端]，这个技术栈支持吗？

每个维度给我的选型打分（1-10），并说明扣分点。
如果总分低于 7 分，请重新推荐。
```

### 2.3 ⚡ 选型文档生成 prompt

```
请把刚才的选型结论写入 docs/技术选型文档.md。
结构：
1. 最终选型（前端 / 后端 / 数据库 / SDK，每个写明版本）
2. 选的理由（结合项目情况）
3. 不选其他方案的理由（至少 3 个被淘汰方案）
4. 重评估条件（什么情况下需要重新选型）
5. 风险提示（已知风险 + 应对策略）

不要在文档里留"建议方案"或"可选方案"——只写最终决定。
```

## 第 3 部分：Agent 宪法阶段

### 3.1 🗣️ 宪法草稿 prompt

```
当前阶段：Agent 宪法创建。暂时不要写业务代码。
你的角色：项目架构顾问。

上下文：
- 立项文档：[贴]
- 技术选型文档：[贴]

请帮我起草 AGENT_CONSTITUTION.md。要求：
1. 短：不超过 200 行，AI 能一次读完
2. 硬：每条都是必须遵守的硬规则，不写"建议""尽量"
3. 清楚：每条能直接判断对错
4. 只写长期原则，不写具体技术细节（目录结构、字段命名等放真源文档）
5. 必须包含：项目专属条款 / 禁止事项 / 验收标准 / 框架优先原则

先列大纲让我确认，确认后再写完整版。
```

### 3.2 🗣️ 宪法自检 prompt

```
请检查你刚才写的宪法每条规则：
1. 这条是长期的吗？只在当前阶段有效的不要写进来
2. 这条能直接判断对错吗？模糊的请改写或删除
3. 这条和真源文档会不会重复？重复的请移到真源文档
4. 项目专属条款够不够？至少补 5 条本项目特有的约束
5. 禁止事项够不够？至少列 10 条"不要做什么"
6. 验收标准够不够？每条规则都要有对应的验收方式

改完重新发我。
```

### 3.3 ⚡ 宪法落地 prompt

```
请写入 AGENT_CONSTITUTION.md，并在各 IDE 入口文件引用它：
- AGENTS.md（Codex）
- CLAUDE.md（Claude Code）
- .trae/rules/project.md（Trae）

入口文件只写一行引用，不复制内容（避免多处不同步）。
完成后给：宪法文件路径 + 各入口文件路径 + 引用方式。
```

## 第 4 部分：前端骨架阶段

### 4.1 🗣️ 前端架构挖掘 prompt

```
当前阶段：前端骨架搭建。本轮只搭最小可运行骨架，不写业务页面。
你的角色：前端架构顾问。

上下文：
- 立项文档：[贴]
- 技术选型文档：[贴]
- Agent 宪法：[贴]

请先输出《前端架构实施计划》，包含：
1. 设计 token 定义（颜色 / 字号 / 间距 / 圆角 / 阴影，给具体值）
2. UI 组件库选择（结合技术栈，给唯一推荐）
3. 目录结构规则（每个目录的职责边界）
4. 路由方案（基于文件 / 配置，理由）
5. 状态管理方案（全局 / 局部边界）
6. 接口层封装规则（请求 / 响应 / 错误 / loading）
7. 多端适配策略（如适用）
8. 本轮交付清单（最小可运行：能启动 / 首页能打开 / token 已定 / 组件库接入）
9. 本轮不做什么

我确认计划后再开始写代码。
```

### 4.2 🗣️ 设计 token 追问 prompt

```
设计 token 还缺：
1. 暗色模式（如果支持）：暗色变量定义
2. 响应式断点：sm / md / lg / xl 的具体值
3. 字号阶梯：h1-h6 + body + caption + small 的具体值
4. 间距阶梯：4 / 8 / 12 / 16 / 24 / 32 / 48 / 64 的使用场景
5. 圆角阶梯：sm / md / lg / full 的具体值
6. 阴影阶梯：sm / md / lg 的具体值
7. 主色 / 辅色 / 警告色 / 错误色 / 成功色的具体值 + hover / active 状态

每个 token 给具体值，不要写"根据设计稿定"。
```

### 4.3 🗣️ 目录结构追问 prompt

```
目录结构请补：
1. 每个目录放什么（具体文件类型）
2. 每个目录不放什么（明确禁止）
3. 命名规则（文件名 / 组件名 / 变量名 / CSS 类名）
4. 子目录规则（什么时候建子目录）
5. 跨目录引用规则（api 能引用 utils 吗？components 能引用 pages 吗？）

给一个完整的目录树示例，标注每个目录的职责。
```

### 4.4 ⚡ 骨架执行 prompt

```
现在按文档搭骨架。本轮范围锁死：
- 项目能启动
- 首页能打开（一个空白页或占位页即可）
- 设计 token 已定义并生效
- UI 组件库已接入（能用一个 Button 验证）
- 目录结构按文档创建（空目录也建出来，带 .gitkeep）
- 路由结构搭好（首页路由 + 一个示例路由）
- 接口层封装好（写一个示例请求验证）

不要写业务页面。不要一次性做完所有功能。
完成后按验收清单自检，每项给证据（文件 / 命令 / 输出）。
```

### 4.5 ✅ 前端验收 prompt

```
请按以下清单自检并交证据：
1. 项目能启动？给启动命令和成功输出
2. 首页能打开？给浏览器 URL 和截图描述
3. 设计 token 已定义？给文件路径 + 关键变量列表
4. UI 组件库接入？给一个使用示例代码
5. 目录结构符合文档？给目录树
6. 路由结构搭好？给路由配置文件路径
7. 接口层封装完成？给示例请求代码 + 实际响应

每项给文件 / 命令 / 输出作为证据。
未验证项标记"未验证"，不允许写"基本完成"。
最后给明确结论：能否进入下一阶段。
```

## 第 5 部分：数据库设计阶段

### 5.1 🗣️ 业务对象挖掘 prompt

```
当前阶段：数据库设计。先出设计文件，确认后再建表。
你的角色：数据库架构师。

上下文：
- 立项文档：[贴业务对象部分]
- 技术选型文档：[贴数据库部分]
- Agent 宪法：[贴数据相关条款]

请按以下顺序推进，每步等我确认再进下一步：

第 1 步：业务对象清单 + 关系图
- 列出所有业务对象
- 标注对象间关系（1:1 / 1:N / N:N）
- 用 Mermaid ER 图可视化

第 2 步：核心表设计
- 每张表的字段（名称 / 类型 / 约束 / 默认值 / 注释）
- 主键 / 外键 / 唯一索引 / 普通索引
- 状态字段的枚举值
- 软删除字段
- 时间字段

第 3 步：三大范式校验
- 1NF：字段原子性
- 2NF：消除部分依赖
- 3NF：消除传递依赖
- 反范式设计需说明理由 + 同步策略

第 4 步：生成 schema.sql / migration 文件
- 不要直接连库建表
- 文件路径：db/schema.sql 或 migrations/

每步给完让我 review，确认后再进下一步。
```

### 5.2 🗣️ 字段规范追问 prompt

```
请主动检查盲区：
1. 金额字段是不是 decimal？（不是就改，禁止 float）
2. 密码字段是不是哈希存储？（不能明文，用 bcrypt/argon2）
3. 软删除字段是否一致（统一 deleted_at，不要混用 is_deleted）
4. 状态字段枚举值是否完整（覆盖所有业务状态）
5. 索引是否覆盖高频查询（外键、状态、时间）
6. 是否有 N:N 关系漏了中间表
7. 时间字段时区策略（统一 UTC 还是本地时间）
8. 字符集是否统一 utf8mb4
9. 主键策略（自增 ID / UUID / 业务 ID）
10. 外键约束要不要建（性能 vs 一致性权衡）
```

### 5.3 🗣️ 状态机追问 prompt

```
请为每个有状态的业务对象画状态机：
1. 列出所有状态
2. 列出所有状态流转（A → B，触发条件是什么）
3. 哪些流转需要权限校验
4. 哪些流转需要通知 / 钩子
5. 异常状态怎么处理（卡在中间状态怎么办）

用 Mermaid stateDiagram 可视化。
```

### 5.4 ⚡ schema 生成 prompt

```
请把设计结果写入 docs/数据库设计文档.md，并生成 db/schema.sql。
文档结构：业务对象 / 关系图 / 核心表 / 字段规范 / 软删除策略 / 三大范式校验 / 索引策略 / 状态机。
schema.sql 必须可独立执行（不依赖手动操作），包含：

3. 列出文档和代码描述不一致的规则
每项给文件 + 行号证据。
确认后我决定：是改文档对齐代码，还是改代码对齐文档。
```

### 12.4 AI 用模糊话糊弄

```
禁止使用"基本完成""应该没问题""理论上可以""大概""可能"。
要么"通过"，要么"未通过"加原因。
刚才说的"基本完成"——具体哪些通过、哪些未通过？给清单。
```

### 12.5 AI 列多方案甩锅

```
你列了 A/B/C 三套让我选——这是甩锅。
请基于我的项目情况给唯一推荐，并说明为什么不选另外两套。
如果三套真的差不多，说明选型维度不够，请补充维度后重新判断。
```

### 12.6 AI 不给证据

```
请不要只说"已完成"。每一项必须说明：
- 规则来源（哪份文档哪条）
- 对应文件（路径 + 行号）
- 验证方式（命令 / 测试用例）
- 实际结果（输出 / 截图 / 日志）
- 是否通过
没有证据的项目标记为"未验证"。
```

### 12.7 AI 解释看不懂

```
用最直观的语言告诉我，不要说一堆技术名词。如果我看不懂，说明你还没讲清楚。
特别是 [挑一处看不懂的]，用大白话重讲一遍，举个生活中的例子。
```

### 12.8 项目失控信号（出现就停）

```
出现以下任一信号，立刻停下来重新对齐：
1. AI 改的代码和我要的对不上
2. 一轮改了超过 10 个文件
3. AI 反复改同一处代码（3 次以上）
4. AI 说"基本完成"但给不出证据
5. 验收清单超过 2 项未通过
6. AI 开始改我没让它改的代码

停。先列出现状，再决定下一步。
```

## 附录 A：Prompt 设计原则

### A.1 高密度 prompt 的 5 个要素

1. **阶段声明**：当前在哪个阶段（关掉 AI 的自动执行模式）
2. **角色指定**：让 AI 用什么视角思考（决定它的追问方向）
3. **上下文**：贴相关文档 / 已完成清单（给 AI 项目记忆）
4. **约束**：给唯一推荐 / 范围锁死 / 验收前置（防止 AI 自由发挥）
5. **验收标准**：明确可检查的标准（防止"基本完成"）

### A.2 挖掘型 / 执行型 / 验收型 的差异

| 类型 | 核心动词 | 禁止动作 | 典型结构 |
|---|---|---|---|
| 🗣️ 挖掘型 | 追问 / 指出 / 判断 | 写代码 / 给方案 | "请追问 / 请补盲区 / 给判断" |
| ⚡ 执行型 | 创建 / 修改 / 执行 | 列多方案 / 越界改 | "本轮范围锁死 / 完成后自检" |
| ✅ 验收型 | 给证据 / 是否通过 | 模糊词 / 只结论 | "每项给文件/命令/输出，未验证标记" |

### A.3 常见 prompt 反模式

- ❌ "帮我做个项目"——没阶段、没角色、没约束
- ❌ "列几个方案让我选"——甩锅式 prompt
- ❌ "看着办"——放弃判断权
- ❌ "做完告诉我"——没验收标准
- ❌ "差不多就行"——给 AI 留模糊空间
- ❌ 一次贴 1000 行代码让 AI "看看"——上下文过载
