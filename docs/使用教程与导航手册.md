# Vibe Coding 使用教程与导航手册

> 这份手册解决四个问题：**不知道先看哪个、不知道何时翻哪个、内容太多记不住、理论和实操有距离**。
>
> **结构**：前半部分 Walkthrough（带你走一遍），后半部分 Navigation（日常速查）。
> **用法**：第一次读从头到尾走一遍；日常用翻后半部分。

---

# 上篇：Walkthrough（新手入门）

## 第 1 章：5 分钟理解这套体系

### 1.1 这套知识库是什么

一句话：**一套"用 AI 做项目"的工作方法论**，把"AI 怎么做事、你怎么管 AI、文档怎么管项目"三件事讲清楚了。

它不是代码库，不是模板包，而是**一套工作流程**——告诉你从有一个想法到项目上线，每一步该做什么、翻什么文档、怎么和 AI 对话。

### 1.2 核心理念（5 条心法）

如果只能记住 5 句话，记这 5 句：

| 心法 | 含义 | 实操动作 |
|---|---|---|
| **AI 没记忆，文档是它的记忆** | AI 每次对话都是全新的，规则必须落文档 | 改规则前先改文档 |
| **AI 默认动手，你要逼它先想** | AI 收到需求就想写代码，你要拦住它 | 先讨论→再计划→最后写 |
| **AI 默认讨好，你要逼它给判断** | AI 会列 ABC 方案让你选，这是甩锅 | 要求"唯一推荐+不选理由" |
| **AI 默认结论，你要逼它给证据** | AI 说"已完成"不可信 | 要文件/命令/输出/截图 |
| **AI 默认前进，你要逼它会停** | AI 会一口气做完，不验收就继续 | 每个子阶段做完必须停下来汇报 |

### 1.3 文档关系图

```
你想做什么
    │
    ├─ 判断项目状态 ──→ 项目可视化流程手册.md（走哪条路）
    │
    ├─ 查某个阶段怎么操作 ──→ AI协作全流程操作手册.md（详细版）
    │                         全流程速查.md（速查版）
    │
    ├─ 找可直接复制的 prompt ──→ 全流程AI对话Prompt模板库.md
    │
    ├─ 验收某个阶段 ──→ templates/门禁卡-XX门.md
    │
    ├─ 查某个 Skill 方法论 ──→ skills/XX.md
    │
    └─ 溯源某条规则 ──→ sources/_archive/
```

### 1.4 四段式工作循环（每个阶段都走一遍）

```
对话挖掘期（你主导）→ 沉淀落地期（AI起草+你确认）→ 执行交付期（AI主导）→ 验收固化期（你判断+AI给证据）
```

| 段 | 谁主导 | 做什么 | 输出 |
|---|---|---|---|
| 1 对话挖掘 | **你** | 追问、反驳、要证据，把模糊需求挖清楚 | 讨论纪要 |
| 2 沉淀落地 | AI 起草 + 你确认 | 把讨论结果写成文档 | .md 文件 |
| 3 执行交付 | AI | 按文档写代码 / 跑验收 / 修问题 | 代码 + 提交 |
| 4 验收固化 | 你判断 + AI 给证据 | 走验收清单、更新文档、Git 提交 | 验收报告 |

**核心**：第 1、4 段你必须深度参与；第 2 段你只需确认；第 3 段可以放手让 AI 做。

### 1.5 你需要记住的唯一一件事

**打开任何项目，第一步永远是：翻 [项目可视化流程手册.md](./项目可视化流程手册.md) 第 1 章，判断项目处于什么状态。**

状态判断对了，后面走哪条路就对了。状态判断错了，越努力越混乱。

## 第 2 章：新项目第一天（从想法到立项）

### 2.1 场景

你脑子有一个产品想法，想用 AI 做出来。项目文件夹还没建，代码一行没写。

### 2.2 Step 1：判断状态

翻 [项目可视化流程手册.md](./项目可视化流程手册.md) 第 1 章，走决策树：

```
有代码吗？→ 没有，只有想法 → 状态 0：纯想法 → 走第 2 章完整流程
```

### 2.3 Step 2：翻哪个文档

状态 0 走完整流程，顺序是：

```
立项 → 选型 → 架构设计文档 → Agent宪法 → 实施真源文档拆解 → 前端骨架 → 数据库 → 后端骨架 → 联调 → 架构验收 → 业务开发 → 安全审查 → 上线验收 → 上线+反哺
```

**第一天只做前两步：立项 + 选型。** 不要急着写代码。

### 2.4 Step 3：开始第一轮对话（立项）

翻 [AI协作全流程操作手册.md](./AI协作全流程操作手册.md) 第 1 章，复制"开场 prompt"：

```
当前阶段：项目立项，只做产品讨论和立项设计，暂时不要写代码。
你的角色：我的产品合伙人。

我想做的是：[一句话描述]
面向用户：[谁]
解决什么问题：[什么]

请按以下方式追问我：
1. 围绕项目类型、目标用户、核心问题逐项深挖
2. 主动指出我没想到的盲区
3. 帮我判断第一版 MVP 该做什么、不该做什么
4. 给唯一推荐，不要列多方案让我选
5. 讨论清楚后告诉我"现在可以写立项文档了"
```

**替换 `[方括号]` 里的内容，发给 AI。**

### 2.5 Step 4：立项讨论怎么聊

AI 会开始问你问题。你的任务：

1. **如实回答**，不知道就说"没想清楚，你帮我想"
2. **追问盲区**：AI 如果没提到权限、数据归属、状态字段，你主动问
3. **要判断不要选择**：AI 如果列了 A/B/C 方案，说"请给唯一推荐"
4. **讨论完确认**：AI 说"现在可以写立项文档了"后，你再让它写

### 2.6 Step 5：立项文档 + 功能清单 + 阶段计划

讨论清楚后，让 AI 一次性生成三份文档：

1. **立项文档**（8 项内容，见操作手册 1.4.4）
2. **功能清单**（三维度拆解：角色 / 业务流程 / 模块）
3. **阶段计划**（第一阶段是 MVP，只给大方向）

**每份文档生成后，你都要 review。看不懂的地方让 AI 用大白话讲。**

### 2.7 Step 6：技术选型

立项完成后，翻操作手册第 2 章，复制选型 prompt：

```
这是我的立项文档：[贴文档]
产品形态：[Web / 小程序 / App / 后台]

请给出唯一推荐技术栈，不要列多个方案。说明：
- 前端框架 + UI 组件库
- 后端语言 + 框架
- 数据库
- 为什么选它们
- 为什么不选其他常见方案（至少列 3 个被淘汰方案及理由）
```

**关键**：AI 必须说清"为什么不选其他方案"。如果它列三套让你选，用追问 prompt 逼它给唯一推荐。

### 2.8 Step 7：Git 提交

```powershell
git add docs/
git commit -m "chore: 完成项目立项和技术选型"
```

### 2.9 第一天结束时的检查清单

- [ ] 立项文档已落盘（8 项内容）
- [ ] 功能清单已拆解（三维度）
- [ ] 复杂功能已单独拆文档（如有）
- [ ] 阶段计划已制定（第一阶段是 MVP）
- [ ] 技术选型文档已落盘（含不选理由）
- [ ] 已 Git 提交
- [ ] **没有写任何代码**（这是对的）

> **新手最常犯的错误**：第一天就开始写代码。记住：立项和选型没做透，后面全返工。

## 第 3 章：开发中的一天（业务模块开发）

### 3.1 场景

项目骨架已搭好（前端能跑、后端能跑、数据库建好），进入业务模块开发阶段。这是项目最长的阶段。

### 3.2 早上开始工作

**第一步永远是看 PROJECT_STATUS.md**，确认：
- 当前在哪个阶段
- 上次做到哪了
- 今天该做什么

### 3.3 开始一个新模块

翻操作手册第 11 章，用模块开场 prompt：

```
当前阶段：业务模块开发 - [模块名]
你的角色：后端开发 + 前端开发

上下文：
- 实施真源文档：[贴当前阶段]
- 后端架构实施真源文档：[贴]
- 前端架构实施文档：[贴]
- Agent 宪法：[贴]

本模块要做的接口 / 页面：[列]

请先输出本模块实施计划，包含：
1. 接口清单（路径 / 方法 / 入参 / 出参 / 权限 / P0-P2）
2. 页面清单
3. 本轮拆分（一个模块拆 N 轮，每轮做什么）
4. 每轮的验收标准

我确认计划后再开始写。
```

### 3.4 每轮开发的节奏

```
开场（锁范围）→ AI 执行 → AI 汇报三件事 → 你验收 → Git 提交 → 下一轮
```

**开场**（复制防漂移指令）：

```
严格遵守 Agent 宪法和实施真源文档，只实施当前子阶段，不要超出范围，做完后停下来汇报，不要自动进入下一个子阶段。

本轮范围锁死：
- [任务1]
- [任务2]
- [任务3]
```

**AI 执行完后**，让它汇报三件事：

```
完成后汇报三件事：
1. 对照实施真源文档，这个阶段哪些做完了，哪些还没做完
2. 到底改了哪些地方，有没有漏做，有没有多做
3. 怎么验证的，验证结果是什么，下一个该做哪个子阶段，为什么
```

### 3.5 遇到问题

| 问题 | 翻哪里 | 用什么话术 |
|---|---|---|
| AI 改乱了越补越乱 | 操作手册 15.1 | "停。请先列出最近 3 次 Git 提交……" |
| AI 理解错需求 | 操作手册 15.2 | "停。我重新对齐需求……" |
| AI 用模糊话糊弄 | 操作手册 15.4 | "禁止使用'基本完成'……" |
| AI 列多方案让你选 | 操作手册 15.5 | "你列了 A/B/C 三套让我选——这是甩锅……" |
| 文档和代码对不上 | 操作手册 15.3 | "请检查真源文档和实际代码是否一致……" |
| 需要调试某个 bug | skills/systematic-debugging.md | 用系统化调试流程 |

### 3.6 结束工作

1. **走验收清单**（翻对应门禁卡）
2. **更新 PROJECT_STATUS.md**
3. **更新模块清单模板**（如完成了新模块）
4. **Git 提交**：`feat: 完成 XX 模块（含 XX 接口）`
5. **如发现盲点**，记入 Skill 反哺清单

### 3.7 一天的节奏总结

```
看 PROJECT_STATUS → 开始新模块（写计划）→ 每轮（锁范围→执行→汇报→验收→提交）→ 遇问题翻话术 → 结束更新状态
```

> **新手最常犯的错误**：一口气让 AI 写完整个模块。正确做法是一个模块拆 N 轮，每轮锁死 2-3 个任务，做完一轮验收一轮。

## 第 4 章：接手项目第一天（状态判断 + 抢救）

### 4.1 场景

你拿到一个项目——可能是别人做的，可能是自己之前做的。代码有了但不知道什么状态。

### 4.2 Step 1：判断状态

翻 [项目可视化流程手册.md](./项目可视化流程手册.md) 第 1 章，走决策树：

```
有代码吗？
├─ 没有 → 状态 0（走第 2 章完整流程）
└─ 有 → 能跑起来吗？
    ├─ 跑不起来 → 状态 1：烂尾（走第 3 章抢救）
    ├─ 能跑但残缺 → 状态 2：半成品（走第 4 章补全）
    └─ 能跑且基本完整 → 有真源文档吗？
        ├─ 没有 → 状态 3：黑盒（走第 5 章考古）
        └─ 有 → 代码和文档对得上吗？
            ├─ 对不上 → 状态 3：黑盒
            └─ 对得上 → 状态 4：健康（走第 6 章迭代）
```

> **判断窍门**：90% 的"已完成项目"实际是状态 3（黑盒），不是状态 4。先按状态 3 走考古流程。

### 4.3 状态 1：烂尾抢救（第一天）

1. **停止继续补**（最重要）
2. **Git 现状盘点**：`git log --oneline -20` + `git status`
3. **找到最后一个能跑的提交**
4. **回退到稳定版本**
5. **转入状态 3 考古流程**

话术（直接复制）：
```
停。当前改动已经偏离目标。请先列出最近 3 次 Git 提交，告诉我每个版本做了什么。
我决定回退到哪个版本后，再重新开始这一轮。
不要继续补，越补越乱。
```

### 4.4 状态 2：半成品补全（第一天）

1. **反推已有部分真源**（让 AI 读代码反推规则）
2. **梳理已完成 vs 待完成清单**
3. **立宪法约束后续开发**
4. **按状态 0 流程补未完成部分**

> 90% 的半成品是"凭感觉写的"，应该先固化规则再继续，否则越写越乱。

### 4.5 状态 3：黑盒考古（第一天）

这是最常见的情况。三阶段走：

**阶段 1：反推固化（不整改）**

让 AI 读代码，反推三份真源文档：

```
请阅读后端代码，反推并生成《后端架构实施真源文档》。不要改任何代码。
包含：语言框架 / 目录责任表 / 接口响应规则 / 错误处理 / 日志 / 数据库连接 / 权限入口 / 框架复用边界 / 新增模块规则 / 启动证据包。
每项给文件路径 + 行号作为证据。
```

同样的方式反推前端真源文档和数据库设计真源文档。

**阶段 2：Skill 校验（不整改）**

固化完才能校验。5 个 Skill 跑一遍：
1. `backend-architecture-acceptance`（后端架构验收）
2. `backend-security-review`（后端安全审查）
3. `database-design`（数据库设计校验）
4. `browser-verification`（前端浏览器验证）
5. `requesting-code-review`（整体代码审查）

**阶段 3：分批整改 + 反哺**

按严重度分批改：严重 → 数据 → 风险 → 建议。每批改完回归一次。

### 4.6 状态 4：健康迭代

项目能跑、有文档、文档对得上。直接进入迭代：

1. 每个新需求先拆计划（用 `writing-plans` skill）
2. 每轮交付必走验收（用 `verification-before-completion` skill）
3. 规则变更同步真源文档
4. 盲点反哺 Skill 库

### 4.7 接手项目第一天的检查清单

- [ ] 已判断项目状态（0/1/2/3/4）
- [ ] 已翻对应章节走流程
- [ ] 如有代码，已反推或确认真源文档
- [ ] 已建立 PROJECT_STATUS.md
- [ ] 已 Git 提交当前状态

# 下篇：Navigation（日常导航）

## 第 5 章：文档使用地图

### 5.1 六种分类总表

#### 🔴 主力查看（每天都要看）

| 文档 | 什么时候看 | 怎么用 |
|---|---|---|
| 项目里的 `PROJECT_STATUS.md` | 每天开始工作前 | 看当前阶段 + 上次做到哪 |
| 项目里的 Agent 宪法 | 熟悉内容即可（AI 自动读） | 改规则时更新 |

#### 🟡 顺着步骤走（按阶段翻）

| 文档 | 什么时候翻 | 定位 |
|---|---|---|
| [项目可视化流程手册.md](./项目可视化流程手册.md) | 拿到项目第一步 | 判断状态 + 走哪条路 |
| [AI协作全流程操作手册.md](./AI协作全流程操作手册.md) | 进入某个阶段前 | 详细操作细则（17 章） |
| [全流程速查.md](../workflows/全流程速查.md) | 快速回顾某个阶段 | 速查版（触发→步骤→判定） |
| `templates/门禁卡-XX门.md` | 每个阶段验收时 | 验收清单 |

#### 🔵 参考查看（遇到问题翻）

| 文档 | 什么时候翻 | 怎么用 |
|---|---|---|
| [全流程AI对话Prompt模板库.md](./全流程AI对话Prompt模板库.md) | 需要和 AI 对话时 | 找对应阶段的 prompt 复制 |
| `skills/XX.md` | 需要某个 Skill 方法论时 | 翻对应 Skill 的规则和步骤 |
| `sources/_archive/` | 需要溯源某条规则时 | 偶尔翻 |

#### 🟢 可以完全复制（直接用）

| 文档 | 复制什么 | 怎么用 |
|---|---|---|
| Prompt 模板库 | 所有 prompt 模板 | 复制 → 替换 `[方括号]` → 发给 AI |
| 操作手册每章的"开场 prompt" | 每章的 prompt 代码块 | 复制 → 替换 → 发送 |
| `templates/4 张门禁卡` | 整份文件 | 复制到项目根目录 |
| `skills/` 下的 skill 文件 | 整份文件 | 复制到项目根目录 |
| 操作手册 17.3 开场模板 | 模板代码块 | 每次对话开始前复制 |
| 操作手册 17.4 收尾模板 | 模板代码块 | 每次宣布完成前复制 |
| 操作手册 17.5 跨阶段原则 | 三条原则的 prompt | 每个子阶段开工/收尾时复制 |
| 操作手册 15 章异常话术 | 5 种异常话术 | 遇到对应问题时复制 |

#### 🟣 需要自己调整（不能照抄）

| 文档 | 为什么不能照抄 | 怎么调整 |
|---|---|---|
| Agent 宪法 | 必须结合项目真实情况 | 把通用模板扔给 AI，说"根据本项目情况定制" |
| `PROJECT_STATUS.md` | 每个项目独立维护 | 复制模板 → 填当前项目信息 |
| `模块清单模板.md` | 每个项目独立维护 | 复制模板 → 填实际模块 |
| 实施真源文档 | 每个项目独立拆解 | 让 AI 基于功能清单和阶段计划拆解 |
| 立项文档 / 选型文档 | 每个项目不同 | 用 prompt 引导 AI 生成 |

#### 🟤 需要后面持续优化（不是一次写完）

| 文档 | 为什么 | 怎么优化 |
|---|---|---|
| Agent 宪法 | 一点点攒 | 每踩一个坑、每定一个规则就加一条 |
| 实施真源文档 | 边做边补 | 完成一个阶段拆下一个；规则变更时同步 |
| `skills/` 库 | 持续迭代 | 每个项目至少反哺 1 条经验 |
| Prompt 模板库 | 使用中改进 | 发现不好用就改，缺了就补 |

**开始对话前**（复制 17.3 开场模板）：


## 第 7 章：常见误区和进阶心法

### 7.1 新手最常犯的 5 个错误

| 错误 | 后果 | 正确做法 |
|---|---|---|
| **跳过立项直接写代码** | AI 自由发挥，方向跑偏 | 先立项→选型→再写代码 |
| **一口气让 AI 做完** | 一口气做完 = 一口气错完 | 每轮锁 2-3 个任务，做完验收 |
| **听"已完成"就放过** | 问题堆积到上线才爆 | 每项要证据（文件/命令/输出） |
| **改了规则不改文档** | 下次 AI 按旧规则写 | 改代码前先改文档 |
| **让 AI 列方案你选** | AI 甩锅，你背锅 | 要求"唯一推荐+不选理由" |

### 7.2 进阶使用的 3 个建议

#### 建议 1：把"三件事汇报"变成肌肉记忆

每完成一个子阶段，不管大小，都让 AI 汇报：
1. 对照真源文档，哪些做完了，哪些没做完
2. 改了哪些地方，有没有漏做、多做
3. 怎么验证的，下一个该做什么

> 这三件事问完，你就能判断 AI 是在按文档走还是凭感觉跑。

#### 建议 2：养成"先评估再动手"的习惯

任何需求变更，先问 AI："这个改动会不会影响地基？" 地基包括：技术栈、目录结构、数据库表、核心字段、登录权限、支付流程。

> 判断不了就直接问 AI 讨论。宁可多问一句，不要图省事让 AI 把架构破坏了。

#### 建议 3：每个项目至少反哺 1 条经验

项目上线后，花 10 分钟回顾：哪个坑是 Skill 没覆盖的？哪个 prompt 不好用？

> 反哺是复利的关键。跳过这一步，用 100 次也不会变强。

### 7.3 心法浓缩（贴在显示器上）

```
┌─────────────────────────────────────────────────┐
│  AI 没记忆 → 文档是它的记忆                        │
│  AI 默认动手 → 你要逼它先想                        │
│  AI 默认讨好 → 你要逼它给判断                      │
│  AI 默认结论 → 你要逼它给证据                      │
│  AI 默认前进 → 你要逼它会停                        │
│                                                   │
│  判断模糊先聊，规则清晰才做                         │
│  改代码前先改文档，改完代码再核文档                  │
│  每推进一块，先验收再往下                           │
│  地基变更必须先评估再动手                           │
└─────────────────────────────────────────────────┘
```

| 阶段 | 目标 | 做什么 |
