# 门禁卡：立项门

> 包含阶段：立项 + 技术选型 + Agent 宪法  
> 过门禁标准：方向定死了，AI 有规矩了

---

## 过门禁标准（硬指标）

- [ ] 立项文档已落盘，覆盖 8 项内容
- [ ] 功能清单已拆解（三维度：角色 / 业务流程 / 模块）
- [ ] 阶段计划已制定（第一阶段必须是 MVP）
- [ ] 技术选型文档已落盘，AI 给了唯一推荐
- [ ] Agent 宪法已落盘，放在 IDE 会读取的位置
- [ ] 已 Git 提交（至少 3 次：立项 / 选型 / 宪法）

---

## 验收清单

### 立项文档
- [ ] 项目是什么 / 面向谁 / 解决什么问题
- [ ] 第一版做什么、不做什么（MVP 边界）
- [ ] 产品流程和用户路径
- [ ] 数据和业务对象清单
- [ ] 技术路线初步判断
- [ ] 验收规则（可检查标准）

### 功能清单
- [ ] 按角色拆解（普通用户 / 管理员 / 商家等各自能做什么）
- [ ] 按业务流程拆解（核心流程走通：如下单 → 支付 → 发货）
- [ ] 按模块拆解（相关功能归到一起：账号 / 商品 / 订单）
- [ ] 每个功能写清：给谁用 / 输入 / 输出 / 权限 / 正常流程 / 异常边界
- [ ] 复杂功能已单独拆文档（订单流转 / 支付退款 / 优惠叠加 / 权限系统）
- [ ] 复杂功能在功能清单里留了索引

### 阶段计划
- [ ] 第一阶段是 MVP（能跑通核心流程的最小可用版本）
- [ ] 每个阶段包含哪些功能已说明
- [ ] 为什么这么排已说明
- [ ] 只给大阶段划分，没细化到技术实现（细节等选型定了再排）

### 技术选型文档
- [ ] 产品形态已确定
- [ ] 前端 / 后端 / 数据库 / SDK 已确定
- [ ] 为什么选、为什么不选其他方案
- [ ] 什么情况下需要重新评估

### Agent 宪法
- [ ] 不超过 200 行
- [ ] 含禁止事项（≥ 5 条）
- [ ] 含验收标准
- [ ] 含框架优先原则
- [ ] 各 IDE 入口文件已创建

## 常用 Prompt（5 个）

### 1. 项目启动（挖掘型）
```
当前阶段：项目立项，只做产品讨论和立项设计，暂时不要写代码。
你的角色：我的产品合伙人。

我想做的是：[一句话描述]
面向用户：[谁]
解决什么问题：[什么]

请按以下方式追问我：
1. 围绕项目类型、目标用户、核心问题逐项深挖
2. 主动指出我没想到的盲区
3. 帮我判断第一版 MVP 该做什么、不该做什么
4. 给唯一推荐，不要列多方案让我选
5. 讨论清楚后告诉我"现在可以写立项文档了"，等我确认再写
```

### 2. 功能清单拆解（挖掘型）
```
请根据立项文档输出一份详细的功能清单，请逐个功能说明：
- 这个功能给谁用
- 需要哪些输入
- 产生什么结果
- 涉及哪些角色和权限
- 正常流程是什么
- 有哪些异常和边界情况

拆解维度：
1. 按角色拆解（普通用户 / 管理员 / 商家各自能做什么）
2. 按业务流程拆解（核心流程走通）
3. 按模块拆解（相关功能归到一起）

如果某个功能特别复杂（如角色权限系统、订单流转、支付退款、优惠叠加），请把它单独写一份文档，并在功能清单里留一个索引指过去。先不要写代码。
```

### 3. 阶段计划（挖掘型）
```
请根据功能清单，把开发拆成几个大阶段：
- 第一阶段必须是一个能跑通核心流程的最小可用版本（MVP）
- 请说明每个阶段包含哪些功能
- 为什么这么排

先只给大的阶段划分，不要现在就细化到具体技术实现和详细开发计划。
（这个时候还不确定开发框架等细节，等确定了再去规划细节）
```

### 4. 技术选型（挖掘型）
```
这是我的立项文档：[贴文档]
产品形态：[Web / 小程序 / App / 后台]
部署要求：[私有 / 公有 / 本地]
维护能力：[1人 / 小团队 / 中型团队]

请给出唯一推荐技术栈，不要列多个方案。说明：
- 前端 / 后端 / 数据库 / SDK
- 为什么选它们（结合我的项目情况）
- 为什么不选其他常见方案（至少 3 个被淘汰方案）
- 哪些情况才需要重新评估
```

### 5. Agent 宪法（挖掘型）
```
当前阶段：Agent 宪法创建。暂时不要写业务代码。
你的角色：项目架构顾问。

上下文：
- 立项文档：[贴]
- 技术选型文档：[贴]

请帮我起草 AGENT_CONSTITUTION.md。要求：
1. 短：不超过 200 行
2. 硬：每条都是必须遵守的硬规则
3. 清楚：每条能直接判断对错
4. 只写长期原则
5. 必须包含：项目专属条款 / 禁止事项 / 验收标准 / 框架优先原则

先列大纲让我确认，确认后再写完整版。
```

## 常见遗漏项

- ⚠️ 立项文档只在对话里聊过没落盘
- ⚠️ **跳过功能清单直接写代码**（AI 自由脑补边界，后续越改越乱）
- ⚠️ **功能清单只按模块拆，漏了角色和业务流程维度**（功能不完整）
- ⚠️ **复杂功能硬塞进功能清单**（订单流转 / 支付退款等应单独拆文档 + 留索引）
- ⚠️ **没做阶段计划就开始写代码**（不知道先做什么后做什么，MVP 跑不通）
- ⚠️ **阶段计划直接细化到技术实现**（选型还没定就排细节 = 白排）
- ⚠️ MVP 边界没划清，第一版什么都想做
- ⚠️ 技术选型让 AI 列了 3 个方案让你选（必须给唯一推荐）
- ⚠️ 宪法写了"建议""尽量"等软词（必须改成硬规则）
- ⚠️ 宪法超过 200 行（把细则移到真源文档）
- ⚠️ IDE 入口文件复制了宪法内容（应该只写引用）
