# 接手项目第一天操作清单

> **什么时候用**：拿到一个已经存在的项目（自己做到一半模糊了 / 接手别人的项目 / 烂尾了想抢救）
> **目标**：第一天结束时，项目状态清晰、PROJECT_STATUS.md 已建立、下一步明确
> **用法**：按顺序走，每步完成后勾选，遇到不确定的翻对应章节

---

## 第一步：判断项目处于哪个状态（5 分钟）

回答 3 个问题，对照下表定位：

| 问题 | 你的答案 | 对应状态 | 走哪条路线 |
|---|---|---|---|
| 有代码吗？ | 没有，只有想法 | 状态 0 | 走新项目流程（翻「门禁卡-立项门」） |
| 能跑起来吗？ | 跑不起来 | 状态 1（烂尾） | 路线 A：烂尾抢救 |
| 能跑起来吗？ | 能跑但功能残缺 | 状态 2（半成品） | 路线 B：半成品补全 |
| 有立项/架构/真源文档吗？ | 能跑但没文档 | 状态 3（黑盒） | 路线 C：黑盒考古 |
| 文档和代码对得上吗？ | 对得上 | 状态 4（健康） | 走「门禁卡-业务门」或「上线门」 |

> 💡 **判断窍门**：90% 的"已完成项目"实际是状态 3（黑盒），不是状态 4。先按状态 3 走考古流程。

**我的项目状态是**：__________  
**走路线**：__________

- [ ] 第一步完成

---

## 路线 A：烂尾抢救（状态 1 — 跑不起来）

> 核心原则：先停，别继续补。AI 越补越乱。

### A-1. 停止继续补，先盘点 Git

复制以下 prompt 发给 AI：

```
停。当前改动已经偏离目标。请先盘点：
1. git log --oneline -20  看最近 20 次提交
2. git status  看当前工作区状态
3. git diff --stat  看未提交改动规模

告诉我：
- 当前能跑的最近版本是哪个（commit-id）？
- 工作区有多少未提交改动？
- 不要改任何代码，只诊断。
```

**完成判定**：AI 给出了最近能跑的 commit-id 和未提交改动规模。

### A-2. 备份当前代码

```powershell
# 复制整个项目目录到 _backup_日期
Copy-Item -Recurse "项目目录" "项目目录_backup_$(Get-Date -Format 'yyyyMMdd')"
```

> 没 Git 就没法回退——这是为什么 Git 是后悔药。

### A-3. 回退到稳定版本（有 Git 时）

```
请帮我回退到 [commit-id]。
说明：
- 回退后会丢失哪些改动？
- 是否需要先 stash 当前未提交改动？
- 用 git reset --hard 还是 git checkout？给建议和理由。
```

**完成判定**：项目已回退到能跑的版本。

### A-4. 能跑后 → 转入路线 C（黑盒考古）

回退到能跑的版本后，按路线 C 走：反推文档 → Skill 校验 → 分批整改。

- [ ] 路线 A 完成，转入路线 C

---

## 路线 B：半成品补全（状态 2 — 能跑但残缺）

> 核心原则：先盘点已有进度，再固化规则，最后补缺口。

### B-1. 盘点现有进度

复制以下 prompt 发给 AI：

```
这是我做到一半的项目，我自己也记不清进度了。
请盘点（不要改代码，只诊断）：

1. 技术栈：从 package.json / requirements.txt 等推断
2. 已完成的功能清单（给文件路径）
3. 未完成的功能清单
4. 已有的文档清单
5. 缺失的文档清单
6. 当前能跑吗？给启动命令和输出
7. 数据库有哪些表？表结构是什么？

输出一张表：已完成 / 未完成 / 有文档 / 无文档
```

**完成判定**：AI 给出了已完成/未完成/有文档/无文档 4 张清单。

### B-2. 反推已有部分的真源文档

```
请阅读现有代码，反推生成以下文档（不要改代码）：

1. 《后端架构实施真源文档》
   包含：语言框架 / 目录责任表 / 接口响应规则 / 错误处理 / 日志 / 数据库连接 / 权限入口
   每项给文件路径 + 行号作为证据。说不清的标记"未明确"。

2. 《数据库设计真源文档》
   包含：数据库服务 / 业务对象 / 对象关系 / 核心表 / 字段规范 / 软删除策略
   每项给 schema 文件 + 行号作为证据。

3. 《前端架构实施真源文档》（如有前端）
   包含：框架+UI库 / 设计风格 / 目录结构 / 设计token / 组件复用规则 / 路由 / 接口层
   每项给文件路径作为证据。
```

**完成判定**：3 份真源文档已生成，每项有文件路径证据。

### B-3. 立现状宪法

```
基于刚反推的真源文档，请帮我起草 AGENT_CONSTITUTION.md。要求：
1. 短：不超过 200 行
2. 硬：每条都是必须遵守的硬规则
3. 清楚：每条能直接判断对错
4. 只写长期原则
5. 必须包含：项目专属条款 / 禁止事项 / 验收标准 / 框架优先原则
6. 引用真源文档（不复制内容，只写路径）
```

**完成判定**：AGENT_CONSTITUTION.md 已生成，放在 IDE 会读取的位置。

### B-4. 对照 4 门禁定位缺口

把 B-1 盘点结果对照 4 门禁：

| 门禁 | 检查项 | 已有 / 缺失 |
|---|---|---|
| 立项门 | 立项文档 | |
| 立项门 | 技术选型文档 | |
| 立项门 | Agent 宪法 | |
| 架构门 | 前端骨架（能启动+首页能开） | |
| 架构门 | 数据库设计文档 | |
| 架构门 | 后端骨架（4 条线跑通） | |
| 架构门 | 后端架构验收 + 真源文档 | |
| 业务门 | P0 功能完成 | |
| 业务门 | 安全审查 | |
| 上线门 | 上线前总验收 | |

### B-5. 从最缺的门禁开始补

- 缺立项门 → 翻「门禁卡-立项门」补
- 缺架构门 → 翻「门禁卡-架构门」补
- 缺业务门 → 翻「门禁卡-业务门」补
- 缺上线门 → 翻「门禁卡-上线门」补

- [ ] 路线 B 完成

---

## 路线 C：黑盒考古（状态 3 — 能跑但没文档）

> 核心原则：先反推文档（不改代码），再 Skill 校验，最后分批整改。

### C-1. 反推三份真源文档（不改代码）

复制以下 prompt 发给 AI：

```
这是一个能跑但没文档的项目。请读代码，反推生成 3 份文档。
不要改任何代码，只生成文档。

1. 《后端架构实施真源文档》
   包含：语言框架 / 目录责任表 / 接口响应规则 / 错误处理 / 日志 /
   数据库连接 / 权限入口 / 框架复用边界 / 新增模块规则 / 启动证据包
   每项给文件路径 + 行号作为证据。说不清的标记"未明确"。

2. 《前端架构实施真源文档》（如有前端）
   包含：框架+UI库 / 设计风格 / 目录结构 / 设计token /
   多语言主题 / 组件复用规则 / 路由 / 接口层 / 状态管理
   每项给文件路径作为证据。

3. 《数据库设计真源文档》
   包含：数据库服务 / 业务对象 / 对象关系 / 核心表 /
   字段规范 / 软删除策略 / 三大范式校验
   每项给 schema 文件 + 行号作为证据。
```

**完成判定**：3 份真源文档已生成，每项有文件路径证据。

### C-2. 立现状宪法

同 B-3，基于反推的真源文档起草 AGENT_CONSTITUTION.md。

### C-3. Skill 校验（5 个 Skill 跑一遍）

按顺序对项目跑 5 个 Skill，输出整改清单：

| 顺序 | Skill | 关注重点 | prompt |
|---|---|---|---|
| 1 | `backend-architecture-acceptance` | 目录责任、11 种接口场景、框架复用 | 见门禁卡-架构门 prompt 3 |
| 2 | `backend-security-review` | 5 道关卡、水平越权、注入 | 见门禁卡-业务门 prompt 3 |
| 3 | `database-design` | 三大范式、密码哈希、金额浮点 | "请对现有数据库做三大范式校验，每项给证据" |
| 4 | `browser-verification` | 设计 token、组件复用、跨端单位 | "请按浏览器验证清单检查前端，每项给截图或证据" |
| 5 | `requesting-code-review` | 5 维度审查、风险分级 | 见门禁卡-上线门 prompt 1 |

**每个 Skill 的 prompt 模板**：

```
当前阶段：[Skill 名称]校验。不改代码，只审查。
你的角色：[架构审查员 / 安全审查员 / 代码审查员]

请按 [Skill 文件路径] 的清单逐项检查，每项必须给证据（文件 + 行号）。
未验证项标记"未验证"。禁止"基本完成"。

输出整改清单，按严重度分级：
🔴 严重：必须修
🟡 风险：建议修
🟢 建议：可选优化
```

**完成判定**：5 份审查报告 + 1 份汇总整改清单。

### C-4. 分批整改（按严重度）

```
基于整改清单，请按严重度分批修复：
- 第 1 批：🔴 严重（阻断使用）
- 第 2 批：数据问题（schema / 范式 / 敏感数据）
- 第 3 批：🟡 风险
- 第 4 批：🟢 建议

每批改完：
1. 回归测试（确保没改坏）
2. 同步真源文档（改了规则就改文档）
3. Git 提交：fix: 修复 [具体问题]
```

**完成判定**：🔴 严重项全部修复，🟡 风险项已登记。

- [ ] 路线 C 完成

## 第二步：建立 PROJECT_STATUS.md（无论走哪条路线都要做）

### 操作

1. 从知识库复制 PROJECT_STATUS.md 到项目根目录：
   ```powershell
   Copy-Item "e:\FiveTierProjectSystem\01-Inbox\Vibe_Coding\templates\PROJECT_STATUS.md" "项目根目录\PROJECT_STATUS.md"
   ```

2. 填写项目信息：
   - 项目名称
   - 开始日期（接手日期）
   - 当前门禁（根据第一步的盘点结果填写）

3. 从对应门禁卡复制验收清单到 PROJECT_STATUS.md

4. 填写「下一步」（不超过 3 项）

5. 在「工作日志」加第一行：今天做了什么

### 完成判定

- [ ] PROJECT_STATUS.md 已在项目根目录
- [ ] 项目信息已填
- [ ] 当前门禁已确定
- [ ] 验收清单已复制
- [ ] 下一步已填

在 AGENT_CONSTITUTION.md 里加引用条款：

## 常见坑

- ⚠️ **直接开始改代码**：接手项目第一天不要改代码，先盘点 + 反推文档
- ⚠️ **推倒重来**：已完成的代码即使不完美也保留，只补缺口
- ⚠️ **跳过真源文档**：不反推文档，AI 会用通用标准乱改，越改越乱
- ⚠️ **一次性整改所有问题**：按严重度分批改，每批改完回归
- ⚠️ **改了代码不改文档**：改了规则必须同步真源文档
- ⚠️ **不建 PROJECT_STATUS.md**：不建状态文件，第二天又不知道进度了
