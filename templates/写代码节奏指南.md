# 写代码节奏指南

> **什么时候用**：每个阶段开始前翻一下，确认"现在该不该写代码"
> **核心原则**：代码是最后一步，不是第一步

---

## 一、什么时候不写代码

| 阶段 | 做什么 | 为什么不写 |
|---|---|---|
| 立项（阶段 1） | 讨论产品方向、功能边界 | 方向没定就写 = 白写 |
| 技术选型（阶段 2） | 确定技术栈、SDK | 技术栈没定就写 = 推倒重来 |
| Agent 宪法（阶段 3） | 给 AI 立规矩 | 规矩没立就写 = AI 自由发挥 |
| 数据库设计（阶段 5） | 先出 schema.sql 设计文件 | 设计没定就建表 = 越建越乱 |
| 后端架构验收（阶段 7） | 验收骨架是否合规 | 验收没做就加业务 = 地基不稳盖楼 |
| 安全审查（阶段 9） | 审查 5 道安全关卡 | 安全没查就上线 = 埋雷 |
| 上线前总验收（阶段 10） | 5 维度系统审查 | 没审就上线 = 带病上线 |

**一句话**：前 3 个阶段（立项 + 选型 + 宪法）绝对不写代码。数据库设计先出文件再建表。验收阶段只审查不改代码。

---

## 二、什么时候开始写代码

### 3 个前提条件（必须全满足）

```
开始写代码前，问自己 3 个问题：

✅ 1. 当前门禁的验收清单全勾了吗？
     没勾 → 先过门禁，再写代码

✅ 2. 本轮范围锁死了吗？（不超过 3 个接口/页面）
     没锁 → 先写实施计划，锁死范围

✅ 3. 验收标准前置了吗？（做完后怎么验收）
     没前置 → 先定义验收标准，再开始写
```

**缺一不可**。

| 缺了什么 | 后果 |
|---|---|
| 门禁没过 | 地基不稳，后面全要推倒 |
| 范围没锁 | AI 自由发挥，写了一堆你不要的 |
| 验收没前置 | 做完不知道怎么验，AI 说"完成"你就信了 |

### 开始写代码的标准 prompt

```
当前阶段：[阶段名称]
你的角色：[角色]

上下文：[贴真源文档/实施计划]

本轮范围：只做以下 [N] 个任务，不多加：
1. [具体任务 1]
2. [具体任务 2]
3. [具体任务 3]

验收标准：
- [可检查标准 1]
- [可检查标准 2]
- [可检查标准 3]

不要一次性做完所有功能。做完后按验收标准自检，每项给证据。
```

## 三、什么时候停

### 5 个停止信号

| 信号 | 表现 | 怎么做 |
|---|---|---|
| **本轮范围写完了** | 计划 3 个接口，3 个都写完了 | 停。不要顺手加第 4 个 |
| **遇到模糊需求** | AI 说"这里可以这样也可以那样" | 停。先聊清楚再写 |
| **AI 偏离规则** | 代码不符合宪法/真源文档 | 停。先纠正，再继续 |
| **每轮结束** | 代码写完了 | 停。走验收清单，验收通过才宣布完成 |
| **改坏了** | 跑不起来、报错了 | 停。先回退到上一个稳定版本 |

### 停止时的标准话术

**范围写完了**：
```
本轮范围已完成。停止写新代码。
请按验收标准逐项自检，每项给证据。
```

**遇到模糊需求**：
```
停。这里我没想清楚，先不写代码。
请围绕这个问题反问我：[问题描述]
讨论清楚后再继续写。
```

**AI 偏离规则**：
```
停。你写的 [具体代码] 不符合 [具体规则]。
规则来源：[宪法/真源文档路径]
请先改回符合规则，再继续。
```

**改坏了**：
```
停。当前改动已经偏离目标。
请执行：
1. git status  看当前状态
2. git diff  看改了什么
3. 建议回退到哪个版本
先回退，再重新开始这一轮。
```

## 四、什么时候继续

### 2 个继续条件

```
本轮验收通过了吗？
  → 没通过 → 先整改，不要继续下一轮
  → 通过了 ↓

真源文档同步了吗？
  → 没同步 → 先同步（改了规则就改文档）
  → 同步了 ↓

→ 可以开始下一轮
```

### 继续时的标准 prompt

```
上一轮验收已通过，真源文档已同步。
现在进入下一轮。

本轮范围：
1. [具体任务 1]
2. [具体任务 2]

验收标准：
- [可检查标准 1]
- [可检查标准 2]

不要回顾上一轮的内容，直接进入本轮。
```

## 五、写代码的完整节奏

### 宏观节奏：门禁制

```
立项门 ──→ 架构门 ──→ 业务门 ──→ 上线门
(不写代码)  (写骨架)   (写业务)   (不写代码)
```

### 微观节奏：轮次制（在业务门内）

```
第 1 轮：模块 A
  锁死范围 → 写代码 → 验收 → 同步文档 → Git 提交
                                                    ↓
第 2 轮：模块 B                                    ↓
  锁死范围 → 写代码 → 验收 → 同步文档 → Git 提交  ↓
                                                    ↓
第 3 轮：模块 C                                    ↓
  锁死范围 → 写代码 → 验收 → 同步文档 → Git 提交  ↓
                                                    ↓
...继续直到所有模块完成 ←──────────────────────────┘
```

### 每轮的完整流程

```
┌─────────────────────────────────────────────┐
│ 1. 写实施计划（你 + AI 对话）                  │
│    → 本轮做什么、不做什么、验收标准              │
├─────────────────────────────────────────────┤
│ 2. 锁死范围                                   │
│    → 不超过 3 个接口/页面                       │
├─────────────────────────────────────────────┤
│ 3. AI 写代码（AI 主导，你盯进度）               │
│    → 遇到模糊/偏离 → 停 → 纠正 → 继续          │
├─────────────────────────────────────────────┤
│ 4. 完成前验收（你判断 + AI 给证据）              │
│    → 7 项验收：功能/规则/安全/数据/接口/日志/回归 │
├─────────────────────────────────────────────┤
│ 5. 同步真源文档                                │
│    → 改了规则就改文档                            │
├─────────────────────────────────────────────┤
│ 6. Git 提交                                   │
│    → feat: 完成 XX 模块                        │
├─────────────────────────────────────────────┤
│ 7. 更新 PROJECT_STATUS.md                     │
│    → 勾选验收清单、更新下一步                    │
└─────────────────────────────────────────────┘
```

## 六、不同阶段的代码量控制

| 阶段 | 代码量 | 控制原则 |
|---|---|---|
| 前端骨架（阶段 4） | 最小可运行 | 能启动 + 首页能打开 + 路由 + token + UI 库接入。**不写业务页面** |
| 后端骨架（阶段 6） | 最小可运行 | 4 条线跑通（启动/接口/业务/运维）。**不写完整业务** |
| 业务模块（阶段 8） | 每轮 ≤ 3 个接口/页面 | 一个模块一轮，多轮交付。**不一口气写完** |
| 整改修复 | 每批 ≤ 5 个问题 | 按严重度分批。**不一次性修完所有** |

## 七、代码节奏速查表

### 开始写代码前检查

```
□ 当前门禁过了吗？
□ 本轮范围锁死了吗？（≤ 3 个接口/页面）
□ 验收标准前置了吗？
□ 上一轮真源文档同步了吗？
□ Git 工作区干净吗？（先提交上一轮）
→ 全勾了才开始写
```

### 写代码中遇到情况

| 情况 | 动作 |
|---|---|
| 范围写完了 | 停 → 走验收 |
| 需求模糊 | 停 → 先聊清楚 |
| AI 偏离规则 | 停 → 纠正 → 继续 |
| 跑不起来 | 停 → 回退 → 重新开始 |
| 发现新问题 | 记下来 → 不顺手改 → 下一轮处理 |

### 写完代码后检查

```
□ 验收清单 7 项全过了吗？
□ 每项有证据吗？（文件/命令/输出/日志）
□ 真源文档同步了吗？
□ Git 提交了吗？
□ PROJECT_STATUS.md 更新了吗？
→ 全勾了才开始下一轮
```

## 八、中途接手/做到一半忘记节奏怎么办

### 情况 1：接手别人的项目

**第一步：不要写代码，先盘点**

```
这是别人做到一半的项目。请先盘点（不要改代码）：

1. 技术栈是什么？（从 package.json / requirements.txt 等推断）
2. 已完成的功能清单（给文件路径）
3. 未完成的功能清单
4. 已有的文档清单（立项/选型/宪法/架构/真源文档）
5. 缺失的文档清单
6. 当前能跑吗？给启动命令和输出
7. 数据库有哪些表？

输出：已完成/未完成/有文档/无文档 四张清单
```

**第二步：对照 4 门禁定位缺口**

| 门禁 | 检查项 | 已有 / 缺失 |
|---|---|---|
| 立项门 | 立项文档 | |
| 立项门 | 技术选型文档 | |
| 立项门 | Agent 宪法 | |
| 架构门 | 前端骨架（能启动+首页能开） | |
| 架构门 | 数据库设计文档 | |
| 架构门 | 后端骨架（4 条线跑通） | |
| 架构门 | 后端架构验收 + 真源文档 | |
| 业务门 | P0 功能完成 | |
| 业务门 | 安全审查 | |
| 上线门 | 上线前总验收 | |

**第三步：从最缺的门禁开始补**

- 缺立项门 → 翻「门禁卡-立项门」补
- 缺架构门 → 翻「门禁卡-架构门」补
- 缺业务门 → 翻「门禁卡-业务门」补
- 缺上线门 → 翻「门禁卡-上线门」补

**第四步：建立 PROJECT_STATUS.md**

```powershell
Copy-Item "e:\FiveTierProjectSystem\01-Inbox\Vibe_Coding\templates\PROJECT_STATUS.md" "项目根目录\PROJECT_STATUS.md"
```

填写：
- 项目信息
- 当前门禁（根据盘点结果）
- 验收清单（从门禁卡复制）
- 下一步（不超过 3 项）

**第五步：导入 skills 到项目**

```powershell
mkdir skills -Force
Copy-Item -Path "e:\FiveTierProjectSystem\01-Inbox\Vibe_Coding\skills\*" -Destination ".\skills\" -Recurse
```

在 AGENT_CONSTITUTION.md 里加引用条款。

**第六步：从当前门禁继续**

翻对应门禁卡，照着做。

### 情况 2：自己做到一半忘记了

**第一步：让 AI 帮你盘点**

```
这是我做到一半的项目，我自己也记不清进度了。
请盘点（不要改代码）：

1. 已完成的功能清单（给文件路径）
2. 未完成的功能清单
3. 已有的文档清单
4. 当前能跑吗？给启动命令和输出
5. 数据库有哪些表？

输出：已完成/未完成/有文档/无文档 四张清单
```

**第二步：对照 4 门禁定位当前位置**

同情况 1 的第二步。

**第三步：建立 PROJECT_STATUS.md**

同情况 1 的第四步。

**第四步：从当前门禁继续**

翻对应门禁卡，照着做。

### 关键提醒

- ⚠️ **不要从头重做**：已完成的代码即使不完美也保留，只补缺口
- ⚠️ **不要跳过盘点直接写**：不盘点就写 = 继续乱
- ⚠️ **不要跳过真源文档**：没有真源文档，AI 会用通用标准乱改
- ⚠️ **不要一次性补完所有门禁**：按严重度分批，每批改完回归

## 九、常见反模式

| 反模式 | 后果 | 正确做法 |
|---|---|---|
| 立项阶段就让 AI 写代码 | 方向漂移，白写 | 前 3 阶段不写代码 |
| 一口气写完所有业务 | 验收不了，改坏了难回退 | 每轮 ≤ 3 个接口/页面 |
| 范围没锁就开始写 | AI 自由发挥 | 先写实施计划，锁死范围 |
| 写完不验收就宣布完成 | 问题积累 | 每轮走验收清单 |
| 改了代码不改文档 | 下次 AI 按旧规则写 | 改了规则就同步真源文档 |
| 验收不通过就继续下一轮 | 问题越积越多 | 验收通过才继续 |
| 遇到模糊继续写 | 写出来不是你想要的 | 停下来先聊清楚 |
| 改坏了不回退继续补 | 越补越乱 | 回退到稳定版本重新开始 |
| 顺手改范围外的代码 | 引入新问题 | 记下来，下一轮处理 |
| 接手项目不盘点直接写 | 继续乱，越写越偏 | 先盘点，定位缺口，再补 |
| 做到一半忘记节奏继续写 | 不知道进度，验收不了 | 先盘点，建 PROJECT_STATUS.md |
